## Chapter 24 – Automation & Scripting

Manual testing has a ceiling. A single tester working by hand can only run so many commands, review so many responses, and correlate so many data points before fatigue, time constraints, or simple human error start costing them findings. Automation removes that ceiling. It doesn't replace the tester's judgment — it removes the repetitive, mechanical work so the tester's judgment can be spent on the parts of an engagement that actually need a human brain: interpreting results, chaining findings together, and deciding what to try next.

This chapter is about building that automation layer on Kali Linux. It is not a tour of pre-built frameworks that do everything for you — it is a practical guide to writing your own tooling, because custom scripts are what separate a tester who depends entirely on public tools from one who can adapt to any target, any environment, and any constraint a client throws at them.

The chapter is organized around the layers of automation a working penetration tester or bug bounty hunter actually uses in practice:

- **Bash automation** — the glue language of Kali itself. Nearly every tool on the distribution is a command-line binary, and Bash is the fastest way to chain them together, loop over targets, and build repeatable workflows without leaving the terminal.
- **Python automation** — the workhorse for anything Bash can't comfortably do: structured data handling, HTTP libraries, concurrency, parsing, and building tools that need to reason about results rather than just pipe text from one command to the next.
- **PowerShell basics** — essential once an engagement touches Windows. Modern Windows environments are administered through PowerShell, and post-exploitation, Active Directory enumeration, and living-off-the-land techniques all assume at least a working knowledge of it.
- **API automation** — the layer above raw tools. Bug bounty platforms, cloud providers, threat intel sources, and even the tools you build yourself increasingly expose REST APIs. Knowing how to script against them turns one-off checks into repeatable, schedulable pipelines.
- **Custom recon scripts** — where everything in the chapter comes together. Reconnaissance is the highest-volume, most repetitive phase of any assessment, which makes it the phase with the best return on automation investment.

By the end of this chapter, the goal is not just familiarity with these languages and interfaces, but the ability to sit down in front of a new target and, within minutes, have a script running that starts doing useful work while you think about the next step.

#### What You'll Need

- A working Kali Linux installation (bare metal, VM, or WSL)
- Basic command-line comfort (navigating directories, redirecting output, reading man pages)
- A text editor or IDE (VS Code, Vim, or Kali's built-in tools all work)
- Python 3.11+ and its `pip`/`venv` tooling, both included on current Kali releases
- PowerShell Core (`pwsh`), installable on Kali via `apt`, for the PowerShell section
- Legal authorization for any target you point these scripts at — every example in this chapter assumes a lab environment, a bug bounty program's defined scope, or a signed engagement

### 24.1 Bash Automation

Bash is the default shell on Kali Linux and the connective tissue between every tool the distribution ships. Before reaching for Python or a heavier framework, most working testers reach for a Bash script first — it starts instantly, has zero dependencies to install, and can wrap any command-line tool without writing a single line of "real" code around it.

This section builds up from the shell fundamentals a pentester actually uses daily to full multi-tool automation scripts.

#### 1 Why Bash First

Every tool on Kali — `nmap`, `nikto`, `gobuster`, `sqlmap`, `hydra`, `subfinder` — is designed to be run from a command line and to accept input/output via standard streams and files. That means:

- Any tool can be piped into any other tool.
- Any repeated command sequence can be captured in a script and re-run in seconds.
- Loops let you point the same tool at hundreds of targets without manual repetition.

The tradeoff is that Bash is a poor fit for anything involving complex data structures, JSON parsing, or heavy logic branching — that's where Python takes over (see 24.2). But for orchestration, file handling, and gluing tools together, nothing beats Bash's speed of iteration.

#### 2 Shell Scripting Fundamentals

A minimal, correct Bash script always starts with a shebang and strict-mode flags:

```bash
#!/usr/bin/env bash
set -euo pipefail
IFS=$'\n\t'
```

- `set -e` exits the script immediately if any command fails, preventing a broken scan from silently continuing on bad data.
- `set -u` treats unset variables as errors — catches typos in variable names before they cause a silent no-op.
- `set -o pipefail` makes a pipeline fail if *any* command in it fails, not just the last one.
- `IFS=$'\n\t'` restricts word-splitting to newlines and tabs, avoiding subtle bugs when filenames or targets contain spaces.

**Variables and quoting**

```bash
target="example.com"
output_dir="./scans/${target}"
mkdir -p "$output_dir"
```

Always quote variable expansions (`"$target"`, not `$target`). Unquoted variables are one of the most common sources of broken recon scripts once a domain list contains anything unexpected.

**Conditionals**

```bash
if [[ -z "$1" ]]; then
    echo "Usage: $0 <target>"
    exit 1
fi
```

`[[ ... ]]` is preferred over `[ ... ]` in Bash-specific scripts — it supports pattern matching and avoids word-splitting pitfalls.

**Loops over target lists**

```bash
while IFS= read -r domain; do
    echo "[*] Scanning: $domain"
    nmap -Pn -T4 -oN "scans/${domain}.txt" "$domain"
done < targets.txt
```

Reading line-by-line with `IFS= read -r` is the safe pattern for iterating a target list file — it won't mangle backslashes or trim whitespace unexpectedly.

**Functions**

```bash
run_nmap() {
    local target="$1"
    local outfile="scans/${target}_nmap.txt"
    nmap -sV -sC -oN "$outfile" "$target"
}

run_nmap "10.10.10.5"
```

Wrapping logical units in functions makes scripts reusable and testable in isolation — critical once a recon script grows past 50-60 lines.

#### 3 Automating Recon Tools with Bash

A realistic subdomain-to-live-host pipeline, chaining several Kali tools together:

```bash
#!/usr/bin/env bash
set -euo pipefail

domain="$1"
outdir="recon/${domain}"
mkdir -p "$outdir"

echo "[*] Enumerating subdomains..."
subfinder -d "$domain" -silent -o "${outdir}/subdomains.txt"

echo "[*] Probing for live hosts..."
httpx -l "${outdir}/subdomains.txt" -silent -o "${outdir}/live_hosts.txt"

echo "[*] Running nmap against live hosts..."
while IFS= read -r host; do
    stripped_host=$(echo "$host" | sed -E 's~https?://~~')
    nmap -Pn -T4 -oN "${outdir}/nmap_${stripped_host}.txt" "$stripped_host"
done < "${outdir}/live_hosts.txt"

echo "[+] Recon complete. Results in ${outdir}/"
```

This is a deliberately simple, linear pipeline: subdomain enumeration → live host probing → port/service scanning. Every intermediate result is written to disk, which matters for two reasons — you can resume a failed run partway through, and every artifact is preserved for the report later.

#### 4 Parallelization with `xargs` and `GNU parallel`

Sequential loops are slow once a target list grows past a handful of hosts. `xargs -P` and GNU `parallel` both let Bash farm work out across multiple processes:

```bash
cat "${outdir}/live_hosts.txt" | xargs -P 10 -I{} nmap -Pn -T4 -oN "scans/{}.txt" {}
```

`-P 10` runs up to 10 concurrent `nmap` processes. This is often the single highest-impact change you can make to a recon script — cutting a multi-hour sequential scan down to minutes.

GNU `parallel` (install with `apt install parallel`) offers more control, including progress bars and job logging:

```bash
parallel --jobs 10 --bar 'nmap -Pn -T4 -oN scans/{}.txt {}' :::: "${outdir}/live_hosts.txt"
```

Be mindful of rate limits — both on the target side (many WAFs will flag aggressive parallel scanning) and on your own network egress. Bug bounty programs frequently specify request-rate limits in their scope; violating them can get you banned from the program regardless of finding quality.

#### 5 Logging, Error Handling, and Output Management

A recon script that silently swallows errors is worse than no script at all — it gives false confidence that a target was checked when it wasn't. Minimum viable error handling:

```bash
log_file="logs/$(date +%Y%m%d_%H%M%S)_recon.log"
mkdir -p logs

exec > >(tee -a "$log_file") 2>&1

trap 'echo "[!] Script failed at line $LINENO"; exit 1' ERR
```

- `exec > >(tee -a "$log_file") 2>&1` mirrors all stdout/stderr to both the terminal and a timestamped log file.
- The `trap ... ERR` line ensures that if any command fails under `set -e`, you get the exact line number, rather than a silent exit.

For long-running scans, consider writing a simple progress counter:

```bash
total=$(wc -l < targets.txt)
count=0
while IFS= read -r target; do
    count=$((count + 1))
    echo "[${count}/${total}] Scanning ${target}"
    nmap -Pn -oN "scans/${target}.txt" "$target"
done < targets.txt
```

#### 6 Scheduling with Cron

Recurring recon — checking a bug bounty program's asset list for new subdomains every day, for example — is a natural fit for `cron`:

```bash
crontab -e
```

```
0 3 * * * /home/user/scripts/daily_recon.sh example.com >> /home/user/logs/cron.log 2>&1
```

This runs `daily_recon.sh` every day at 3 AM and appends output to a log. A few practical notes for cron jobs specifically:

- Cron runs with a minimal environment — always use absolute paths inside scripts run by cron (`/usr/bin/nmap`, not `nmap`), since `PATH` may not match your interactive shell.
- Test the script manually first with `bash -x script.sh` to catch path or permission issues before trusting it to run unattended.
- Pair scheduled recon with a diffing step (compare today's subdomain list to yesterday's) so you're alerted only to *new* assets, not re-shown the same list daily — see 24.5 for a worked example.

#### 7 A Complete Worked Example

Putting the pieces together — a script that takes a domain, runs subdomain enumeration, filters live hosts, scans them in parallel, and logs everything:

```bash
#!/usr/bin/env bash
set -euo pipefail
IFS=$'\n\t'

domain="${1:?Usage: $0 <domain>}"
outdir="recon/${domain}"
mkdir -p "$outdir" logs

log_file="logs/$(date +%Y%m%d_%H%M%S)_${domain}.log"
exec > >(tee -a "$log_file") 2>&1
trap 'echo "[!] Failed at line $LINENO"; exit 1' ERR

echo "[*] Starting recon for ${domain}"

subfinder -d "$domain" -silent -o "${outdir}/subdomains.txt"
echo "[+] Found $(wc -l < "${outdir}/subdomains.txt") subdomains"

httpx -l "${outdir}/subdomains.txt" -silent -o "${outdir}/live_hosts.txt"
echo "[+] $(wc -l < "${outdir}/live_hosts.txt") hosts are live"

parallel --jobs 8 --bar \
    'nmap -Pn -T4 -sV -oN "'"${outdir}"'/nmap_{/}.txt" {}' \
    :::: "${outdir}/live_hosts.txt"

echo "[+] Recon complete. Results in ${outdir}/"
```

`${1:?Usage: $0 <domain>}` is a compact idiom that exits with a usage message if no argument was supplied — cleaner than a separate `if` block for simple scripts.

### 24.2 Python Automation

Python is the second layer of automation on Kali, and for many working testers it's the primary one. Where Bash excels at chaining existing command-line tools, Python excels at building tools — parsing structured data, making concurrent HTTP requests, maintaining state, and handling errors gracefully instead of just failing the whole pipeline.

Kali ships with Python 3 pre-installed, along with `pip` and `venv` for dependency management, and a large ecosystem of security-focused libraries.

#### 1 Why Python for Security Automation

Most modern web targets speak JSON. Most APIs you'll automate against — bug bounty platforms, cloud providers, threat intel feeds — return JSON. Bash can technically parse JSON with `jq`, but once the logic gets more complex than "extract this one field," Python's native data structures (dicts, lists, native JSON parsing) become dramatically easier to reason about and maintain.

Python also gives you:

- Real exception handling (`try`/`except`) instead of Bash's blunt exit-code checking
- Native concurrency (`threading`, `asyncio`, `concurrent.futures`) for scanning hundreds of targets without external tools like `parallel`
- A massive library ecosystem (`requests`, `httpx`, `scapy`, `paramiko`, `beautifulsoup4`) purpose-built for network and web tasks

#### 2 Environment Setup

Always use a virtual environment for tooling projects, rather than installing packages system-wide:

```bash
python3 -m venv venv
source venv/bin/activate
pip install requests beautifulsoup4 dnspython
```

A `requirements.txt` file keeps your tooling reproducible across machines:

```
requests==2.32.3
beautifulsoup4==4.12.3
dnspython==2.6.1
```

```bash
pip install -r requirements.txt
```

#### 3 Core Scripting Patterns

**Reading target lists and handling files**

```python
def load_targets(filepath: str) -> list[str]:
    with open(filepath, "r") as f:
        return [line.strip() for line in f if line.strip()]

targets = load_targets("targets.txt")
```

**Basic HTTP requests with the `requests` library**

```python
import requests

def check_host(url: str) -> dict:
    try:
        resp = requests.get(url, timeout=5, allow_redirects=True)
        return {
            "url": url,
            "status": resp.status_code,
            "server": resp.headers.get("Server", "unknown"),
            "length": len(resp.content),
        }
    except requests.exceptions.RequestException as e:
        return {"url": url, "error": str(e)}
```

Wrapping every network call in a `try`/`except` block is non-negotiable for recon scripts — a single unreachable host or timeout should never crash a scan that's checking 500 other hosts.

**Argument parsing with `argparse`**

```python
import argparse

parser = argparse.ArgumentParser(description="Basic host status checker")
parser.add_argument("-l", "--list", required=True, help="File with list of targets")
parser.add_argument("-o", "--output", default="results.json", help="Output file")
parser.add_argument("-t", "--threads", type=int, default=10, help="Concurrent threads")
args = parser.parse_args()
```

Every custom tool should accept its inputs via flags rather than hardcoded paths — it's the difference between a one-off script and a reusable tool you can hand to a teammate.

#### 4 Concurrency: Threading and `asyncio`

Sequential requests are the single biggest bottleneck in Python recon scripts. Two common approaches:

**`concurrent.futures.ThreadPoolExecutor`** — simplest option, good for I/O-bound tasks like HTTP requests:

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def scan_targets(targets: list[str], threads: int = 10) -> list[dict]:
    results = []
    with ThreadPoolExecutor(max_workers=threads) as executor:
        futures = {executor.submit(check_host, t): t for t in targets}
        for future in as_completed(futures):
            results.append(future.result())
    return results
```

**`asyncio` with `httpx`** — scales further for very large target lists, at the cost of slightly more complex code:

```python
import asyncio
import httpx

async def check_host_async(client: httpx.AsyncClient, url: str) -> dict:
    try:
        resp = await client.get(url, timeout=5)
        return {"url": url, "status": resp.status_code}
    except httpx.RequestError as e:
        return {"url": url, "error": str(e)}

async def scan_all(targets: list[str], concurrency: int = 50) -> list[dict]:
    limits = httpx.Limits(max_connections=concurrency)
    async with httpx.AsyncClient(limits=limits) as client:
        tasks = [check_host_async(client, t) for t in targets]
        return await asyncio.gather(*tasks)

results = asyncio.run(scan_all(targets))
```

`ThreadPoolExecutor` is usually the right default — it's simpler to debug and sufficient for target lists in the hundreds. Reach for `asyncio` when scanning thousands of hosts and thread overhead starts to matter.

#### 5 Parsing and Structuring Output

Security tooling lives or dies on how usable its output is. Writing structured JSON, rather than free-text logs, means results can feed directly into other scripts, dashboards, or reports:

```python
import json
from datetime import datetime, timezone

def save_results(results: list[dict], outfile: str) -> None:
    report = {
        "scan_time": datetime.now(timezone.utc).isoformat(),
        "total_targets": len(results),
        "results": results,
    }
    with open(outfile, "w") as f:
        json.dump(report, f, indent=2)

save_results(results, "scan_report.json")
```

Parsing tool output that isn't already JSON — e.g. `nmap`'s XML output — is a common Python task:

```python
import xml.etree.ElementTree as ET

def parse_nmap_xml(filepath: str) -> list[dict]:
    tree = ET.parse(filepath)
    root = tree.getroot()
    hosts = []
    for host in root.findall("host"):
        addr = host.find("address").get("addr")
        ports = []
        for port in host.findall(".//port"):
            state = port.find("state").get("state")
            if state == "open":
                ports.append({
                    "port": port.get("portid"),
                    "protocol": port.get("protocol"),
                    "service": port.find("service").get("name", "unknown")
                    if port.find("service") is not None else "unknown",
                })
        hosts.append({"address": addr, "open_ports": ports})
    return hosts
```

Running `nmap` with `-oX -` and piping its XML into a parser like this turns raw scan output into structured data you can filter, sort, and feed into later stages of a pipeline automatically.

#### 6 Error Handling and Retries

Network operations fail constantly — timeouts, connection resets, rate limiting. A production-quality script retries transient failures with backoff rather than treating every failure as final:

```python
import time

def request_with_retry(url: str, retries: int = 3, backoff: float = 1.5) -> requests.Response | None:
    for attempt in range(retries):
        try:
            return requests.get(url, timeout=5)
        except requests.exceptions.RequestException:
            if attempt < retries - 1:
                time.sleep(backoff ** attempt)
            else:
                return None
```

Combine this with logging (Python's built-in `logging` module rather than bare `print()`) so failures are visible after the fact without cluttering live output:

```python
import logging

logging.basicConfig(
    filename="scan.log",
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s",
)

logging.info("Starting scan of %d targets", len(targets))
logging.warning("Host %s timed out after %d retries", host, retries)
```

#### 7 A Complete Worked Example: Multi-threaded Port/Header Scanner

```python
#!/usr/bin/env python3
"""Simple multi-threaded HTTP recon scanner."""

import argparse
import json
import logging
from concurrent.futures import ThreadPoolExecutor, as_completed
from datetime import datetime, timezone

import requests

logging.basicConfig(level=logging.INFO, format="%(asctime)s %(message)s")


def load_targets(filepath: str) -> list[str]:
    with open(filepath) as f:
        return [line.strip() for line in f if line.strip()]


def check_host(url: str) -> dict:
    if not url.startswith(("http://", "https://")):
        url = f"https://{url}"
    try:
        resp = requests.get(url, timeout=6, allow_redirects=True)
        return {
            "url": url,
            "status": resp.status_code,
            "server": resp.headers.get("Server", "unknown"),
            "title_hint": "<title>" in resp.text[:2000].lower(),
        }
    except requests.exceptions.RequestException as e:
        logging.warning("Failed: %s (%s)", url, e)
        return {"url": url, "error": str(e)}


def main():
    parser = argparse.ArgumentParser(description="HTTP recon scanner")
    parser.add_argument("-l", "--list", required=True)
    parser.add_argument("-o", "--output", default="scan_report.json")
    parser.add_argument("-t", "--threads", type=int, default=10)
    args = parser.parse_args()

    targets = load_targets(args.list)
    logging.info("Loaded %d targets", len(targets))

    results = []
    with ThreadPoolExecutor(max_workers=args.threads) as executor:
        futures = {executor.submit(check_host, t): t for t in targets}
        for future in as_completed(futures):
            results.append(future.result())

    report = {
        "scan_time": datetime.now(timezone.utc).isoformat(),
        "total": len(results),
        "results": results,
    }
    with open(args.output, "w") as f:
        json.dump(report, f, indent=2)

    logging.info("Done. Report saved to %s", args.output)


if __name__ == "__main__":
    main()
```

Run it with:

```bash
python3 scanner.py -l targets.txt -o report.json -t 20
```

#### 8 Summary

Python is where recon scripts stop being "a sequence of tool calls" and start being real tools — with structured input/output, concurrency, retries, and error handling. It's the natural language for anything that needs to reason about data rather than just move it from one command to another. The next section, 24.3, shifts focus to PowerShell — necessary the moment an engagement moves onto Windows infrastructure.

### 24.3 PowerShell Basics

Bash and Python cover Linux-centric tooling well, but the moment an engagement touches a Windows domain — Active Directory enumeration, lateral movement, post-exploitation on a compromised Windows host — PowerShell becomes unavoidable. It is the native automation and administration language of Windows, which means it's also the native language of Windows attack surface. Kali includes PowerShell Core (`pwsh`), which lets you write and test PowerShell logic directly from your Kali box before it ever touches a target.

#### 1 Why PowerShell Matters for Pentesters

Windows administrators use PowerShell to manage everything from user accounts to scheduled tasks to remote systems. That means:

- Almost every Windows enterprise environment has PowerShell available on every host by default.
- Living-off-the-land (LOLBin) techniques rely heavily on abusing legitimate PowerShell functionality rather than dropping custom binaries that get flagged by AV/EDR.
- Post-exploitation frameworks (Empire, PowerSploit, Covenant) are built around PowerShell precisely because it's already trusted and present on the target.

Understanding PowerShell isn't optional if Windows is in scope — it's the difference between running someone else's canned script and understanding exactly what that script does to a target system before you run it.

#### 2 Installing and Running PowerShell on Kali

```bash
sudo apt update
sudo apt install -y powershell
pwsh
```

This gives you a real PowerShell Core session on Kali, useful for writing and syntax-testing scripts locally, connecting to remote Windows hosts via WinRM, and parsing Windows-native output formats (like event logs exported as XML/CSV) without needing a Windows VM for every step.

#### 3 Core Syntax Fundamentals

**Variables and types**

```powershell
$target = "192.168.1.10"
$ports = @(80, 443, 3389)
$credentials = Get-Credential
```

PowerShell is object-oriented at its core — variables hold typed .NET objects, not just strings, which matters once you start piping output between commands.

**Cmdlets — the Verb-Noun pattern**

PowerShell commands follow a strict `Verb-Noun` naming convention: `Get-Process`, `Test-Connection`, `Invoke-WebRequest`, `New-Item`. This predictability makes it easy to discover functionality:

```powershell
Get-Command -Verb Get -Noun *Process*
Get-Help Test-Connection -Examples
```

**The pipeline**

PowerShell's pipeline passes full objects between commands, not just text — a major difference from Bash:

```powershell
Get-Process | Where-Object { $_.CPU -gt 100 } | Sort-Object CPU -Descending
```

This means you can filter and sort by actual object properties (`.CPU`, `.Name`, `.Id`) rather than parsing text columns, which is far less fragile than the equivalent Bash/`awk` approach.

**Conditionals and loops**

```powershell
if ($response.StatusCode -eq 200) {
    Write-Output "Target is reachable"
} else {
    Write-Output "Target returned $($response.StatusCode)"
}

foreach ($port in $ports) {
    Test-NetConnection -ComputerName $target -Port $port
}
```

**Functions**

```powershell
function Test-HostAlive {
    param(
        [Parameter(Mandatory = $true)]
        [string]$ComputerName
    )
    $result = Test-Connection -ComputerName $ComputerName -Count 1 -Quiet
    return $result
}

Test-HostAlive -ComputerName "10.10.10.5"
```

#### 4 Network and Recon Cmdlets

**Basic connectivity and port testing**

```powershell
Test-Connection -ComputerName $target -Count 2
Test-NetConnection -ComputerName $target -Port 445
```

**Web requests**

```powershell
$response = Invoke-WebRequest -Uri "https://example.com" -UseBasicParsing
$response.StatusCode
$response.Headers["Server"]
```

`Invoke-RestMethod` is the equivalent for JSON APIs — it automatically deserializes JSON responses into PowerShell objects:

```powershell
$data = Invoke-RestMethod -Uri "https://api.example.com/status" -Method Get
$data.status
```

**Simple port scan loop**

```powershell
$targetHost = "10.10.10.5"
$ports = 20..1024

foreach ($port in $ports) {
    $connection = New-Object System.Net.Sockets.TcpClient
    try {
        $connection.Connect($targetHost, $port)
        if ($connection.Connected) {
            Write-Output "Port $port is OPEN"
            $connection.Close()
        }
    } catch {
        # Port closed or filtered — no output
    }
}
```

This is deliberately basic — in a real engagement you'd use `nmap` from Kali for actual port scanning. This kind of script matters when you're already executing PowerShell *on* a compromised or client-provided Windows host and need to enumerate its internal network without introducing new binaries.

#### 5 Active Directory Enumeration Basics

Once you have a foothold on a domain-joined host, PowerShell's AD-aware cmdlets (available when the `ActiveDirectory` module is present, or via raw .NET/ADSI calls when it isn't) are core to enumeration:

```powershell
# Requires RSAT / ActiveDirectory module
Get-ADUser -Filter * -Properties MemberOf | Select-Object Name, MemberOf
Get-ADGroupMember -Identity "Domain Admins"
```

Where the module isn't installed (common on a plain workstation), the same information is reachable via raw ADSI/.NET calls that don't require any extra module:

```powershell
$searcher = New-Object System.DirectoryServices.DirectorySearcher
$searcher.Filter = "(objectClass=user)"
$results = $searcher.FindAll()
foreach ($result in $results) {
    $result.Properties["samaccountname"]
}
```

This ADSI approach is worth knowing specifically because it works without installing anything on the target — a meaningful consideration for OPSEC and for not modifying a client's environment beyond what's authorized.

#### 6 Execution Policy and Why It Matters

Windows restricts script execution by default via `ExecutionPolicy`. Understanding this is essential both for legitimate administration work and for understanding how engagements actually deliver PowerShell payloads:

```powershell
Get-ExecutionPolicy
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
```

`-Scope Process` limits the change to the current session only, rather than altering system-wide policy — the responsible default when you need to run a script temporarily during an authorized engagement. Be aware that execution policy is a convenience guardrail, not a security boundary — it's trivially bypassed (`-ExecutionPolicy Bypass`, encoded commands via `-EncodedCommand`, or simply pasting commands directly into an interactive session), which is exactly why relying on it as your only defense is a common finding in security assessments.

#### 7 Logging and Output for Reporting

Just as with Bash and Python scripts, PowerShell output should be captured and structured for later use:

```powershell
$logFile = "recon_$(Get-Date -Format yyyyMMdd_HHmmss).log"
Start-Transcript -Path $logFile

# ... script logic ...

Stop-Transcript
```

`Start-Transcript` captures the full console session — input and output — which is useful both for your own record-keeping and, in an authorized red-team context, for demonstrating exactly what actions were taken during an engagement.

For structured output, PowerShell's native object pipeline exports cleanly to both CSV and JSON:

```powershell
$results | Export-Csv -Path "results.csv" -NoTypeInformation
$results | ConvertTo-Json -Depth 4 | Out-File "results.json"
```

PowerShell isn't optional knowledge for anyone doing Windows-inclusive engagements — it's the language the target environment itself is built on. ### 24.4 API Automation

Modern security work increasingly happens against APIs rather than raw hosts. Bug bounty platforms expose program and scope data through APIs. Cloud providers expose their entire infrastructure surface through APIs. Threat intelligence feeds, certificate transparency logs, DNS databases, and even your own custom tooling are all reachable — and automatable — through REST endpoints. This section covers how to script against them reliably.

#### 1 Why API Automation Matters

Manually checking a bug bounty platform's dashboard for scope changes, or manually querying a certificate transparency log for new subdomains, doesn't scale. APIs let you:

- Pull structured, machine-readable data instead of scraping HTML
- Schedule recurring checks (new scope, new subdomains, new CVEs) without manual effort
- Chain multiple data sources together programmatically — e.g., pull scope from a bounty platform, then feed it directly into a recon script

The core skills are the same regardless of which specific API you're calling: authentication, request construction, pagination, rate limiting, and response handling.

#### 2 REST API Fundamentals

Nearly every security-relevant API follows the same shape: HTTP requests (`GET`, `POST`, `PUT`, `DELETE`) against URL endpoints, with JSON request/response bodies and authentication via headers.

**Basic authenticated request in Python:**

```python
import requests

API_TOKEN = "your_api_token_here"  # Load from environment, never hardcode
headers = {
    "Authorization": f"Bearer {API_TOKEN}",
    "Accept": "application/json",
}

response = requests.get("https://api.example.com/v1/programs", headers=headers)
response.raise_for_status()
data = response.json()
```

Never hardcode API tokens in scripts you intend to keep, share, or commit to a repository. Load them from environment variables or a local `.env` file excluded via `.gitignore`:

```python
import os

API_TOKEN = os.environ.get("BOUNTY_API_TOKEN")
if not API_TOKEN:
    raise EnvironmentError("BOUNTY_API_TOKEN not set")
```

```bash
export BOUNTY_API_TOKEN="your_token_here"
python3 script.py
```

#### 3 Common Authentication Patterns

**Bearer token (most common):**

```python
headers = {"Authorization": f"Bearer {token}"}
```

**API key in header:**

```python
headers = {"X-API-Key": api_key}
```

**Basic auth:**

```python
response = requests.get(url, auth=(username, api_key))
```

**OAuth 2.0 (more involved — used by platforms like Google Cloud, some bug bounty platforms):**

```python
import requests

def get_oauth_token(client_id: str, client_secret: str, token_url: str) -> str:
    resp = requests.post(token_url, data={
        "grant_type": "client_credentials",
        "client_id": client_id,
        "client_secret": client_secret,
    })
    resp.raise_for_status()
    return resp.json()["access_token"]

token = get_oauth_token(CLIENT_ID, CLIENT_SECRET, "https://api.example.com/oauth/token")
headers = {"Authorization": f"Bearer {token}"}
```

OAuth tokens typically expire — production scripts should cache the token with its expiry time and refresh automatically rather than requesting a new one on every call.

#### 4 Handling Pagination

Most APIs paginate list responses rather than returning everything at once. Two common patterns:

**Offset/limit pagination:**

```python
def get_all_results(base_url: str, headers: dict) -> list[dict]:
    results = []
    offset = 0
    limit = 100
    while True:
        resp = requests.get(base_url, headers=headers, params={"offset": offset, "limit": limit})
        resp.raise_for_status()
        page = resp.json()
        results.extend(page["data"])
        if len(page["data"]) < limit:
            break
        offset += limit
    return results
```

**Cursor/token-based pagination:**

```python
def get_all_results_cursor(base_url: str, headers: dict) -> list[dict]:
    results = []
    cursor = None
    while True:
        params = {"cursor": cursor} if cursor else {}
        resp = requests.get(base_url, headers=headers, params=params)
        resp.raise_for_status()
        page = resp.json()
        results.extend(page["data"])
        cursor = page.get("next_cursor")
        if not cursor:
            break
    return results
```

Always check the specific API's documentation — the exact field names (`next_cursor`, `next_page_token`, `has_more`) vary, but the loop structure is nearly identical across platforms.

#### 5 Rate Limiting and Backoff

APIs enforce rate limits, typically communicated via `429 Too Many Requests` responses and a `Retry-After` header. Respecting these is both good practice and often a condition of API access:

```python
import time

def request_with_backoff(url: str, headers: dict, max_retries: int = 5) -> requests.Response:
    for attempt in range(max_retries):
        resp = requests.get(url, headers=headers)
        if resp.status_code == 429:
            retry_after = int(resp.headers.get("Retry-After", 2 ** attempt))
            time.sleep(retry_after)
            continue
        resp.raise_for_status()
        return resp
    raise RuntimeError(f"Max retries exceeded for {url}")
```

Exponential backoff (`2 ** attempt`) as a fallback when no `Retry-After` header is present is a reasonable default — it avoids hammering an API that's already telling you to slow down.

#### 6 Working with Bug Bounty Platform APIs

Both major platforms expose APIs for pulling program scope, submitted reports, and program updates programmatically. The specific endpoints and auth schemes differ between platforms and change over time, so always check current API documentation before building against it — but the general pattern for a scope-monitoring script looks like this:

```python
import requests
import json
import os

def get_program_scope(api_token: str, program_handle: str) -> list[dict]:
    headers = {"Authorization": f"Bearer {api_token}"}
    url = f"https://api.example-platform.com/v1/programs/{program_handle}/scope"
    resp = requests.get(url, headers=headers)
    resp.raise_for_status()
    return resp.json()["scope_items"]

def diff_scope(old_scope: list[dict], new_scope: list[dict]) -> list[dict]:
    old_assets = {item["asset_identifier"] for item in old_scope}
    return [item for item in new_scope if item["asset_identifier"] not in old_assets]

api_token = os.environ["BOUNTY_API_TOKEN"]
current_scope = get_program_scope(api_token, "example-program")

scope_file = "scope_cache.json"
if os.path.exists(scope_file):
    with open(scope_file) as f:
        previous_scope = json.load(f)
    new_items = diff_scope(previous_scope, current_scope)
    if new_items:
        print(f"[+] {len(new_items)} new scope items found!")
        for item in new_items:
            print(f"  - {item['asset_identifier']}")

with open(scope_file, "w") as f:
    json.dump(current_scope, f, indent=2)
```

Running this on a daily cron job (see 24.1.6) turns scope monitoring from a manual dashboard check into an automatic alert — you find out about new scope the same day it's added, which matters in bug bounty because new assets are frequently where the easiest findings live before other researchers get to them.

#### 7 Working with Cloud Provider APIs

Cloud recon (finding exposed S3 buckets, misconfigured storage, overly permissive IAM roles) increasingly relies on scripting directly against provider APIs/SDKs rather than generic web requests:

```python
import boto3
from botocore.exceptions import ClientError

def check_public_buckets(bucket_names: list[str]) -> list[dict]:
    s3 = boto3.client("s3")
    results = []
    for bucket in bucket_names:
        try:
            acl = s3.get_bucket_acl(Bucket=bucket)
            public = any(
                grant["Grantee"].get("URI", "").endswith("AllUsers")
                for grant in acl["Grants"]
            )
            results.append({"bucket": bucket, "public": public})
        except ClientError as e:
            results.append({"bucket": bucket, "error": str(e)})
    return results
```

This example uses AWS's `boto3` SDK, which wraps the raw API in a much friendlier interface than hand-rolling authenticated requests — most major cloud providers (AWS, Azure, GCP) offer equivalent official SDKs, and using them is strongly preferable to reimplementing their signature/auth schemes by hand.

#### 8 Building a Reusable API Client Wrapper

For any API you'll query repeatedly, wrapping it in a small class pays off quickly — centralizing auth, retries, and base URL handling:

```python
import requests
import time


class APIClient:
    def __init__(self, base_url: str, token: str):
        self.base_url = base_url.rstrip("/")
        self.session = requests.Session()
        self.session.headers.update({"Authorization": f"Bearer {token}"})

    def get(self, endpoint: str, params: dict | None = None) -> dict:
        url = f"{self.base_url}/{endpoint.lstrip('/')}"
        for attempt in range(5):
            resp = self.session.get(url, params=params, timeout=10)
            if resp.status_code == 429:
                time.sleep(int(resp.headers.get("Retry-After", 2 ** attempt)))
                continue
            resp.raise_for_status()
            return resp.json()
        raise RuntimeError(f"Failed after retries: {url}")


client = APIClient("https://api.example.com/v1", token=os.environ["API_TOKEN"])
programs = client.get("/programs")
```

This pattern — a thin client class around `requests.Session` — scales well: every new API you automate against just needs a new instance of the same class, rather than rewriting auth and retry logic each time.

#### 9 Summary

API automation is the layer that connects your tooling to the outside world's data — bounty scope, cloud metadata, threat intel, certificate logs — without manual dashboard-checking. The core skills (auth handling, pagination, backoff, response parsing) transfer across virtually every API you'll encounter. ### 24.5 Custom Recon Scripts

Reconnaissance is the highest-volume, most repetitive phase of any security assessment — and consequently the phase where custom automation pays off the most. This section brings together Bash (24.1), Python (24.2), PowerShell (24.3), and API automation (24.4) into complete, working recon scripts you can adapt directly to your own workflow.

The goal here isn't to replace dedicated recon frameworks — it's to understand exactly what they're doing under the hood, and to be able to build your own the moment an engagement needs something a public tool doesn't provide out of the box.

#### 1 Designing a Recon Pipeline

A useful recon pipeline generally moves through the same stages regardless of target type:

1. **Asset discovery** — subdomains, IP ranges, cloud assets
2. **Liveness/filtering** — which discovered assets actually respond
3. **Service/technology fingerprinting** — what's running on live assets
4. **Change detection** — what's new since the last run
5. **Output/reporting** — structured data ready for the next phase (vuln scanning, manual testing)

Building your own pipeline means each stage is a swappable, inspectable script — rather than a monolithic black box — so when a specific target needs a step no public tool provides, you can slot it in without reworking everything else.

#### 2 Stage 1 — Subdomain and Asset Discovery (Bash)

```bash
#!/usr/bin/env bash
set -euo pipefail

domain="${1:?Usage: $0 <domain>}"
outdir="recon/${domain}"
mkdir -p "$outdir"

echo "[*] Passive subdomain enumeration..."
subfinder -d "$domain" -silent -o "${outdir}/subfinder.txt"
assetfinder --subs-only "$domain" > "${outdir}/assetfinder.txt"

echo "[*] Merging and deduplicating..."
cat "${outdir}/subfinder.txt" "${outdir}/assetfinder.txt" \
    | sort -u > "${outdir}/all_subdomains.txt"

echo "[+] $(wc -l < "${outdir}/all_subdomains.txt") unique subdomains found"
```

Using two independent passive sources (`subfinder`, `assetfinder`) and merging results catches assets that any single tool would miss on its own — a pattern worth applying to every discovery stage, not just subdomains.

#### 3 Stage 2 — Liveness Filtering (Bash + httpx)

```bash
echo "[*] Probing for live HTTP(S) hosts..."
httpx -l "${outdir}/all_subdomains.txt" \
    -silent -follow-redirects \
    -status-code -title -tech-detect \
    -o "${outdir}/live_hosts.txt"

echo "[+] $(wc -l < "${outdir}/live_hosts.txt") live hosts identified"
```

`httpx`'s `-tech-detect` flag is worth calling out specifically — identifying the technology stack (CMS, framework, server software) at the liveness-check stage means you don't need a separate fingerprinting pass later; it comes for free.

#### 4 Stage 3 — Structured Fingerprinting (Python)

Bash is fine for calling `httpx`, but once you want to reason about the results — filter by technology, group by status code, flag anomalies — Python takes over:

```python
#!/usr/bin/env python3
"""Parse httpx output and produce a structured fingerprint report."""

import json
import re
import sys
from collections import defaultdict

def parse_httpx_line(line: str) -> dict | None:
    # Matches lines like: https://sub.example.com [200] [Login Page] [nginx,PHP]
    match = re.match(
        r"^(\S+)\s+\[(\d+)\](?:\s+\[([^\]]*)\])?(?:\s+\[([^\]]*)\])?", line
    )
    if not match:
        return None
    url, status, title, tech = match.groups()
    return {
        "url": url,
        "status": int(status),
        "title": title or "",
        "technologies": [t.strip() for t in (tech or "").split(",") if t.strip()],
    }

def build_report(filepath: str) -> dict:
    entries = []
    with open(filepath) as f:
        for line in f:
            parsed = parse_httpx_line(line.strip())
            if parsed:
                entries.append(parsed)

    by_status = defaultdict(list)
    by_tech = defaultdict(list)
    for entry in entries:
        by_status[entry["status"]].append(entry["url"])
        for tech in entry["technologies"]:
            by_tech[tech].append(entry["url"])

    return {
        "total_hosts": len(entries),
        "by_status_code": dict(by_status),
        "by_technology": dict(by_tech),
        "entries": entries,
    }

if __name__ == "__main__":
    report = build_report(sys.argv[1])
    with open("fingerprint_report.json", "w") as f:
        json.dump(report, f, indent=2)
    print(f"[+] {report['total_hosts']} hosts parsed. Report saved.")
```

Grouping by technology (`by_tech`) is particularly useful in practice — if a recon run turns up 40 hosts running the same outdated CMS version, that's an immediate, high-value signal worth investigating before anything else.

#### 5 Stage 4 — Change Detection Across Runs

The highest-value recon automation isn't a single scan — it's noticing what changed since the last one. New subdomains and newly-appearing services are disproportionately likely to be untested and misconfigured:

```python
#!/usr/bin/env python3
"""Diff current recon results against the previous run."""

import json
import os
from datetime import datetime, timezone

def load_previous(cache_file: str) -> set[str]:
    if not os.path.exists(cache_file):
        return set()
    with open(cache_file) as f:
        return set(json.load(f))

def save_current(cache_file: str, current: set[str]) -> None:
    with open(cache_file, "w") as f:
        json.dump(sorted(current), f, indent=2)

def diff_and_report(domain: str, current_subdomains: list[str]) -> None:
    cache_file = f"recon/{domain}/.subdomain_cache.json"
    previous = load_previous(cache_file)
    current = set(current_subdomains)

    new_subs = current - previous
    removed_subs = previous - current

    if new_subs:
        print(f"[+] {len(new_subs)} NEW subdomains since last run:")
        for sub in sorted(new_subs):
            print(f"    + {sub}")

    if removed_subs:
        print(f"[-] {len(removed_subs)} subdomains no longer resolving:")
        for sub in sorted(removed_subs):
            print(f"    - {sub}")

    if not new_subs and not removed_subs:
        print("[*] No changes since last run.")

    save_current(cache_file, current)

    log_entry = {
        "timestamp": datetime.now(timezone.utc).isoformat(),
        "new_count": len(new_subs),
        "removed_count": len(removed_subs),
    }
    with open(f"recon/{domain}/change_log.jsonl", "a") as f:
        f.write(json.dumps(log_entry) + "\n")

if __name__ == "__main__":
    import sys
    domain = sys.argv[1]
    with open(f"recon/{domain}/all_subdomains.txt") as f:
        subs = [line.strip() for line in f if line.strip()]
    diff_and_report(domain, subs)
```

Appending each run's summary to a `.jsonl` change log (rather than overwriting a single file) gives you a full history over time — useful both for spotting patterns (a program that adds new subdomains every Monday, for instance) and for demonstrating ongoing monitoring in a report.

#### 6 Stage 5 — Tying It Together with a Master Script

A single entry-point script that runs the full pipeline and schedules itself via cron:

```bash
#!/usr/bin/env bash
set -euo pipefail

domain="${1:?Usage: $0 <domain>}"
scriptdir="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
outdir="recon/${domain}"
mkdir -p "$outdir" logs

log_file="logs/$(date +%Y%m%d_%H%M%S)_${domain}.log"
exec > >(tee -a "$log_file") 2>&1
trap 'echo "[!] Pipeline failed at line $LINENO"; exit 1' ERR

echo "[*] === Recon pipeline started for ${domain} ==="

bash "${scriptdir}/01_discover.sh" "$domain"
bash "${scriptdir}/02_liveness.sh" "$domain"
python3 "${scriptdir}/03_fingerprint.py" "${outdir}/live_hosts.txt"
python3 "${scriptdir}/04_diff.py" "$domain"

echo "[+] === Pipeline complete for ${domain} ==="
```

Scheduled with cron (see 24.1.6) for daily or weekly runs against programs you're actively monitoring:

```
0 4 * * * /home/user/recon/run_pipeline.sh example.com
```

#### 7 API-Driven Scope Expansion

Combining this pipeline with the bug bounty API patterns from 24.4 closes the loop completely — instead of manually feeding domains into the pipeline, pull current in-scope assets directly from the platform API and run recon automatically against anything new:

```python
#!/usr/bin/env python3
"""Pull current scope from a bounty platform and trigger recon on new assets."""

import os
import subprocess
import sys

sys.path.insert(0, os.path.dirname(__file__))
from api_client import APIClient  # the wrapper built in 24.4.8

def main():
    token = os.environ["BOUNTY_API_TOKEN"]
    client = APIClient("https://api.example-platform.com/v1", token)

    scope = client.get(f"/programs/{sys.argv[1]}/scope")
    domains = [item["asset_identifier"] for item in scope["scope_items"]
               if item["asset_type"] == "domain"]

    for domain in domains:
        print(f"[*] Triggering recon pipeline for {domain}")
        subprocess.run(["bash", "run_pipeline.sh", domain], check=True)

if __name__ == "__main__":
    main()
```

This is the complete automation loop described at the start of the chapter: an API call determines what's in scope, a Bash/Python pipeline discovers and fingerprints assets within that scope, and a diffing stage flags exactly what changed — all without manual intervention beyond reviewing the final output.

#### 8 Responsible Use Considerations

A few things worth stating explicitly for any custom recon script you build and run on a schedule:

- **Respect program scope precisely.** An automated pipeline is only as safe as the scope list feeding it — a stale or manually-edited domain list can silently start scanning out-of-scope assets. Pulling scope directly from the platform API (7) reduces this risk by keeping the source of truth authoritative and current.
- **Respect rate limits and program-specific testing restrictions.** Automated, scheduled scanning makes it easy to accidentally exceed a program's stated request-rate limits — build throttling into the pipeline itself rather than relying on manual restraint.
- **Store credentials and API tokens securely**, never in scripts committed to a shared or public repository.
- **Log everything.** The `exec > >(tee -a ...)` pattern from 24.1.5 and the change-log pattern from 5 both exist so that, months later, you can reconstruct exactly what a script did and when — valuable both for your own workflow and for any client or platform that asks.

#### 9 Summary

Custom recon scripts are where the whole chapter converges: Bash for fast tool orchestration, Python for structured data handling and concurrency, PowerShell for anything touching Windows infrastructure, and API automation for keeping the entire pipeline fed with current, authoritative scope data. None of the individual pieces are complicated in isolation — the value comes from chaining them into a pipeline that runs itself, flags only what's new or interesting, and frees you to spend your actual attention on the findings rather than the process of finding them.
