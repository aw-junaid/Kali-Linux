### Chapter 21 – OSINT (Open Source Intelligence)

Open Source Intelligence, commonly abbreviated as **OSINT**, is the practice of collecting, analyzing, and correlating information that is publicly and legally available from open sources. These sources include search engines, social media platforms, public records, WHOIS databases, DNS records, code repositories, breach databases, forums, and countless other publicly accessible repositories of information.

In the context of penetration testing, red teaming, and security research, OSINT forms the foundation of the **reconnaissance** phase. Before an attacker (or an authorized penetration tester) can exploit a target, they first need to understand it — its infrastructure, its employees, its email conventions, its exposed assets, its digital footprint, and its social media presence. OSINT tools automate and streamline this information-gathering process, transforming scattered public data into structured, actionable intelligence.

This chapter covers seven of the most widely used OSINT tools available in Kali Linux and the broader security community. Each tool approaches intelligence gathering from a different angle:

- Some tools focus on **passive infrastructure reconnaissance** (domains, subdomains, IP ranges, emails).
- Some tools provide **full-spectrum automated reconnaissance frameworks** that aggregate dozens of data sources.
- Some tools are built for **visual link analysis**, mapping relationships between people, organizations, and infrastructure.
- Some tools specialize in **username and identity correlation** across social platforms.
- Some tools are purpose-built for a single platform, such as Google/Gmail account reconnaissance or email-based account discovery.

Mastering these tools — their installation, syntax, command-line options, and practical usage — is essential for any security professional conducting authorized penetration tests, red team engagements, bug bounty reconnaissance, or digital footprint audits.

> **Legal and Ethical Notice:** OSINT tools query publicly available data, but the *use* of the gathered intelligence can still carry legal and ethical weight. Always ensure you have explicit written authorization (a signed scope of work, rules of engagement, or bug bounty program terms) before running reconnaissance against any target that is not your own. Unauthorized reconnaissance against third parties, even using "passive" OSINT tools, may violate computer misuse laws, terms of service, or privacy regulations (e.g., GDPR) depending on jurisdiction.

### 21.1 theHarvester

#### 1. Tool Introduction

`theHarvester` is an open-source reconnaissance tool originally developed by **Christian Martorella** and now maintained by the Kali/OSINT community on GitHub. It is designed to gather open-source intelligence about a target domain or organization, including:

- Email addresses
- Subdomains and hostnames
- Employee names
- Open ports and service banners (when active modules are used)
- IP addresses
- URLs

theHarvester works by querying dozens of public data sources — search engines (Bing, Google, DuckDuckGo, Yahoo), PGP key servers, certificate transparency logs (crt.sh), Shodan, VirusTotal, Hunter.io, SecurityTrails, DNSdumpster, and many more — and aggregating the results into a single unified report. It is almost always the **first tool** run in a penetration test's reconnaissance phase because it requires no target interaction (fully passive) and produces a quick overview of an organization's external footprint.

theHarvester ships pre-installed on Kali Linux under `/usr/bin/theHarvester` (or as a Python package `theHarvester`).

###### 1.1 Key Capabilities

- Passive data collection (no direct contact with target infrastructure)
- Aggregation across 15+ public/free sources and additional paid API sources
- DNS brute-forcing and resolution of discovered subdomains
- Screenshot capture of discovered subdomains (via `--screenshot`)
- Output in text, XML, JSON, or HTML formats
- Integration with Shodan for port/banner data

The general syntax for theHarvester is:

```bash

```

#### 4. Command-Line Options

| Flag | Long Form | Description |
|------|-----------|--------------|
| `-d` | `--domain` | Target domain or company name to search (mandatory) |
| `-b` | `--source` | Data source to use: baidu, bevigil, binaryedge, bing, bingapi, bufferoverun, brave, censys, certspotter, criminalip, crtsh, dehashed, dnsdumpster, duckduckgo, fullhunt, github-code, hackertarget, hunter, hunterhow, intelx, leakix, netlas, onyphe, otx, pentesttools, projectdiscovery, rapiddns, rocketreach, securityscorecard, securityTrails, shodan, sitedossier, subdomaincenter, subdomainfinderc99, threatminer, tomba, urlscan, virustotal, yahoo, zoomeye, all |
| `-l` | `--limit` | Limit the number of search results returned (default varies by source) |
| `-S` | `--start` | Start result index (offset) for the search engine query |
| `-p` | `--proxies` | Use proxies configured in `proxies.yaml` for requests |
| `-s` | `--shodan` | Use Shodan to query discovered hosts for open ports/banners |
| `-v` | `--virtual-host` | Perform reverse DNS resolution and identify virtual hosts |
| `-e` | `--dns-server` | Specify a custom DNS server to use for resolution |
| `-t` | `--dns-tld` | Perform a TLD expansion discovery |
| `-r` | `--take-over` | Check for potential subdomain takeover vulnerabilities |
| `-n` | `--dns-lookup` | Enable DNS resolution of all found hosts |
| `-c` | `--dns-brute` | Perform DNS brute-force against the target domain |
| `-f` | `--filename` | Save results to an HTML and/or XML file |
| `-x` | `--dns-expander` | Enable dns expansion so subdomains found in report generate more results |

| `--dns-resolve` | `--dns-resolve` | Perform DNS resolve on all identified hosts |
| `-h` | `--help` | Show help menu |

#### 8. Automation with Bash

theHarvester's output can be piped into other tools or looped across multiple domains for large-scale reconnaissance.

###### 8.1 Multi-Domain Automation Script

```bash
#!/bin/bash
# harvest_multi.sh - Run theHarvester against a list of domains

DOMAIN_LIST="domains.txt"
OUTPUT_DIR="harvester_results"
mkdir -p "$OUTPUT_DIR"

while IFS= read -r domain; do
    echo "[*] Harvesting: $domain"
    theHarvester -d "$domain" -b all -f "$OUTPUT_DIR/${domain}_report"
    sleep 5   # avoid rate limiting
done < "$DOMAIN_LIST"

echo "[+] All domains processed. Results saved in $OUTPUT_DIR/"
```

Run it:
```bash
chmod +x harvest_multi.sh
./harvest_multi.sh
```

###### 8.2 Extracting Only Emails Across All Reports

```bash
#!/bin/bash
# extract_emails.sh - Consolidate emails from all theHarvester JSON reports

for file in harvester_results/*.json; do
    jq -r '.emails[]?' "$file"
done | sort -u > all_emails.txt

echo "[+] Unique emails saved to all_emails.txt"
```

###### 8.3 Chaining with Subdomain Resolution

```bash
#!/bin/bash
# harvest_and_resolve.sh

theHarvester -d example.com -b all -f harvest_out
jq -r '.hosts[]?' harvest_out.json | cut -d: -f1 | sort -u > subdomains.txt
cat subdomains.txt | while read -r sub; do
    ip=$(dig +short "$sub" | tail -n1)
    echo "$sub -> $ip"
done
```

#### 9. Tips and Best Practices

- Always run `-b all` for a comprehensive first pass, then re-run targeted sources if you need higher result limits or fresher data.
- Configure API keys in `api-keys.yaml` (located in the theHarvester config directory) for premium sources like Hunter.io, SecurityTrails, Shodan, and Censys — free sources alone often miss significant data.
- Combine theHarvester's subdomain output with active tools (`httpx`, `nmap`, `nuclei`) for a full reconnaissance-to-scanning pipeline.
- Use the `-c` (DNS brute-force) flag sparingly and only with authorization, as it generates a higher volume of DNS traffic against the target's nameservers.
- Space out large multi-domain scans to avoid being rate-limited or blocked by search engines.
- Store historical scan results with timestamps to track changes in an organization's external footprint over time.

#### 10. Troubleshooting

| Issue | Cause | Resolution |
|-------|-------|------------|
| `No results found` from a source | Source API key missing or rate-limited | Add valid API key to `api-keys.yaml`; wait and retry |
| `Error: 429 Too Many Requests` | Search engine has rate-limited your IP | Use `-p` with proxies configured, or slow down request frequency |
| `ModuleNotFoundError` on launch | Missing Python dependency | Run `pip install -r requirements/base.txt --break-system-packages` |
| Screenshot feature fails | Missing headless browser (e.g., Chromium) dependency | Install `chromium` and required Selenium/Playwright packages |
| Empty results with `-b all` | Network connectivity or DNS issues | Verify internet access with `curl https://bing.com`; check `/etc/resolv.conf` |
| `command not found: theHarvester` | Not installed or not in PATH | Reinstall via `apt install theharvester` or `pipx ensurepath` |

### 21.2 SpiderFoot

#### 1. Tool Introduction

`SpiderFoot` is an open-source automated OSINT reconnaissance framework created by **Steve Micallef**. It is designed to automate the process of gathering intelligence about a target — whether an IP address, domain name, hostname, network subnet, ASN, email address, phone number, username, or person's name — by querying more than **200 individual data sources**, including search engines, DNS records, WHOIS databases, breach databases, threat intelligence feeds, social media, paste sites, dark web indexes, and SSL certificate data.

SpiderFoot is unique among OSINT tools in that it offers both a **command-line interface (`sf.py`)** and a **full-featured web-based GUI (`sf.py -l`)**, allowing analysts to visually explore relationships between discovered entities through an interactive graph. It supports both fully passive scanning (no interaction with the target) and more invasive/active scanning modes (e.g., port scanning, banner grabbing) depending on the modules enabled.

SpiderFoot is commonly used for:
- Attack surface mapping and asset discovery
- Digital footprint / personal OSINT investigations
- Threat intelligence enrichment
- Data breach and leaked credential correlation
- Continuous monitoring of an organization's external exposure

#### 9. Tips and Best Practices

- Use `-u passive` for fully non-intrusive engagements where the client explicitly requires zero direct interaction with their infrastructure.
- Enable API keys for premium modules (Shodan, Hunter.io, SecurityTrails, HaveIBeenPwned) in the web GUI's Settings panel for significantly richer results.
- Use the web GUI's graph visualization to identify relationships that are difficult to spot in flat CLI output.
- Schedule recurring scans (via cron) for continuous attack-surface monitoring and diff the JSON output between runs to detect new exposures.
- Apply correlation rules (`-C`) after scans to automatically surface high-risk findings instead of manually reviewing hundreds of raw results.
- Respect target scope carefully — SpiderFoot's `all` module set includes active modules (port scanning, spidering) that could be considered intrusive.

#### 10. Troubleshooting

| Issue | Cause | Resolution |
|-------|-------|------------|
| `ModuleNotFoundError` | Missing Python dependencies | Run `pip install -r requirements.txt --break-system-packages` |
| Web GUI won't load | Port already in use or firewall blocking | Change port with `-l 127.0.0.1:5002`, check `netstat -tulnp` |
| Scan hangs indefinitely | A module without a timeout is stuck on a slow API | Cancel scan in GUI, disable the specific module, or set module timeouts in settings |

| High memory usage on large scans | Too many modules enabled simultaneously | Use `-m` to select fewer, targeted modules instead of `-u all` |

### 21.3 Maltego

#### 1. Tool Introduction

`Maltego` is a graphical link-analysis and data-mining platform developed by **Paterva (Maltego Technologies)**. Unlike most OSINT tools which are purely command-line based, Maltego is a Java-based desktop application that visually represents relationships between pieces of information as an interactive node graph. Each piece of information (a domain, person, email address, phone number, IP address, company, document, etc.) is represented as an **Entity**, and the process of retrieving related information about an entity is called a **Transform**.

Maltego is widely used by penetration testers, threat intelligence analysts, law enforcement, and investigative journalists because it turns raw, disconnected OSINT data into a visual map of relationships — making it much easier to spot patterns, hidden connections, and pivot points that would be difficult to notice in flat text output.

Maltego ships in several editions:
- **Maltego CE (Community Edition)** – free, limited to 12 results per transform, requires a free Maltego account
- **Maltego Classic / Pro** – paid, higher result limits, more transform hubs
- **Maltego XL** – paid, designed for large graphs (10,000+ entities)

Maltego CE comes pre-installed on Kali Linux under Applications > Information Gathering.

| Maltego won't launch / crashes on startup | Corrupted cache or Java heap issue | Launch with `maltego --clean`, or increase heap with `-J-Xmx4096m` |

### 21.4 Recon-ng

#### 1. Tool Introduction

`Recon-ng` is a full-featured, modular web reconnaissance framework written in Python, developed by **Tim Tomes (LaNMaSteR53)**. It provides a command-line environment intentionally modeled after the **Metasploit Framework**, giving penetration testers a familiar `workspace → module → set options → run` workflow for conducting open-source reconnaissance.

Unlike single-purpose tools, Recon-ng organizes its functionality into independent **modules**, grouped by category: `recon` (data-gathering modules), `discovery`, `import`, `reporting`, and `exploitation`. Each module targets a specific data source (e.g., Shodan, Hackertarget, Whois, VirusTotal, Bing) and stores its findings in a built-in SQLite database tied to the current **workspace**, allowing results from multiple modules to be automatically correlated (e.g., a hostname discovered by one module becomes input for a follow-on module).

Recon-ng ships pre-installed on Kali Linux and supports both free/API-key-based and marketplace-installed modules via its own module marketplace system.

### 21.5 GHunt

#### 1. Tool Introduction

`GHunt` is an open-source OSINT tool created by **mxrch** that specializes exclusively in investigating **Google accounts**. Given a Gmail address, a Google account ID, or other Google-linked identifiers, GHunt can extract a wide range of associated information exposed through Google's own services, including:

- Google account creation date (approximate, via account ID structure)
- Profile name and profile photo
- Associated YouTube channel (subscriptions count, videos, etc., if public)
- Google Maps contributions and reviews (if public)
- Google Calendar events (if the calendar is publicly shared)
- Google Photos albums (if publicly shared)
- Google Play Games/Store activity (if public)

GHunt (version 2.x) is built in Python and requires the user to provide their **own authenticated Google session cookies** to function, since it queries Google's internal APIs the same way a logged-in browser session would — but only accesses data that Google itself already treats as visible/public for the target account, not private data.

GHunt is widely used in OSINT investigations, social engineering target profiling, and by investigators trying to attribute an anonymous Gmail address to a real identity via publicly exposed Google service activity.

#### 9. Tips and Best Practices

- Use a **dedicated, non-personal Google account** for GHunt's authentication cookies rather than your primary account, since repeated automated lookups can trigger Google's abuse detection.

- Cross-reference GHunt results with Sherlock and Holehe for a more complete identity picture — GHunt covers Google's ecosystem, while the other two cover broader platforms.
- Respect privacy: only investigate email addresses within the authorized scope of your engagement or investigation.
- Keep GHunt updated (`git pull` or `pipx upgrade ghunt`) — Google frequently changes internal API endpoints, which can break older GHunt versions.

### 21.6 Sherlock

#### 1. Tool Introduction

`Sherlock` is an open-source username reconnaissance tool maintained by the **Sherlock Project** community (originally created by Siddharth Dushantha). Given a single username, Sherlock checks its existence across **hundreds of social media platforms and websites simultaneously** — including Twitter/X, Instagram, GitHub, Reddit, TikTok, Facebook, Steam, Twitch, YouTube, Pinterest, and hundreds more — by sending HTTP requests to each platform's profile URL pattern and analyzing the response to determine whether an account with that exact username exists.

Sherlock is one of the most widely used tools for **digital footprinting** and **identity correlation** in OSINT investigations. Since many people reuse the same username across multiple platforms, Sherlock allows an investigator to quickly discover the full breadth of a target's online presence starting from just one known handle.

Sherlock is written in Python and comes pre-installed on Kali Linux.

#### 9. Tips and Best Practices

- Use `--print-found` (the default) for quick scans, and reserve `--print-all` for thorough investigations where you want to confirm negative results too.

- Use `--site` to scope searches to only the most relevant platforms when investigating time-sensitive engagements, since a full scan across 400+ sites can take several minutes.

### 21.7 Holehe

#### 1. Tool Introduction

`Holehe` is an open-source OSINT tool created by **megadose**, designed to check whether an email address is registered on numerous websites and online services — such as Twitter, Instagram, Amazon, Spotify, Adobe, GitHub, Snapchat, and dozens more. It works by abusing each platform's public **"forgot password"** or **account registration** endpoints: these endpoints typically respond differently depending on whether an account already exists for the submitted email address, without requiring authentication and, critically, **without sending any notification or reset email to the target**.

This makes Holehe a purely passive, non-intrusive reconnaissance tool from the target's perspective — the account holder is never alerted that a lookup occurred. Holehe is written in Python and is commonly paired with Sherlock (username-based) and GHunt (Google-specific) to build a comprehensive picture of an individual's online presence starting from a single email address.

###### 8.3 Full Identity Correlation Pipeline (Holehe + GHunt + Sherlock)

```bash
#!/bin/bash
# full_identity_recon.sh - Combine all three identity tools for one target

EMAIL="$1"
USERNAME="$2"

echo "[*] Step 1: Holehe email registration check"
holehe "$EMAIL" --only-used

echo ""
echo "[*] Step 2: GHunt Google account investigation"
ghunt email "$EMAIL"

echo ""
echo "[*] Step 3: Sherlock username enumeration"
sherlock "$USERNAME" --print-found

echo ""
echo "[+] Full identity recon complete for $EMAIL / $USERNAME"
```

Run it:
```bash
chmod +x full_identity_recon.sh
./full_identity_recon.sh jane.doe@example.com janedoe_osint
```

#### 9. Tips and Best Practices

- Use `--only-used` for cleaner, report-ready output when you only care about confirmed registrations, not the full negative list.
- Route scans through a proxy or VPN when performing sensitive investigations to avoid revealing your own infrastructure to the platforms being queried.
- Combine Holehe results with Sherlock and GHunt for a triangulated identity profile — email registration data, username presence, and Google-specific activity each reveal different facets of a target's digital footprint.
- Keep Holehe updated regularly (`pip install --upgrade holehe` or `pipx upgrade holehe`), as platforms frequently change their password-reset/registration endpoints, which can silently break individual modules.
- Be mindful of rate limiting — running Holehe against a large batch of emails in rapid succession can trigger CAPTCHAs or temporary IP blocks from some platforms.
- Always respect legal and ethical boundaries — Holehe should only be used against email addresses within the authorized scope of an engagement or investigation.

#### 10. Troubleshooting

| Issue | Cause | Resolution |
|-------|-------|------------|
| Many modules return `[x]` errors | Platform changed its API/endpoint, or rate-limited your IP | Update Holehe to the latest version; wait before retrying |
| `ModuleNotFoundError` on launch | Missing Python dependency | Run `pip install -r requirements.txt --break-system-packages` |
| CSV export file is empty | No accounts found, or write permission issue | Verify output directory is writable; confirm results with default (non-CSV) run first |
| Proxy connection fails | Proxy not running or wrong URL format | Confirm proxy service is active; verify URL format (`socks5://host:port`) |
| Scan is very slow | Checking all 139+ modules sequentially with default timeout | Reduce `-t` timeout value, or verify network latency to target platforms |
| False negative for a known-active account | Platform's detection logic changed and module wasn't updated | Check the project's GitHub issues for that specific module; report if unresolved |
