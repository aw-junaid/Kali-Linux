## Chapter 25 – Reporting

Reporting is the final and, in many professional contexts, the single most important phase of a penetration testing or red team engagement. A security assessment is not measured by the number of shells popped, the number of CVEs triggered, or the cleverness of the exploit chain used to reach domain admin. It is measured by whether the client walks away with a clear, accurate, and actionable understanding of their risk — and whether they can use that understanding to make their environment measurably safer. All of that understanding is delivered through the report.

Kali Linux does not include a single monolithic "reporting suite." Instead, reporting on Kali is a workflow built from several purpose-specific tools and disciplines working together: drafting findings in Markdown for speed and version control, capturing and annotating screenshots as visual proof, collecting and preserving evidence in a chain-of-custody-safe manner, and finally compiling everything into two distinct deliverables — an Executive Report for leadership and a Technical Report for the engineers who will actually fix the issues.

### 25.1 Markdown Reports

Markdown has become the working language of modern penetration testing documentation. Long before a report is ever exported to a polished DOCX or PDF for a client, it exists as a set of plain-text Markdown files, notes, and finding templates being filled in live, in real time, as the engagement unfolds. Understanding why Markdown dominates this space — and how to use it effectively on Kali Linux — is foundational to producing good reports quickly and consistently.

#### 1 Why Penetration Testers Use Markdown

Traditional word processors like Microsoft Word or LibreOffice Writer are heavy, mouse-driven, and slow to work with mid-engagement. A tester who has just achieved remote code execution does not want to switch context into a formatting-heavy document editor, fight with auto-numbering, or wait for a large DOCX file to save. Markdown solves this by being:

- **Fast to write** – plain text with lightweight syntax (`#`, `-`, `` ``` ``, `**bold**`) that can be typed as quickly as the tester can think, directly in a terminal-based editor like Vim, Nano, or a lightweight GUI editor like VS Code or Obsidian.
- **Diffable and version-controllable** – because Markdown is plain text, it works naturally with Git. Teams can track exactly when a finding was added, who added it, and what changed, which is invaluable on multi-tester engagements and for maintaining an audit trail.
- **Portable** – a Markdown file has no proprietary format lock-in. It can be opened, edited, and rendered on any operating system, in any text editor, without needing licensed software.
- **Convertible** – Markdown converts cleanly into DOCX, PDF, HTML, and even LaTeX using tools like Pandoc, meaning the tester only has to write the content once and can generate multiple client-facing formats from that single source of truth.

#### 2 The Structure of a Markdown-Based Report

Most Markdown-based penetration test reports follow a layered structure that separates administrative content from technical findings:

1. **Front Matter** – engagement metadata such as client name, engagement dates, scope (IP ranges, domains, applications), authorized testers, and rules of engagement. This is often stored as YAML front matter at the top of the Markdown file so that reporting tools can parse it programmatically.
2. **Methodology** – a description of the testing framework followed, such as PTES (Penetration Testing Execution Standard), OWASP Testing Guide, or NIST SP 800-115, along with the phases performed (reconnaissance, scanning, exploitation, post-exploitation, reporting).
3. **Findings** – the core of the report, with one Markdown section per vulnerability, following a consistent sub-template (covered in depth in section 25.5 – Technical Reports).
4. **Appendices** – raw tool output, full host inventories, word lists used, and any supplementary data too verbose for the main findings section.

A minimal finding entry in Markdown might look like this:

```markdown
## Finding: SQL Injection in Login Form
**Severity:** Critical (CVSS 9.8)
**Affected Host:** app.client-domain.com

### Description
The login form's `username` parameter is not sanitized before being
passed to a SQL query, allowing authentication bypass and data
extraction.

### Proof of Concept
\`\`\`
sqlmap -u "https://app.client-domain.com/login" --data="username=*&password=x" --batch
\`\`\`

### Remediation
Use parameterized queries or a vetted ORM for all database access.
```

#### 3 Live Note-Taking During an Engagement

Good Markdown reporting starts long before the report is written — it starts with disciplined note-taking during active testing. Testers typically keep a running Markdown log per target or per phase, timestamping entries as they go:

```markdown
## [2026-07-14 09:32] Nmap scan of 10.10.10.0/24 initiated
## [2026-07-14 09:47] Port 445 open on 10.10.10.15 — possible SMB target
## [2026-07-14 10:05] EternalBlue check positive on 10.10.10.15
```

This running log later becomes the raw material from which formal findings are drafted, ensuring nothing discovered during the engagement is forgotten or misremembered by the time the final report is written days or weeks later.

#### 4 Kali Tools That Support Markdown Reporting

- **Sysreptor** – a modern, purpose-built reporting platform where findings are stored as structured Markdown-compatible templates. It supports live collaborative editing (multiple testers writing findings simultaneously), custom CVSS calculators, and direct PDF rendering from the underlying Markdown/HTML content.
- **PwnDoc** – an open-source alternative that stores findings in a MongoDB backend but exposes editing through a Markdown-capable rich text interface, with audit-ready export to Word.
- **Dradis Framework** – aggregates output from dozens of scanning tools (Nmap, Nessus, Burp Suite) automatically into a shared Markdown-based notes structure, cutting down on manual copy-pasting of raw tool output.
- **Obsidian** – not built for pentesting specifically, but widely adopted by individual testers as a fast, local-first Markdown vault for engagement notes, linked findings, and evidence references.
- **VS Code with Markdown Preview / Markdown All in One extensions** – a common lightweight choice for testers who prefer working directly in a code editor with live-rendered previews alongside raw syntax.

#### 5 Converting Markdown to Client Deliverables

Once findings are complete, Pandoc is the standard tool on Kali for transforming the Markdown source into a client-ready file:

```bash
# Convert to a branded Word document using a reference template
pandoc report.md -o report.docx --reference-doc=company_template.docx

# Convert to a print-ready PDF using the LaTeX engine
pandoc report.md -o report.pdf --pdf-engine=xelatex --toc

# Convert to a self-contained HTML report for web viewing
pandoc report.md -o report.html --standalone --css=style.css
```

Using a `--reference-doc` template ensures the exported DOCX automatically inherits the client-approved branding, fonts, and header/footer layout, saving significant manual formatting time compared to writing directly in Word.

#### 6 Common Pitfalls in Markdown Reporting

- **Inconsistent heading levels** across findings, which breaks the auto-generated table of contents when converted to PDF/DOCX
- **Unescaped special characters** (such as underscores in file paths or asterisks in payloads) that render incorrectly unless wrapped in code blocks or escaped
- **Mixing raw HTML into Markdown** inconsistently, which can render fine in one converter but break in another — best practice is to stick to pure Markdown syntax wherever possible
- **Losing the single source of truth** by editing the exported DOCX directly instead of the original Markdown, which causes the two versions to diverge and creates confusion about which is authoritative

#### 7 Summary

Markdown reporting is the backbone of an efficient, auditable, and collaborative penetration testing workflow. It allows testers to capture findings the moment they are discovered, keeps engagement documentation version-controlled and diffable, and — through tools like Pandoc, Sysreptor, and Dradis — converts cleanly into whatever polished final format a client contract requires, without ever forcing the tester to leave the terminal during active testing.

### 25.2 Screenshots

Screenshots are the primary form of visual proof in a penetration testing report. A finding that claims "SQL injection allowed extraction of the entire customer database" is a bare assertion until it is backed by a screenshot showing the actual query, the actual response, and the actual data returned. On Kali Linux, a combination of native GNOME tools, dedicated annotation utilities, and command-line capture tools gives testers full control over how evidence is captured, marked up, and preserved.

#### 1 Why Screenshots Matter So Much in Security Reporting

Unlike many other technical fields, penetration testing findings are often disputed or scrutinized closely by the client's engineering teams, especially when the finding implicates their own code or configuration. A screenshot serves three purposes simultaneously:

1. **Proof** – it demonstrates, beyond argument, that the vulnerability was successfully exploited and not simply theorized.
2. **Reproducibility aid** – it shows the exact command, URL, or payload used, allowing the client to reproduce and later verify the fix.
3. **Communication** – for non-technical stakeholders skimming a technical appendix, a screenshot often communicates impact faster than a paragraph of text.

#### 2 Screenshot Capture Tools on Kali Linux

- **Flameshot** – the most widely used tool among Kali testers for its built-in annotation toolbar. After a region is selected, Flameshot immediately offers arrows, rectangles, freehand drawing, text boxes, a blur/pixelate tool for redaction, and a numbered-step marker, all before the image is even saved. This eliminates the need for a separate image editor in most cases.

```bash
# Launch Flameshot in region-selection capture mode
flameshot gui
```

- **GNOME Screenshot** – the default capture utility on Kali's GNOME desktop, suitable for quick full-screen or window captures but lacking Flameshot's inline annotation capability.
- **scrot** – a minimal, scriptable command-line screenshot tool, useful for automated or headless evidence capture during scripted exploitation chains:

```bash
# Capture the full screen after a 5-second delay, saving with a timestamp
scrot -d 5 -q 100 'evidence_%Y%m%d_%H%M%S.png'
```

- **peek** and **kazam** – lightweight screen recorders used when a single static image cannot capture the proof, such as a timing-based race condition, a multi-step privilege escalation chain, or an interactive shell session that needs to be shown in motion.
- **asciinema** – for terminal-only proof (e.g., a full exploitation sequence run entirely from the command line), asciinema records the terminal session as a lightweight, replayable text-based recording rather than a video file, which is easy to embed in a report and far smaller than a screen recording.

#### 3 Annotation Best Practices

Raw, unannotated screenshots are a common weakness in amateur reports. A strong screenshot workflow always includes markup that directs the reader's eye to exactly what matters:

- **Highlight the proof, not the whole screen.** Circle or box the specific command output, returned data, or shell prompt that proves impact — don't submit a full, cluttered desktop screenshot and expect the reader to find it themselves.
- **Number sequential steps.** When a finding requires multiple screenshots to show a chain of actions (e.g., step 1: initial injection, step 2: extracted table names, step 3: dumped credentials), number each image clearly and reference that number in the corresponding write-up text.
- **Include the addressable context.** Keep the browser's URL bar, the terminal's prompt/hostname, or the tool's title bar visible in the frame so the reader can immediately verify what host or endpoint the evidence corresponds to.
- **Use consistent annotation colors.** Many reporting teams standardize on red for "the vulnerability itself" and yellow or blue for "supporting context," so that readers scanning a large report quickly learn what each color means.

#### 4 Redaction and Scope Discipline

Screenshots frequently capture more than intended — background browser tabs, chat notifications, other client systems outside scope, or genuinely sensitive data (real customer PII, live credentials, financial records) that should not be reproduced verbatim in a report even as "proof."

- Use Flameshot's blur/pixelate tool, or `exiftool`/`ImageMagick`, to redact anything outside the defined scope or anything sensitive that doesn't need to be shown in full to prove the point.
- When a finding involves dumping large amounts of sensitive data (e.g., a full customer table), it is standard practice to show only a small representative excerpt (2–3 rows) rather than the entire dataset, with a note stating the full extent of the exposure without reproducing it in full.
- Never include real, currently-valid credentials in a screenshot without redacting at least part of the password/token — even in an internal report, since reports themselves become sensitive documents that must be protected.

#### 5 Naming and File Organization

Once a report contains dozens of findings, each potentially with multiple screenshots, disciplined naming and folder structure become essential to avoid confusion:

```
[HostID]_[FindingID]_[StepNumber]_[ShortDescription].png

Examples:
WEB01_F003_01_sqli-login-bypass.png
WEB01_F003_02_sqli-dump-users-table.png
DC01_F007_01_zerologon-domain-admin.png
```

Screenshots should live in an evidence folder structure that mirrors the findings themselves, so any team member can locate the supporting image for a given finding without needing to ask the original tester:

```
/evidence
  /WEB01
    /F003_sql_injection
      WEB01_F003_01_sqli-login-bypass.png
      WEB01_F003_02_sqli-dump-users-table.png
```

#### 6 Embedding Screenshots Back into the Markdown Report

Screenshots are referenced directly inside the Markdown finding using standard image syntax, keeping the visual evidence tightly linked to its written description:

```markdown
### Proof of Concept

Step 1 — Authentication bypass using a crafted `username` parameter:

![SQLi login bypass](../evidence/WEB01/F003_sql_injection/WEB01_F003_01_sqli-login-bypass.png)

Step 2 — Extraction of the `users` table via UNION-based injection:

![SQLi data dump](../evidence/WEB01/F003_sql_injection/WEB01_F003_02_sqli-dump-users-table.png)
```

When the report is later converted to PDF or DOCX using Pandoc, these relative image paths are resolved automatically and the images are embedded directly into the final document.

#### 7 Summary

Screenshots turn a written claim into demonstrable proof, and the difference between a mediocre report and a highly credible one often comes down to screenshot discipline: capturing the right frame, annotating it clearly, redacting anything out of scope, naming it consistently, and linking it directly back into the corresponding Markdown finding so the evidence and the narrative never drift apart.

### 25.3 Evidence Collection

Evidence collection is the broader discipline that screenshots (25.2) are only one part of. It covers every artifact produced during an engagement — command output, packet captures, extracted files, log excerpts, timestamps, and hashes — and the practices used to gather, preserve, and organize that material so that it remains trustworthy, reproducible, and defensible if a finding is ever challenged.

#### 1 What Counts as Evidence

A comprehensive evidence set for a single finding typically includes several different artifact types working together:

- **Raw tool output** – the unedited console output of tools such as Nmap, Nikto, Burp Suite, Hydra, and Metasploit, saved directly rather than paraphrased from memory
- **Packet captures (PCAP)** – full or filtered network captures proving, for example, that credentials were transmitted in cleartext or that a specific exploit payload was sent and a specific response received
- **Screenshots and screen recordings** – the visual proof discussed in section 25.2
- **Extracted or exfiltrated data** – copies of files, database dumps, or configuration files retrieved during exploitation, handled carefully and often redacted or hashed rather than stored in full
- **Log excerpts** – relevant lines from client-side logs (web server logs, authentication logs) that corroborate the timing and nature of the tester's actions
- **Timestamps and clock sync records** – ensuring the tester's system clock is synchronized with a reliable time source (e.g., NTP) so that evidence timestamps can be accurately correlated with the client's own log timestamps during post-engagement review

#### 2 Why Chain of Custody Matters in Offensive Security

Chain of custody is usually associated with digital forensics and law enforcement, but it matters in penetration testing too, for several practical reasons:

- **Disputed findings** – if a client's engineering team disputes that a vulnerability is real or exploitable, a well-documented evidence trail with timestamps and hashes settles the disagreement quickly and objectively
- **Legal and contractual protection** – in the rare case that an engagement's authorization or scope is later questioned, precise evidence logs (what was done, when, from which IP, under which signed authorization) protect both the testing firm and the client
- **Regulatory and compliance requirements** – engagements performed for PCI-DSS, HIPAA, or SOC 2 purposes often have explicit requirements around how testing evidence must be retained and protected

#### 3 Practical Chain-of-Custody Steps

- **Log the exact time, host, and command** used to generate each piece of evidence, ideally in the same running Markdown log described in section 25.1
- **Hash every collected file immediately** upon capture, using SHA-256, to create a fingerprint that proves the file has not been altered between capture and final report submission:

```bash
sha256sum extracted_database.sql > extracted_database.sql.sha256
```

- **Store evidence in an access-controlled, encrypted location** for the duration of the engagement, such as a LUKS-encrypted volume or an encrypted cloud storage bucket accessible only to the assigned testing team
- **Define and follow a retention policy** agreed upon with the client in the statement of work — many engagements require evidence to be securely destroyed (not just deleted, but cryptographically wiped) after a fixed retention period, commonly 30 to 90 days after report delivery

#### 4 Tools for Evidence Collection on Kali Linux

- **script** – a classic Unix utility that records an entire terminal session, including every command typed and every line of output produced, to a plain-text typescript file. This provides a verbatim, tamper-evident record of an entire exploitation sequence:

```bash
script -a evidence_session_$(date +%Y%m%d_%H%M%S).log
# ... perform exploitation ...
exit
```

- **asciinema** – similar in spirit to `script` but records the session as a structured, replayable format that can be played back at actual speed later, which is particularly persuasive evidence for timing-sensitive exploits
- **Wireshark and tcpdump** – used to capture network traffic supporting a finding, such as proving that an application transmits session tokens over unencrypted HTTP:

```bash
tcpdump -i eth0 -w evidence_capture_$(date +%Y%m%d_%H%M%S).pcap
```

- **exiftool** – used both to inspect metadata on collected evidence files (confirming capture timestamps, camera/device info on any photographed physical security evidence) and to strip metadata before evidence is shared externally, preventing accidental disclosure of internal tester system information
- **CherryTree, Dradis, and Sysreptor** – centralized note-taking and evidence-aggregation platforms that automatically timestamp entries as evidence is logged, reducing reliance on the tester's memory for exact sequencing
- **Metasploit's `loot` and session logging features** – Metasploit automatically stores extracted credentials, hashes, and downloaded files in its loot database, which can be exported directly as part of the evidence set for a given engagement

#### 5 Organizing the Evidence Repository

A consistent, predictable directory structure is critical once an engagement produces evidence across dozens of hosts and findings. A typical structure separates evidence by host, then by category:

```
/evidence
  /host_10.10.10.5
    /screenshots
    /pcap
    /command_output
    /extracted_files
    /hashes
  /host_10.10.10.15
    /screenshots
    /pcap
    /command_output
    /extracted_files
    /hashes
```

This structure ensures that when the final report is written, every claim made in a finding can be traced back, folder by folder, to a specific, verifiable piece of supporting evidence — which is exactly what a technical reader or an auditor will expect to be able to do.

#### 6 Handling Sensitive Extracted Data

Evidence collection frequently produces genuinely sensitive material — password hashes, PII, financial records, proprietary source code. Best practice for handling this material includes:

- Storing extracted sensitive data encrypted at rest, separate from the general evidence folder, with access restricted to only the lead tester or engagement manager
- Referencing the existence and extent of sensitive data in the report ("a full dump of the `customers` table, containing 42,000 records including names, emails, and hashed passwords, was retrieved") without embedding the full dataset in the report itself
- Securely destroying extracted sensitive data at the end of the agreed retention period, using tools like `shred` or full-disk cryptographic erasure, and documenting that destruction for the client's records

#### 7 Summary

Evidence collection is the connective tissue between "something was found" and "something can be proven, reproduced, and trusted." A disciplined approach — capturing raw output, hashing files immediately, recording full terminal sessions, capturing corroborating network traffic, and organizing everything into a predictable folder structure — ensures that the final report, whether Executive or Technical, rests on a foundation that can withstand scrutiny from even the most skeptical client engineering team.

### 25.4 Executive Reports

The Executive Report is the first — and often the only — part of a penetration testing deliverable that senior leadership will actually read. CISOs, IT directors, compliance officers, and board members are not evaluating the report to learn how a SQL injection payload was crafted; they are evaluating it to answer one question: *what is our risk, and what should we do about it?* Writing this section well requires a completely different set of skills than writing the technical findings — it is closer to business communication than technical documentation.

#### 1 Purpose and Audience

Executive stakeholders typically skim, rather than read, technical appendices. The Executive Report must therefore be able to stand entirely on its own: a reader who reads only these two or three pages, and nothing else in the full deliverable, should walk away with a complete and accurate understanding of:

- What was tested, and over what time period
- How risky the environment currently is, in relative terms
- What the most serious problems are, described in business language
- What could realistically happen if those problems are not fixed
- What the organization should prioritize doing next

Because this audience often includes people from finance, legal, and general management — not just IT — the writing must assume no prior security knowledge whatsoever.

#### 2 Core Components of an Executive Report

1. **Engagement Summary** – a short, plain-language restatement of the scope and purpose of the test: what type of assessment was performed (e.g., external network penetration test, web application assessment, red team simulation), the dates it was conducted, and who authorized it.

2. **Overall Risk Rating** – a single, unambiguous statement of the organization's current security posture based on the aggregate of all findings, typically expressed on a simple scale such as Critical / High / Moderate / Low. This single rating is often the only number a board member will remember from the entire engagement.

3. **Key Findings Summary** – a curated selection of the three to five most impactful issues, described entirely in terms of business consequence rather than technical mechanism. Compare these two descriptions of the same vulnerability:

   - *Technical framing (wrong for this section):* "SQL injection was identified in the `login.php` script's `user` GET parameter, allowing UNION-based extraction of the `users` table."
   - *Executive framing (correct for this section):* "A flaw in the company's customer login page could allow an attacker to view and steal the personal information of every registered customer, without needing a valid account."

4. **Business Impact** – translating findings into concrete potential consequences: financial loss (fraud, ransomware payouts, incident response costs), regulatory penalties (GDPR, HIPAA, PCI-DSS fines), reputational damage (customer trust, media coverage), and operational disruption (downtime, halted production).

5. **Strategic Recommendations** – high-level priorities for investment and action, phrased at the level a budget-approving executive can act on, such as "invest in a centralized patch management program" rather than "apply KB5034441 to all Windows Server 2019 hosts."

6. **Findings Severity Distribution** – a simple visual, typically a bar chart or donut chart, showing the count of findings at each severity level (Critical/High/Medium/Low/Informational), giving an at-a-glance sense of scale without requiring the reader to count findings themselves.

#### 3 Writing Style for Executive Audiences

The single most common mistake junior testers make when writing this section is failing to fully translate technical detail into business language. Concrete guidelines to avoid this:

- **Never use tool names, CVE identifiers, payloads, or command-line syntax** in this section — none of it belongs here, no matter how impressive the exploit chain was
- **Lead with impact, not method.** State what could happen before, or instead of, how it was technically achieved
- **Keep it short.** Regardless of how many pages the technical findings section runs to, the executive summary should typically remain one to two pages
- **Use active, concrete language.** "An attacker could take control of the finance server" is stronger and clearer than "the finance server was found to be vulnerable to a critical remote code execution vulnerability"
- **Avoid hedging language and excessive qualifiers.** Executives need clear signal, not a technically-precise but noncommittal wall of caveats
- **Use visuals generously.** A risk heat map or severity distribution chart communicates scale in two seconds what a paragraph of text would take a minute to convey

#### 4 A Sample Structure in Markdown

```markdown
## Executive Summary

### Engagement Overview
Between July 1–14, 2026, [Firm Name] conducted an external network
penetration test of [Client Name]'s internet-facing infrastructure,
at the request of the Chief Information Security Officer.

### Overall Risk Rating: **Critical**

### Key Findings
1. Customer database fully exposed via an unauthenticated web
   application flaw, affecting all 42,000 registered users.
2. A misconfigured remote access service allowed full administrative
   control of the internal network from the public internet.
3. Password policies across the organization permitted trivially
   guessable passwords, contributing to multiple account compromises.

### Business Impact
These issues collectively expose [Client Name] to significant risk
of a large-scale data breach, associated regulatory penalties under
[applicable regulation], and potential operational disruption from
ransomware deployment via the compromised remote access pathway.

### Recommendations
- Prioritize immediate remediation of the two Critical findings above
- Establish a formal patch management program
- Enforce multi-factor authentication for all remote access
```

#### 5 Tools Used to Produce Executive Reports

- **Sysreptor / PwnDoc / Dradis Framework** – most modern reporting platforms auto-generate a draft executive summary from the severity metadata attached to each technical finding, which the lead tester then rewrites and refines for tone and business relevance rather than accepting verbatim
- **LibreOffice Impress or Microsoft PowerPoint** – executive findings are frequently also condensed into a short slide deck for a live readout meeting, since many organizations prefer a verbal walkthrough with leadership in addition to the written document
- **Pandoc with a branded DOCX or LaTeX template** – ensures the executive report visually matches the client's expected corporate branding and formatting standards, which matters more for this audience than for the technical appendix
- **Chart-generation libraries or spreadsheet tools** (LibreOffice Calc, Python's matplotlib) – used to produce the severity distribution chart and any risk trend visuals, especially on retesting engagements where showing improvement over time is valuable

#### 6 Common Mistakes to Avoid

- Writing the executive summary as an afterthought, copy-pasted and trimmed from the technical findings, rather than deliberately rewritten for the audience
- Burying the overall risk rating or key findings below several paragraphs of engagement logistics
- Including so many "key findings" that the summary loses focus — three to five is usually the right range, not fifteen
- Failing to connect findings to *this specific organization's* business context (e.g., a healthcare client cares about HIPAA and patient safety; a fintech client cares about PCI-DSS and financial fraud)

#### 7 Summary

The Executive Report is a translation exercise, not a summarization exercise. Its job is to take everything discovered technically during the engagement and re-express it in terms of business risk, financial exposure, and strategic priority, using language and visuals that a non-technical decision-maker can absorb in a few minutes and act on with confidence.

### 25.5 Technical Reports

If the Executive Report answers "what is our risk," the Technical Report answers "exactly what is broken, and exactly how do we fix it." This section is written for system administrators, developers, network engineers, and blue team analysts — the people who will actually reproduce, validate, and remediate each finding. Unlike the Executive Report, this section must be exhaustive: nothing should be left out, simplified, or assumed, because the reader's job depends on having complete and accurate technical detail.

#### 1 Purpose and Audience

The Technical Report exists to eliminate any ambiguity between "a vulnerability was found" and "the vulnerability is fixed." A well-written technical finding should allow an engineer with no prior involvement in the engagement to:

- Understand precisely what the vulnerability is and why it exists
- Reproduce the exploitation exactly as the tester performed it
- Understand the real-world impact if the vulnerability is exploited
- Apply a specific, actionable fix
- Later verify, independently, that the fix actually worked

#### 2 Core Components of a Technical Finding

Every finding in the technical report should follow the same repeatable structure, regardless of which tester wrote it, so that the report reads consistently from finding one to finding fifty:

1. **Finding Title** – short, specific, and descriptive (e.g., "Unauthenticated SQL Injection in Customer Login Form," not simply "SQL Injection")
2. **Severity** – both a CVSS score/vector string and a plain qualitative rating (Critical/High/Medium/Low/Informational), so the finding can be sorted and prioritized consistently
3. **Affected Assets** – the specific hosts, IP addresses, URLs, or application components impacted, listed explicitly rather than referenced vaguely ("the web application")
4. **Description** – a precise technical explanation of the vulnerability, including its root cause (e.g., unsanitized user input concatenated directly into a SQL query string)
5. **Proof of Concept** – exact, reproducible steps: the specific commands run, payloads used, and parameters targeted, cross-referenced with the screenshots and evidence gathered in sections 25.2 and 25.3
6. **Impact** – the technical consequence of successful exploitation (e.g., full read/write access to the customer database, remote code execution as SYSTEM, lateral movement to the domain controller)
7. **Remediation** – specific, actionable guidance: exact configuration changes, patch versions, or code-level fixes, not generic advice like "improve input validation"
8. **References** – links to relevant CVEs, vendor security advisories, or OWASP guidance that support the finding's severity rating and remediation approach

#### 3 CVSS Scoring in Detail

Standardized scoring using the Common Vulnerability Scoring System (CVSS), typically version 3.1 or the newer 4.0, allows findings to be compared consistently both within a single report and against industry benchmarks and other engagements. Each metric in the vector string reflects a specific dimension of the vulnerability:

```
CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H  →  Base Score: 9.8 (Critical)
```

- **AV (Attack Vector)** – Network, Adjacent, Local, or Physical — how remotely the flaw can be exploited
- **AC (Attack Complexity)** – Low or High — how much specialized preparation or conditions the attacker requires
- **PR (Privileges Required)** – None, Low, or High — what access level the attacker must already have
- **UI (User Interaction)** – None or Required — whether a victim must take some action (e.g., clicking a link)
- **C / I / A (Confidentiality, Integrity, Availability)** – each rated None, Low, or High, reflecting the impact on data secrecy, data trustworthiness, and system uptime respectively

Testers should always include both the full vector string and the resulting numeric score in each finding, since the vector string alone lets a technical reader independently verify that the score was calculated correctly, rather than simply trusting an unexplained number.

#### 4 Reproducibility as the Core Design Principle

A technical report is only as valuable as its reproducibility. If an engineer cannot follow the proof-of-concept steps and get the same result the tester got, the finding cannot be confidently validated as fixed later. Every proof-of-concept section should therefore include:

- The exact tool and version used (tool behavior can change meaningfully between versions)
- The exact command or request, including all relevant parameters and headers
- The exact target (specific host, port, endpoint, or parameter)
- The expected output if the vulnerability is present, and — ideally — a note on what the output should look like once the fix has been correctly applied

```bash
# Exact reproducible command referenced in a technical finding
sqlmap -u "https://app.client-domain.com/login.php?user=1" \
  --batch --level=3 --risk=2 --dump -T users
```

#### 5 Writing Effective Remediation Guidance

Remediation is frequently the weakest part of technical reports written by less experienced testers, who often default to generic, unhelpful advice. Strong remediation guidance is specific to the actual root cause identified:

- **Weak:** "Improve input validation on the login form."
- **Strong:** "Replace the current string-concatenated SQL query in `login.php` (line 47) with a parameterized query using PDO prepared statements. Example: `$stmt = $pdo->prepare('SELECT * FROM users WHERE username = ?'); $stmt->execute([$username]);`"

Where possible, remediation guidance should include a short, correct code sample or exact configuration directive, not just a description of the desired end state, since this dramatically reduces the back-and-forth needed between the client's engineers and the testing team during the fix-and-retest cycle.

#### 6 Appendices in the Technical Report

The technical report's appendices carry the full weight of raw supporting material that would clutter the main findings section if included inline:

- **Full raw tool output** – complete Nmap scans, Nessus/OpenVAS vulnerability scanner exports, and full Burp Suite request/response logs
- **Complete host inventory** – every IP address, hostname, and operating system version that was in scope and tested, regardless of whether a vulnerability was found on that host
- **Custom tooling** – any word lists, scripts, or payloads developed specifically for the engagement, so the client can understand exactly what was used against their environment
- **Remediation tracking table** – a structured table listing every finding alongside a severity, an assigned internal owner (filled in by the client), and a target fix date, used to drive the post-engagement remediation process and any agreed retest

```markdown
| Finding ID | Title                          | Severity | Owner        | Target Fix Date |
|------------|---------------------------------|----------|--------------|------------------|
| F001       | SQL Injection in Login Form     | Critical | Web Team     | 2026-07-28       |
| F002       | Exposed RDP on Perimeter Firewall | High   | Network Team | 2026-07-25       |
```

#### 7 Tools for Building Technical Reports on Kali

- **Sysreptor / PwnDoc** – store structured per-finding data (CVSS vectors, descriptions, remediation text, evidence links) in a database and auto-generate a consistently formatted technical section, enforcing the same template across every finding and every tester
- **Faraday** – aggregates output directly from scanning and exploitation tools (Nmap, Nessus, Burp Suite, Metasploit) into structured findings, reducing the amount of manual transcription required to get raw tool output into report-ready form
- **Dradis Framework** – similarly ingests tool output automatically via plugins and organizes it into a shared technical findings workspace for multi-tester engagements
- **Pandoc combined with LaTeX templates** – used when a highly formatted, print-ready PDF technical report is required, which is common for compliance-driven engagements such as PCI-DSS attestation support or SOC 2 audit evidence

#### 8 Quality Control Before Delivery

Before a technical report is sent to a client, it should pass a structured internal review, since technical findings are what the client's engineers will scrutinize most closely:

- Every finding follows the same template and heading structure
- Every CVSS vector string, when recalculated independently, produces the stated score
- Every proof-of-concept command has been re-tested against the evidence to confirm it matches
- Every screenshot referenced actually exists at the linked path and shows what the text claims it shows
- Remediation guidance is specific enough that an engineer unfamiliar with the engagement could act on it without further clarification

#### 9 Summary

The Technical Report is where credibility is either earned or lost. It must be exhaustive, consistent, and precisely reproducible, giving engineering teams everything they need — root cause, proof, impact, and specific remediation — to close every finding without ever needing to circle back to the testing team for clarification.
