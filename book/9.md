## Chapter 9 – CMS Security

Content Management Systems (CMS) such as WordPress, Joomla, Drupal, and dozens of others power a huge percentage of websites on the internet today. Because these platforms are so widely deployed, they are also one of the most frequently targeted categories of web applications by attackers. A single outdated plugin, a misconfigured theme, or a weak administrator password on a CMS-driven site can be enough to give an attacker full control of a server.

This chapter focuses on **CMS enumeration and vulnerability scanning**, a critical phase of the web application penetration testing methodology. Before any exploitation can take place, a penetration tester must first identify:

- Which CMS platform is running on the target (WordPress, Joomla, Drupal, Magento, etc.)
- The exact version of the CMS core
- Installed plugins, themes, modules, or extensions and their versions
- Known CVEs (Common Vulnerabilities and Exposures) associated with the detected versions
- Exposed configuration files, backup files, and directory listings
- Usernames and potential login points that could be targeted for brute-force or credential stuffing attacks

Kali Linux ships with a dedicated set of tools built specifically for this purpose. Each tool in this chapter targets either a specific CMS platform or is capable of fingerprinting multiple CMS platforms automatically. Used together, they form a complete CMS reconnaissance toolkit.

The tools covered in this chapter are:

- **WPScan** – a dedicated, purpose-built vulnerability scanner for WordPress, the most widely used CMS in the world. It enumerates plugins, themes, users, and cross-references findings against the WPScan Vulnerability Database.
- **JoomScan** – an OWASP project designed exclusively for scanning and enumerating Joomla installations, detecting core version, components, and known vulnerabilities.
- **Droopescan** – a plugin-based scanner originally built for Drupal but extended to support other CMS platforms such as WordPress, Joomla, SilverStripe, and Moodle.
- **CMSeeK** – a modern, all-in-one CMS detection and exploitation framework that supports fingerprinting for over 180 different CMS platforms, with deep-scan modules for the most common ones.

Each of these tools is explored in full depth in its own dedicated markdown file, including installation, complete command-line reference, syntax, practical hands-on examples with real terminal output, automation scripts, troubleshooting, and best practices for use in authorized penetration testing engagements.

> **Legal Notice:** All tools and techniques described in this chapter must only be used against systems you own or have explicit written authorization to test. Unauthorized scanning or exploitation of CMS platforms is illegal in most jurisdictions.

### 9.1 WPScan

#### 1. Tool Introduction

WPScan is a free (for non-commercial use), black-box WordPress security scanner written in Ruby. It is developed and maintained by the WPScan team (now part of Automattic) and comes pre-installed on Kali Linux. WPScan is purpose-built to enumerate and audit WordPress installations, and it is considered the de-facto standard tool for WordPress penetration testing.

WPScan works by sending a series of HTTP requests to a target WordPress site and analyzing the responses (HTML source, HTTP headers, readme files, changelogs, JSON REST API endpoints, and more) to fingerprint:

- The WordPress core version
- Installed themes and their versions
- Installed plugins and their versions
- Registered usernames
- Configuration weaknesses (e.g., exposed `wp-config.php` backups, directory listing, debug logs)
- Weak/default credentials via password brute-forcing

Every detected component (core, theme, plugin) is automatically checked against the **WPScan Vulnerability Database**, a continuously updated repository of thousands of WordPress-related CVEs, giving testers immediate insight into exploitable weaknesses.

##### 4.1.1 Key Features
- Non-intrusive and aggressive enumeration modes
- Vulnerability data pulled live from the WPScan API (requires a free or paid API token)
- User enumeration via multiple techniques (author sitemap, REST API, login error messages)
- Password and username brute-forcing modules
- Plugin/theme vulnerability detection, including nulled/backdoored plugin detection
- Output in CLI-readable text, JSON, or CLI-No-Color formats for automation

```bash
git clone https://github.com/wpscanteam/wpscan.git
cd wpscan
sudo gem install bundler
bundle install --without test
```

##### Post-Installation: Updating the Vulnerability Database
WPScan requires a free API token from **https://wpscan.com/api** to pull live vulnerability data (up to 25 requests/day on the free tier). Once you have a token:

```bash
wpscan --api-token YOUR_API_TOKEN_HERE --update
```

You can also save the token permanently so you don't need to pass it every time:

```bash
mkdir -p ~/.wpscan
echo "YOUR_API_TOKEN_HERE" > ~/.wpscan/scan.log
```
(Alternatively, export it as an environment variable: `export WPSCAN_API_TOKEN=YOUR_API_TOKEN_HERE`)

To manually refresh the local vulnerability database cache:
```bash
wpscan --update
```

#### 7. Common Use Cases

- **Pre-engagement recon:** Quickly identify WordPress version, theme, and plugin footprint before deeper testing.
- **Vulnerability validation:** Cross-check plugin/theme versions against known CVEs during a formal penetration test.
- **User enumeration for password attacks:** Build a target username list to feed into credential-stuffing or brute-force campaigns.
- **Configuration audit:** Detect exposed backup files, debug logs, or directory listing misconfigurations.
- **Bug bounty reconnaissance:** Fast triage of WordPress targets in scope to identify low-hanging fruit (outdated plugins).
- **Continuous security monitoring:** Scheduled scans (via cron) of an organization's own WordPress sites to catch newly disclosed vulnerabilities in installed plugins.

#### 8. Automation with Bash

##### 8.1 Automated Multi-Target Scan with Report Aggregation
```bash
#!/bin/bash
# wpscan_batch.sh - scan a list of WordPress sites and aggregate vulnerable findings

TARGETS="targets.txt"
API_TOKEN="YOUR_API_TOKEN_HERE"
OUTDIR="wpscan_reports_$(date +%F)"
mkdir -p "$OUTDIR"

while IFS= read -r url; do
    name=$(echo "$url" | sed -E 's#https?://##; s#[/:]#_#g')
    echo "[*] Scanning $url ..."
    wpscan --url "$url" \
        --api-token "$API_TOKEN" \
        -e vp,vt,u \
        --format json \
        --output "$OUTDIR/${name}.json" \
        --random-user-agent \
        --request-timeout 30
    echo "[+] Done: $OUTDIR/${name}.json"
done < "$TARGETS"

echo "[*] Extracting vulnerable plugins across all targets..."
for f in "$OUTDIR"/*.json; do
    echo "== $f =="
    jq -r '.plugins | to_entries[] | select(.value.vulnerabilities | length > 0) | .key' "$f"
done
```

##### 8.2 Cron Job for Weekly Compliance Scanning
```bash
# Add to crontab -e
0 3 * * 1 /usr/local/bin/wpscan_batch.sh >> /var/log/wpscan_weekly.log 2>&1
```

##### 8.3 Slack/Webhook Alert on New Vulnerability Found
```bash
#!/bin/bash
WEBHOOK="https://hooks.slack.com/services/XXX/YYY/ZZZ"
RESULT=$(wpscan --url http://target-site.com --api-token "$API_TOKEN" -e vp,vt --format json)
COUNT=$(echo "$RESULT" | jq '[.plugins[].vulnerabilities[]?] | length')

if [ "$COUNT" -gt 0 ]; then
    curl -X POST -H 'Content-type: application/json' \
    --data "{\"text\":\"⚠️ WPScan found $COUNT vulnerabilities on target-site.com\"}" \
    "$WEBHOOK"
fi
```

#### 9. Tips and Best Practices

- Always obtain and use a free WPScan API token — passive scanning without it will still fingerprint components but will not return live CVE matches.
- Start with `--stealthy` or `--detection-mode passive` when testing production systems to avoid unnecessary load or WAF triggers.
- Use `-e vp,vt` (vulnerable plugins/themes only) instead of `ap,at` (all plugins/themes) for faster scans when you only care about known vulnerabilities.
- Combine WPScan results with manual verification — automated tools can produce false positives, especially around version detection via passive methods.
- Throttle requests (`--throttle`) on shared hosting targets to avoid rate-limiting or accidental denial-of-service.
- Rotate or randomize the User-Agent (`--random-user-agent`) when the target has bot-detection systems in place.
- Never brute-force login credentials without explicit written authorization — this is often outside the scope of many penetration test agreements.
- Keep the local vulnerability database updated (`wpscan --update`) before every engagement.
- Save all scans in JSON format for later parsing, diffing between engagements, and integration into reporting pipelines.

#### 10. Troubleshooting

| Issue | Cause | Solution |
|-------|-------|----------|
| `Error: 403 Forbidden` on all requests | WAF/CDN (e.g., Cloudflare) blocking scanner | Use `--random-user-agent`, add delays with `--throttle`, or route through a residential proxy (only if authorized) |
| `The remote website is up, but does not seem to be running WordPress` | Detection heuristics failed (custom setup) | Force the scan with `--force` |
| `No WPScan API Token given` | Missing/expired API token | Register at wpscan.com/api and pass `--api-token` |
| SSL certificate errors | Self-signed cert on staging/test environments | Add `--disable-tls-checks` |
| Extremely slow scans | Too many threads on a slow target or rate-limited connection | Lower `--max-threads`, increase `--request-timeout` |
| `Connection refused` | Target down, firewall blocking, or wrong port/protocol | Verify with `curl -I <url>` first |
| Ruby gem dependency errors during manual install | Missing system libraries | Install `libcurl4-openssl-dev`, `libxml2-dev`, `libxslt1-dev`, `zlib1g-dev` before `gem install wpscan` |
| API requests exhausted (`Requests Remaining: 0`) | Free tier daily limit (25/day) reached | Wait for daily reset or upgrade to a paid WPScan.com plan |

### 9.2 JoomScan

#### 1. Tool Introduction

JoomScan (Joomla Vulnerability Scanner) is an open-source OWASP project written in Perl, purpose-built to detect, fingerprint, and audit Joomla-based websites. It was one of the earliest dedicated CMS scanners and remains one of the most reliable tools for Joomla-specific reconnaissance on Kali Linux.

JoomScan works by crawling a target site, parsing manifest/version files (such as `language/en-GB/en-GB.xml` or `includes/framework.php`), inspecting HTTP response headers, and comparing discovered components against a vulnerability signature database maintained by the OWASP JoomScan project. It can identify:

- The Joomla core version (via multiple detection techniques for reliability)
- Firewall / Web Application Firewall (WAF) presence in front of the target
- Installed components, modules, and templates
- Publicly known vulnerabilities tied to the detected Joomla version
- Common sensitive files and directory listing issues
- Robots.txt disclosure of admin/backup paths

##### 4.1.1 Key Features
- Version detection using multiple fallback methods for higher accuracy
- Firewall/WAF detection before scanning begins
- Vulnerability reporting tied to specific Joomla core versions
- Directory listing checks
- Session-based scanning with random User-Agent support
- Report generation in HTML format for client-ready deliverables
- Update mechanism to keep the vulnerability database current

#### 9. Tips and Best Practices

- Always run `joomscan --update` before a new engagement to ensure the vulnerability signature database is current.
- Use `--random-agent` on targets with basic bot-blocking to reduce the chance of being fingerprinted as an automated scanner.
- Pair JoomScan's core-version detection with manual verification of third-party components, since JoomScan's vulnerability database is strongest for Joomla core, not every extension.
- Use the `-ir` report flag for every engagement — having a timestamped HTML report is invaluable for client deliverables and later diffing.
- Run component enumeration (`-ec`) even when the core Joomla version appears secure, since many real-world Joomla compromises originate from vulnerable third-party components, not the core.
- Combine JoomScan output with CMSeeK or manual OSINT for a second, independent confirmation of the Joomla version, since HTTP header spoofing or version hiding plugins can occasionally mislead single-tool scans.

#### 10. Troubleshooting

| Issue | Cause | Solution |
|-------|-------|----------|
| `Target is not running Joomla` | Detection heuristics failed on a heavily customized install | Manually verify via `curl -s http://target/administrator/manifests/files/joomla.xml` |
| Scan hangs indefinitely | Target is slow or rate-limiting the scanner | Increase `--timeout`, or scan during off-peak hours |
| `Firewall detected` blocking accurate scans | WAF is intercepting/altering scanner requests | Note the WAF in your report; consider `--random-agent` and slower request pacing (manual delay via wrapper script) |
| Report file not generating | Write-permission issue in the current directory | Run from a writable directory or specify a full path with `-ir /home/user/reports/name` |
| `--update` fails with connection error | No internet access or blocked GitHub access | Verify connectivity: `curl -I https://github.com`; check proxy settings |
| Outdated vulnerability results | Local database not updated recently | Run `joomscan --update` |
| Perl module errors when running from source | Missing dependencies | `sudo apt install liblwp-useragent-determined-perl libwww-perl` |

### 9.3 Droopescan

#### 1. Tool Introduction

Droopescan is a Python-based, plugin-oriented CMS security scanner. It was originally created to scan Drupal websites (hence the name), but its architecture is built around independent "scanner plugins," which means it can also scan other CMS platforms including WordPress, Joomla, SilverStripe, and Moodle using the same core engine.

Droopescan's main value in a CMS Security workflow is as a **cross-verification tool**: because it uses an entirely independent detection engine and signature set compared to WPScan or JoomScan, running it alongside the CMS-specific tools helps confirm findings and can occasionally catch things the primary scanner misses, particularly on Drupal, which lacks a dedicated tool as mature as WPScan.

It identifies:

- CMS platform and version (Drupal, WordPress, Joomla, SilverStripe, Moodle)
- Installed themes and modules/plugins
- Interesting URLs (admin panels, changelog files, install files)
- Version fingerprinting via static file hashes (MD5-based signature matching)

##### 4.1.1 Key Features
- Plugin-based architecture — new CMS scanners can be added without modifying the core
- Multi-threaded scanning for speed
- Hash-based version fingerprinting (compares static file hashes against a known signature database)
- Simple, scriptable output suitable for automation
- Host list scanning for bulk assessments

#### 9. Tips and Best Practices

#### 10. Troubleshooting

| Issue | Cause | Solution |
|-------|-------|----------|
| `No plugins/themes found` on a site clearly using known plugins | Signature database out of date for newer plugin releases | Update droopescan via `pip3 install --upgrade droopescan --break-system-packages` or pull latest from GitHub |

| `pip3 install` fails with "externally-managed-environment" | Newer Debian/Kali Python policy | Add `--break-system-packages` flag or use a virtual environment |
| Bulk scan (`-U`) skips some hosts | Malformed URLs in the input file (missing scheme) | Ensure every line starts with `http://` or `https://` |

### 9.4 CMSeeK

#### 1. Tool Introduction

CMSeeK is a modern, Python 3-based CMS detection and exploitation framework. Unlike WPScan or JoomScan, which are dedicated to a single platform, CMSeeK is designed as a **universal CMS fingerprinting tool**, capable of identifying more than 180 different Content Management Systems using a signature-detection engine (built on top of the CMSeeK detection database, which is compatible with the widely-used `whatcms`/`cms-detector` style signature format).

For the most common and high-value CMS platforms — WordPress, Joomla, Drupal, Magento, PrestaShop, OpenCart, and others — CMSeeK goes a step further and triggers dedicated **deep-scan modules** that extract:

- CMS name, version, and detection confidence
- Users/authors (where applicable)
- Installed plugins/themes/extensions and their versions
- Known vulnerabilities associated with the detected CMS/version
- Server technology stack (PHP version, web server, etc.)
- robots.txt and admin panel discovery
- Whether the target is running behind a CDN/WAF

CMSeeK is typically the **first tool run** in a CMS security workflow because of its speed and broad platform coverage — its output determines which CMS-specific deep-scan tool (WPScan, JoomScan, Droopescan) should be used next.

##### 4.1.1 Key Features
- Detects 180+ CMS platforms via signature matching
- Deep-scan modules for major platforms (WordPress, Joomla, Drupal, Magento, etc.)
- Result caching in structured JSON per target for easy automation/integration
- Bulk scanning support via list input
- Update mechanism to refresh CMS detection signatures
- Lightweight and fast compared to full CMS-specific scanners — ideal for the initial triage phase

#### 7. Common Use Cases

- **First-pass CMS triage:** Quickly determine which CMS platform (out of 180+ possibilities) a target is running before deciding which specialized tool to run next.
- **Unknown-CMS environments:** In large external attack-surface assessments where the CMS platform per host is unknown, CMSeeK's bulk-scan mode rapidly categorizes hundreds of hosts.
- **Lightweight WordPress/Joomla/Drupal auditing:** When a full WPScan/JoomScan run isn't necessary, CMSeeK's built-in deep-scan modules provide a fast summary of version, users, and plugins.
- **Asset inventory building:** Security teams use CMSeeK output (cached as structured JSON) to build and maintain an inventory of CMS platforms and versions across an organization's web properties.
- **Shadow IT discovery:** Identifying forgotten or unmanaged CMS installations (old Joomla/Drupal sites) during external reconnaissance.

#### 8. Automation with Bash

##### 8.1 Bulk Triage and Auto-Routing to Specialized Scanners
```bash
#!/bin/bash
# cmseek_triage.sh - detect CMS for each target, then route to the right specialized tool

TARGETS="mixed_targets.txt"
mkdir -p triage_results

while IFS= read -r url; do
    echo "[*] Detecting CMS for $url"
    cmseek -u "$url" -b > /dev/null 2>&1

    name=$(echo "$url" | sed -E 's#https?://##; s#[/:]#_#g')
    result="$HOME/CMSeeK/Result/${name}/cms.json"

    if [ -f "$result" ]; then
        cms=$(jq -r '.cms_id' "$result")
        echo "[+] $url -> $cms"

        case "$cms" in
            wordpress)
                wpscan --url "$url" -e vp,vt --format json --output "triage_results/${name}_wpscan.json" ;;
            joomla)
                joomscan -u "$url" -ec -ir "triage_results/${name}_joomscan" ;;
            drupal)
                droopescan scan drupal -u "$url" -e a -o json > "triage_results/${name}_droopescan.json" ;;
            *)
                echo "[i] No specialized scanner routed for CMS: $cms" ;;
        esac
    fi
done < "$TARGETS"
```

##### 8.2 Nightly CMS Inventory Snapshot
```bash
#!/bin/bash
# nightly_cms_inventory.sh
cmseek --update
cmseek -l company_hosts.txt -b

echo "== CMS Inventory Report $(date) ==" > inventory_report.txt
```

#### 9. Tips and Best Practices

- Always run CMSeeK first in a CMS security assessment — its speed and broad coverage make it ideal for triage before committing time to a heavier, CMS-specific scan.
- Use `-b` (batch mode) for any scripted or automated workflow; without it, CMSeeK may prompt interactively and break automation pipelines.
- Periodically run `cmseek --update` to keep the 180+ platform signature database current, since new CMS platforms and detection signatures are added regularly.
- Use `--clear-cache` before re-scanning a target that you know has changed (e.g., after a patch or migration), since CMSeeK caches results per-target by default.
- Combine CMSeeK's JSON output with `jq` for easy integration into asset inventory databases or SIEM pipelines.
- When CMSeeK's auto-detection is inconclusive or ambiguous (common on heavily customized sites), use the `--is-wp`, `--is-joomla`, or `--is-drupal` flags to force the relevant deep-scan module and get a second opinion.
- For any positively identified WordPress, Joomla, or Drupal target, follow up CMSeeK's initial pass with the dedicated deep-scan tool (WPScan, JoomScan, or Droopescan respectively) for the most authoritative, up-to-date vulnerability data.

#### 10. Troubleshooting

| Issue | Cause | Solution |
|-------|-------|----------|
| `CMS Not Detected` on a known CMS site | Custom install hides generator tags/signatures | Try `-v` verbose mode to see which signatures were tested; force the module with `--is-wp` / `--is-joomla` / `--is-drupal` |
| Script hangs waiting for input | Ran without `-b` (batch mode) in an automated pipeline | Always include `-b` for non-interactive scripted scans |
| `ModuleNotFoundError` on startup | Missing Python dependencies | Re-run `pip3 install -r requirements.txt --break-system-packages` inside the CMSeeK directory |
| Cached results are stale after target was patched | CMSeeK reuses cached JSON by target name | Run `cmseek --clear-cache` before re-scanning |
| SSL handshake failures | Self-signed certificate on staging environment | Add `--no-check-certificate` |
| `--update` fails | No internet access or GitHub connectivity blocked | Verify with `curl -I https://github.com`; check firewall/proxy rules |
| Deep-scan module doesn't trigger even though CMS is correctly detected | Confidence threshold not met, or module intentionally skipped for low-confidence matches | Force the module manually with the relevant `--is-*` flag |
| Global `cmseek` command not found | Tool installed via git clone without symlink setup | Either run `python3 cmseek.py` from within the cloned directory, or create the symlink as shown in the Installation section |
