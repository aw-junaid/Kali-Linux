## Chapter 22 – Malware Analysis

Malware analysis is the process of studying a malicious sample — a piece of software, script, or document designed to harm, spy on, or exploit a system — in order to understand what it does, how it works, how it got in, and how to detect and stop it in the future. This chapter focuses on **static and semi-static analysis tooling** available on Kali Linux: tools that let an analyst inspect a suspicious binary's structure, strings, embedded indicators, and behavioral capabilities *without* necessarily executing it in a sandbox.

The tools covered in this chapter form a natural analysis pipeline that a malware analyst, SOC analyst, or threat hunter would typically follow when a new sample lands on their desk:

1. **Detect It Easy (DIE)** and **PEStudio** — first-pass triage. Identify the file type, packer/compiler, architecture, and suspicious PE header artifacts before doing anything else.
2. **FLOSS** — extract readable strings (including ones hidden by obfuscation, stack strings, or simple encoding) to get quick clues about functionality, C2 domains, file paths, and registry keys.
3. **CAPA** — automatically identify the *capabilities* of the binary (e.g., "can create a service," "may inject into a remote process," "can establish a TCP connection") by mapping code patterns to the MITRE ATT&CK framework.
4. **YARA** — write and run pattern-matching rules to classify, hunt for, and detect the sample (and its variants) across a fleet of machines or a large file corpus.


> **Legal and ethical note:** All tools in this chapter are legitimate, widely used defensive security tools. They are intended to be used on samples you are authorized to analyze — in an isolated lab, a malware analysis VM, or a sandboxed environment. Never execute unknown malware on a production or personally-identifiable system, and always follow your organization's handling policy for malicious samples.

### 22.1 YARA

#### 1. Tool Introduction

YARA ("Yet Another Recursive Acronym," or informally "the pattern matching swiss knife for malware researchers") is an open-source tool created by Victor Manuel Álvarez (originally at VirusTotal) used to identify and classify malware samples by creating descriptions — called **rules** — based on textual or binary patterns. Each rule is a set of strings (plain text, hex bytes, or regular expressions) plus a boolean condition that decides whether a file "matches."

YARA is used for:
- Classifying and naming malware families
- Hunting for malicious files across a filesystem, a memory image, or a network share
- Building detection content for EDR/AV engines
- Triggering automated response (e.g., via ClamAV, THOR, Loki, or custom SOC scripts)
- Scanning running processes and memory dumps for injected code or in-memory-only malware

YARA ships as both a standalone command-line binary (`yara`) and a Python library (`yara-python`) for scripting.

###### Key Concepts
- **Rule** — a named block containing optional `meta`, `strings`, and a mandatory `condition`.
- **Strings** — text strings (`$a = "cmd.exe"`), hex byte patterns (`$b = { 6A 40 68 00 30 00 00 }`), or regular expressions (`$c = /https?:\/\/[a-z]+\.com/`).
- **Condition** — a boolean expression (e.g., `$a and $b`, `all of them`, `#a > 3`) that determines a match.
- **Modules** — built-in extensions such as `pe`, `elf`, `math`, `hash`, `magic`, and `cuckoo` that expose structured file metadata to conditions.

#### 3. Syntax

```
yara [OPTIONS] RULES_FILE TARGET
yara [OPTIONS] RULES_FILE PID
```
Where:
- `RULES_FILE` — path to a `.yar`/`.yara` file, a compiled rules file (`-C`), or `-` to read a rule from stdin.
- `TARGET` — a file, a directory (with `-r` for recursive), or a running process ID.

Basic rule syntax:
```yara
rule RuleName : tag1 tag2
{
    meta:
        author = "analyst"
        description = "detects sample X"
        date = "2026-07-19"

    strings:
        $s1 = "malicious_string" nocase
        $s2 = { E8 ?? ?? ?? ?? 5D C3 }
        $s3 = /regex_pattern/

    condition:
        uint16(0) == 0x5A4D and ($s1 or $s2) and $s3
}
```

#### 4. All Command-Line Options (Kali Linux `yara` binary)

| Option | Long form | Description |
|---|---|---|
| `-t <tag>` | `--tag=<tag>` | Only print matches for rules tagged with `<tag>` |
| `-i <identifier>` | `--identifier=<identifier>` | Only print matches for rules named `<identifier>` |
| `-n` | `--negate` | Print only rules that did **not** match |
| `-g` | `--print-tags` | Print rule tags |
| `-m` | `--print-meta` | Print rule metadata |
| `-D` | `--print-module-data` | Print module data (e.g., PE header fields used) |
| `-e` | `--print-namespace` | Print rule namespace |
| `-S` | `--print-stats` | Print rule-matching performance statistics |
| `-s` | `--print-strings` | Print matching strings and their offsets |
| `-L` | `--print-string-length` | Print length of matched strings |
| `-X` | `--print-xor-key` | Print XOR key used, if `xor` modifier matched |
| `-r` | `--recursive` | Recursively scan directories |
| `-f` | `--fast-scan` | Fast matching mode (stop after first match per string) |
| `-w` | `--no-warnings` | Disable warning messages |
| `-d <id>=<value>` | `--define=<id>=<value>` | Define an external variable for use in conditions |
| `-x <mod>=<file>` | `--module-data=<mod>=<file>` | Pass extra data to a module |
| `-a <seconds>` | `--timeout=<seconds>` | Abort scanning after N seconds |

| `-p <n>` | `--threads=<n>` | Use N threads for scanning |

| `-h` | `--help` | Show help |
| `-v` | `--version` | Show version |

#### 7. Common Use Cases

- **Malware family classification** — tag and name samples based on known indicators.
- **Threat hunting** — sweep an entire disk image, mounted evidence, or file share for IOCs.
- **YARA-on-memory** — scan live processes or memory dumps (via Volatility's `yarascan`) for injected shellcode or fileless malware.

- **VirusTotal retro-hunting** — the same rule syntax is used to hunt VirusTotal's live and historical corpus.
- **Incident response** — quickly sweep an entire fleet for a newly discovered IOC before full remediation.

#### 8. Automation with Bash

**Batch scan and log results with timestamps:**

```bash

    echo "$MATCHES" | mail -s "YARA ALERT: Malware detected" soc@example.com

```

**Recompile all rule files nightly for performance:**
```bash
#!/bin/bash
for rule in /opt/yara-rules/*.yar; do
    yarac "$rule" "${rule%.yar}.yrc"
done
```

#### 9. Tips and Best Practices

- Keep string identifiers descriptive (`$cmd_exec` not `$s1`) — future-you will thank you.
- Use `nocase`, `wide`, and `ascii` modifiers to catch case variants and UTF-16 strings common in Windows malware.
- Prefer specific hex/opcode patterns over long plaintext strings — plaintext strings are trivial for malware authors to obfuscate.
- Use the `pe`, `elf`, and `hash` modules for structural conditions instead of relying purely on strings — this reduces false positives.
- Compile rules (`yarac`) in production/high-volume pipelines — compiled rules load significantly faster.
- Version-control your rule repository and tag rules with `author`, `date`, `reference`, and `hash` metadata for provenance.
- Test new rules against a large "known-good" corpus before deploying to avoid false-positive floods.
- Use `-f` (fast-scan) for high-volume scanning where only "does it match" matters, not every occurrence.

#### 10. Troubleshooting

| Problem | Cause / Fix |
|---|---|
| `error: syntax error, unexpected ...` | Malformed rule syntax — check for missing colons, braces, or semicolons |
| `warning: rule "X" is slow` | Overly broad regex or wildcard hex string — narrow the pattern |
| No matches on a known-bad file | Check string modifiers (`nocase`/`wide`), confirm condition logic, test each string individually with `-s` |
| `yara: error scanning ...: Permission denied` | Run with elevated privileges when scanning process memory (`sudo`) |
| Compiled rules (`-C`) fail to load | Compiled rules are version-specific — recompile with the same `yara`/`yarac` version installed |
| Extremely slow scans on large directories | Use `-p` for multithreading, `-f` for fast-scan mode, and compiled rulesets |
| Module (`pe`, `elf`) fields return undefined | The module wasn't built into your YARA binary — rebuild from source with `--enable-<module>`, or install `yara` with full module support |

### 22.2 CAPA

#### 1. Tool Introduction

CAPA (short for "capability") is an open-source tool developed by Mandiant (formerly FireEye) that automatically identifies **capabilities** in executable files — what a program can *do* — rather than what family it belongs to. Instead of manually reverse engineering a binary to figure out whether it can persist via a scheduled task, inject into a process, or communicate over HTTP, CAPA statically analyzes the code and reports these capabilities in plain English, each mapped to the relevant **MITRE ATT&CK** tactic and technique, and to the **Malware Behavior Catalog (MBC)**.

CAPA works on PE files, ELF binaries, .NET binaries, shellcode, and (via extensions) sandbox reports. It ships with a large open rule set (like YARA rules but for capabilities, written in CAPA's own rule format) that continues to grow with community contributions.

Typical uses:
- Rapid triage — get an instant capability summary of an unknown binary in seconds
- Prioritization — decide which of hundreds of samples deserves deep manual reverse engineering
- Report writing — auto-generate an ATT&CK-mapped capability list for an incident report
- Detection engineering — use capability output to write better YARA/Sigma rules

#### 2. How to Install

CAPA is distributed as a standalone binary and as a Python package. On Kali Linux, the Python package route is most common.

```bash
chmod +x /usr/local/bin/capa
```

#### 8. Automation with Bash

**Batch scan a directory of samples and save JSON results:**
```bash
#!/bin/bash
SAMPLE_DIR="/mnt/samples"
OUT_DIR="/mnt/capa_results"
mkdir -p "$OUT_DIR"

for f in "$SAMPLE_DIR"/*; do
    name=$(basename "$f")
    echo "[*] Analyzing $name"
    capa -j "$f" > "$OUT_DIR/${name}.json" 2>> "$OUT_DIR/errors.log"
done
```

**Extract only high-confidence ATT&CK techniques across a batch, into a CSV:**
```bash
#!/bin/bash
echo "sample,technique" > capa_summary.csv
for jsonfile in /mnt/capa_results/*.json; do
    sample=$(basename "$jsonfile" .json)
    jq -r '.rules | to_entries[] | .value.meta.attack[]?.technique' "$jsonfile" 2>/dev/null | \
        while read -r tech; do
            echo "$sample,$tech" >> capa_summary.csv
        done
done
```

**Nightly rule-update + scan cron job:**
```bash
#!/bin/bash
cd /opt/capa-rules && git pull
capa -r /opt/capa-rules -j /mnt/incoming/latest_sample.exe > /var/log/capa_latest.json
```

#### 9. Tips and Best Practices

- Always keep the `capa-rules` repository up to date (`git pull`) — new capability rules are added frequently.
- Use `-vv` when you need to understand *why* a rule matched, not just that it did — invaluable for validating findings before they go in a report.
- Combine CAPA with FLOSS and CAPE sandbox output for the fullest picture; CAPA is static, so packed/obfuscated code may hide capabilities until unpacked.
- Use `-j` and `jq`/Python for any workflow processing more than a handful of samples — the table output is for humans, JSON is for automation.
- Run CAPA before manual reverse engineering — it dramatically narrows down which functions deserve a disassembler's attention.
- For shellcode, always specify `-f sc32` or `-f sc64` explicitly; auto-detection can misidentify raw shellcode blobs.

#### 10. Troubleshooting

| Problem | Cause / Fix |
|---|---|
| `ERROR: unsupported file format` | Force the format explicitly with `-f` (e.g., `-f elf`, `-f sc32`) |
| `no rules loaded` | Point `-r` at a valid `capa-rules` directory, or set the `CAPA_RULES_PATH` environment variable |
| Very few/no capabilities found on an obviously malicious sample | Sample may be packed — unpack first (see DIE chapter) or run CAPA against a memory dump / unpacked payload |
| Slow analysis on large binaries | Use `--restrict-to-functions` to narrow scope, or increase available RAM/CPU |
| `vivisect` backend errors on malformed PE | Try `--backend binja` or `--backend ida` if those tools/plugins are installed |
| JSON output missing ATT&CK mappings | Some rules have no `att&ck` metadata (custom/community rules) — check `mbc` (Malware Behavior Catalog) field instead |

### 22.3 PEStudio

#### 1. Tool Introduction

PEStudio (by Marc Ochsenmeier / Winitor) is a static malware triage tool purpose-built to inspect Windows **Portable Executable (PE)** files — `.exe`, `.dll`, `.sys`, `.scr`, `.ocx` — **without executing them**. It surfaces the artifacts a human analyst would otherwise have to dig for manually: imports, exports, sections, strings, embedded resources, digital signatures, version info, and — critically — it automatically cross-references indicators (imphash, file hash, embedded domains/IPs) against VirusTotal and flags anything on its built-in "suspicious indicators" blacklist (dangerous imports like `VirtualAllocEx`+`WriteProcessMemory`+`CreateRemoteThread`, suspicious strings, anti-debug APIs, etc.).

**Important platform note:** PEStudio is a **native Windows application** (no official Linux build). On Kali Linux it is run under **Wine**, or inside a Windows analysis VM that is part of the same lab setup. This chapter documents installing and using it under Wine on Kali, which is the standard approach for analysts who want to keep PE triage inside their Linux toolchain.

PEStudio has a free edition (manual use, GUI only) and a paid Pro edition (command-line/batch automation, XML reporting, custom blacklists).

#### 2. How to Install

**Step 1 — Install Wine on Kali Linux:**
```bash
sudo dpkg --add-architecture i386
sudo apt update
sudo apt install wine wine32 wine64 winetricks -y
```

**Step 2 — Create a clean Wine prefix for analysis (isolated from your main Wine environment):**
```bash
export WINEPREFIX=~/.wine-pestudio
export WINEARCH=win64
winecfg
```

**Step 3 — Download PEStudio:**
Download the ZIP from the official site (https://www.winitor.com) on a trusted host and transfer it into your Kali analysis VM — do **not** fetch it through a browser inside the analysis VM where malware will later be detonated.
```bash
mkdir -p ~/tools/pestudio
unzip pestudio.zip -d ~/tools/pestudio
```

**Step 4 — Run it under Wine:**
```bash
cd ~/tools/pestudio
WINEPREFIX=~/.wine-pestudio wine pestudio.exe
```

**Recommended: isolate networking.** Since PEStudio itself can optionally reach out to VirusTotal, and because the samples you load are live malware, run this inside an isolated VM/snapshot with host-only or no networking unless you specifically want VirusTotal lookups, in which case use a dedicated non-attributable connection.

#### 3. Syntax

**Free/GUI edition:** no meaningful CLI — launched interactively:
```bash
wine pestudio.exe [path\to\sample.exe]
```

**Pro edition (licensed) command-line / batch mode:**
```
pestudio.exe -file <path> -xml <output.xml> [options]
```

#### 8. Automation with Bash

Because the free GUI edition isn't scriptable, automation on Kali typically wraps the Pro edition's CLI, or uses Wine's headless capabilities:

**Batch analyze a folder of samples with the Pro CLI, producing one XML report per sample:**
```bash
#!/bin/bash
export WINEPREFIX=~/.wine-pestudio
SAMPLE_DIR="/mnt/samples"
OUT_DIR="/mnt/pestudio_reports"
mkdir -p "$OUT_DIR"

for f in "$SAMPLE_DIR"/*.exe; do
    name=$(basename "$f" .exe)
    wine pestudio.exe -file "$f" -xml "$OUT_DIR/${name}.xml" -quiet
done
```

**Extract high-severity indicators from all XML reports into a single CSV summary:**
```bash
#!/bin/bash
echo "sample,severity,indicator" > pestudio_summary.csv
for xmlfile in /mnt/pestudio_reports/*.xml; do
    sample=$(basename "$xmlfile" .xml)
    grep -oP '(?<=<indicator severity=")3(?=")[^>]*text="\K[^"]*' "$xmlfile" | \
        while read -r ind; do
            echo "$sample,3,$ind" >> pestudio_summary.csv
        done
done
```

**Launch PEStudio pre-loaded with a sample from a right-click-style shell alias:**
```bash
echo 'alias pestudio="WINEPREFIX=~/.wine-pestudio wine ~/tools/pestudio/pestudio.exe"' >> ~/.bashrc
```

#### 9. Tips and Best Practices

- Always run PEStudio inside an isolated analysis VM/snapshot — it loads and parses the file, but you still don't want the sample anywhere near a production network.
- Disable the automatic VirusTotal lookup (or use a dedicated research API key) if you don't want to alert the malware author that their sample is under analysis (uploading a hash reveals detection to anyone monitoring VT).
- Cross-check high entropy sections (>7.0) with DIE's packer detection — the two tools complement each other.
- Note down the **imphash** value every time — it's one of the fastest ways to pivot to related samples in VT/MISP.
- Treat the `version` panel's claims (company name, product name) with suspicion — these fields are free-text and trivially spoofed by malware authors.
- Combine PEStudio's `strings` panel output with FLOSS for a fuller string picture — PEStudio shows plain static strings; FLOSS additionally decodes obfuscated ones.

#### 10. Troubleshooting

| Problem | Cause / Fix |
|---|---|
| `wine: cannot find L"Z:\\...\\pestudio.exe"` | Wrong path or Wine prefix not initialized — re-run `winecfg` under the same `WINEPREFIX` first |
| GUI renders with broken fonts/icons | Run `winetricks corefonts` inside the same `WINEPREFIX` |
| PEStudio crashes on 32-bit sample under a 64-bit prefix | Create a `WINEARCH=win32` prefix specifically for 32-bit samples |
| VirusTotal panel always empty | No API key configured, or no network access from the isolated VM (expected/by design in many labs) |
| "Unknown file format" error | The file isn't a valid PE (check with `file` or DIE first — could be a script, archive, or corrupted binary) |
| Pro CLI switches don't work in free edition | Batch/headless CLI scanning requires the licensed Pro edition; free edition is GUI-only |

### 22.4 Detect It Easy (DIE)

#### 1. Tool Introduction

Detect It Easy (DIE) is a free, open-source, cross-platform program for determining the **type**, **compiler/linker**, **packer/protector**, and general **file format** of a binary. It is the modern, actively maintained successor to older signature-based identification tools like PEiD. DIE ships with:

- A large, extensible **signature database** (written in a custom scripting language) covering compilers (MSVC, MinGW, Delphi, Go, Rust, .NET), packers (UPX, ASPack, Themida, VMProtect, Enigma), and installers (NSIS, InnoSetup, InstallShield).
- A **GUI application** with a hex viewer, entropy graph, and a built-in scripting/signature editor.
- A **command-line utility** (`diec`) for headless/batch scanning — ideal for pipelines and automation.

DIE's core value in a malware analysis pipeline is answering, in seconds, "what is this file, what compiled it, and is it packed?" — which directs everything downstream (whether to unpack before running FLOSS/CAPA, which YARA rule families are likely relevant, etc.).

#### 8. Automation with Bash

**Batch-scan and sort samples into folders by detected packer:**
```bash
#!/bin/bash
SAMPLE_DIR="/mnt/samples"
SORTED_DIR="/mnt/sorted"
mkdir -p "$SORTED_DIR/upx" "$SORTED_DIR/themida" "$SORTED_DIR/unpacked" "$SORTED_DIR/other"

for f in "$SAMPLE_DIR"/*; do
    result=$(diec -j "$f" 2>/dev/null)
    if echo "$result" | grep -qi "UPX"; then
        cp "$f" "$SORTED_DIR/upx/"
    elif echo "$result" | grep -qi "Themida"; then
        cp "$f" "$SORTED_DIR/themida/"
    elif echo "$result" | grep -qi "Packer"; then
        cp "$f" "$SORTED_DIR/other/"
    else
        cp "$f" "$SORTED_DIR/unpacked/"
    fi
done
echo "[*] Sorting complete."
```

#### 9. Tips and Best Practices

- Always run DIE (or an equivalent) **first**, before FLOSS/CAPA/YARA — knowing whether a sample is packed changes how you interpret (or discount) results from the other tools.
- Use `-d` (deep scan) on anything DIE's normal scan reports as "Unknown" — many modern protectors need heuristic detection.
- Keep the signature database updated regularly (`Options → Update database` in the GUI, or re-download `db.zip`) — new packers/compilers appear constantly.
- Cross-check DIE's entropy readings against PEStudio's per-section entropy for a fuller packing picture.
- Use JSON/CSV output for anything beyond a single manual look — the GUI is great for one-off deep dives, `diec` is for pipelines.
- If DIE reports "Unknown: Heuristic scan" only, don't assume the file is benign — it may simply use a custom or unrecognized packer; escalate to manual entropy/section analysis.

### 22.5 FLOSS

#### 1. Tool Introduction

FLOSS (**FLARE Obfuscated String Solver**) is an open-source tool from Mandiant's FLARE team that automatically extracts printable strings from a malware binary — going far beyond what the standard Unix `strings` utility can find. Malware authors routinely hide meaningful strings (C2 domains, mutex names, file paths, registry keys, ransom notes) by:

- Building them at runtime on the **stack** one character at a time ("stack strings")
- **XOR/ADD/base64/other simple encoding** and decoding them just before use
- Storing them in **tight decoding loops** that a static `strings` scan will never see because the bytes on disk don't look like text

FLOSS statically emulates the binary's code (via the `vivisect` and `smda` analysis frameworks) to recover all four categories of strings:
1. **Static strings** — same as plain `strings` would find
2. **Stack strings** — reconstructed from stack/register operations
3. **Decoded strings** — recovered by identifying and running decoding routines
4. **Tight-loop strings** — recovered from small decode loops that don't fit the "decoding function" pattern

The result is a vastly more complete string listing that dramatically speeds up manual reverse engineering and gives immediate IOC leads.

#### 7. Common Use Cases

- **Rapid IOC extraction** — C2 domains, IPs, URIs, mutex names, and file paths that never appear in plain-`strings` output because they're built/decoded at runtime.
- **Speeding up reverse engineering** — analysts get a head start on "what does this malware talk to / where does it write files" before opening a disassembler.
- **YARA rule authoring** — decoded strings recovered by FLOSS make excellent, high-fidelity YARA string signatures.
- **Shellcode triage** — quickly understand embedded shellcode from exploit documents, macros, or memory dumps.
- **Config extraction** — many malware families store base64/XOR-encoded configuration blocks that FLOSS's decoding-string detection recovers automatically.

#### 8. Automation with Bash

**Batch-run FLOSS across a sample directory, saving JSON per sample:**
```bash
#!/bin/bash
SAMPLE_DIR="/mnt/samples"
OUT_DIR="/mnt/floss_results"
mkdir -p "$OUT_DIR"

for f in "$SAMPLE_DIR"/*; do
    name=$(basename "$f")
    floss -j "$f" > "$OUT_DIR/${name}.json" 2>> "$OUT_DIR/errors.log"
    echo "[*] Processed $name"
done
```

**Aggregate all decoded strings across a batch into a single deduplicated IOC list:**
```bash
#!/bin/bash
> all_decoded_strings.txt
for jsonfile in /mnt/floss_results/*.json; do
    jq -r '.strings.decoded_strings[].string' "$jsonfile" 2>/dev/null >> all_decoded_strings.txt
done
sort -u all_decoded_strings.txt -o all_decoded_strings.txt
echo "[*] $(wc -l < all_decoded_strings.txt) unique decoded strings collected."
```

**Pipeline: DIE unpack check -> FLOSS -> grep for URLs, auto-append to a threat-intel feed file:**
```bash
#!/bin/bash
SAMPLE="$1"
if diec "$SAMPLE" | grep -qi "packer"; then
    echo "[!] $SAMPLE appears packed — unpack before running FLOSS for best results."
fi
floss --only decoded,stack "$SAMPLE" | grep -oE 'https?://[^ ]+' >> threat_intel_urls.txt
```

#### 9. Tips and Best Practices

- Run FLOSS **after** confirming (via DIE) whether the sample is packed — FLOSS analyzes the code as laid out on disk, so a packed sample will only reveal the packer's own strings, not the payload's, until unpacked.
- Use `-n` to bump the minimum string length when static output is dominated by short, meaningless noise strings.
- `--only decoded,stack,tight` is a good default when you only care about hidden/obfuscated strings and want to skip re-reading the (usually large) static string list.
- Pipe output straight into `grep -E` for URLs/IPs/domains to get instant IOC candidates for a report or a blocklist.
- FLOSS's `decoded strings` output is one of the best sources of raw material for new YARA rule strings — high signal, low false-positive risk since they come directly from the sample's own runtime behavior.
- For very large or heavily obfuscated samples, expect longer run times — the tight-loop and decoding-routine emulation is CPU-intensive; use `--functions` to scope down when you already know the region of interest from CAPA/disassembly.

#### 10. Troubleshooting

| Problem | Cause / Fix |
|---|---|
| FLOSS finds almost no decoded/stack strings on a known-obfuscated sample | Sample may be packed — confirm with DIE/PEStudio and unpack first |
| Very slow analysis (minutes+) | Expected on large binaries with many functions; use `--no-tight-strings` or `--functions` to scope the analysis |
| `ERROR: unsupported file format` | Force the format explicitly with `-f pe`, `-f sc32`, or `-f sc64` |
| Output cluttered with junk static strings | Raise `-n` (minimum length) or use `--only` to exclude the `static` category |
| `vivisect` analysis errors on malformed/corrupted PE | Try repairing the PE headers first, or fall back to plain `strings`/`rabin2` for a rough pass |
| JSON output missing expected fields | Confirm you're on the latest FLOSS release — output schema has changed across major versions |
