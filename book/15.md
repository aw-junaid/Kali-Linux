## Chapter 15 – Active Directory

Active Directory (AD) is the identity and access management backbone of the vast majority of enterprise Windows networks. It centralizes authentication (Kerberos, NTLM), authorization (Group Policy, ACLs), and directory services (LDAP) for users, computers, groups, and trusts across a domain or forest. Because AD sits at the center of nearly every corporate network, it is also one of the highest-value targets during a penetration test or red team engagement — compromising a single low-privileged domain user often leads, through a chain of misconfigurations, excessive permissions, and trust relationships, all the way to Domain Admin or Enterprise Admin.

### 15.1 BloodHound

#### 1. Tool Introduction

BloodHound is a graph-analysis application used to reveal hidden and often unintended relationships within an Active Directory (or Azure AD/Entra ID) environment. It ingests data collected by a companion "ingestor" (usually **SharpHound**, see 15.2, or **AzureHound** for cloud) and loads it into a graph database (Neo4j). Analysts can then run pre-built or custom Cypher queries to instantly identify attack paths — for example, "show me the shortest path from any Domain User to Domain Admin."

BloodHound models AD objects (users, groups, computers, GPOs, OUs, domains) as **nodes** and relationships (`MemberOf`, `AdminTo`, `HasSession`, `GenericAll`, `ForceChangePassword`, `CanRDP`, etc.) as **edges**. This makes it trivial to spot excessive nested group memberships, dangerous ACLs, Kerberoastable accounts, unconstrained delegation, and more — all of which are extremely difficult to identify manually via `net` commands or PowerShell alone.

There are two major branches in active use:
- **BloodHound Legacy (BloodHound GUI, Neo4j + Electron app)** – the original open-source community edition.
- **BloodHound Community Edition (BHCE)** – the modern, Docker-based, web-UI rewrite (Postgres + Neo4j + API), maintained by SpecterOps, which replaces the legacy Electron app.

This chapter documents both, with emphasis on BHCE since it is the actively maintained version shipped in current Kali releases.

#### 9. Tips and Best Practices

- Always collect data with the **matching version** of SharpHound/AzureHound for your BloodHound version — schema mismatches cause silent ingestion errors.
- Mark your compromised principals as **"Owned"** as you go; this keeps "Shortest Path from Owned" queries accurate and saves huge amounts of manual graph-hunting.
- Use **High Value Targets** tagging on top of the defaults (DA, EA, DC computers) to include business-critical assets (e.g., a CI/CD service account).
- For large environments (100k+ objects), increase Neo4j's heap size (`dbms.memory.heap.max_size`) to avoid OOM crashes during ingestion.
- Combine BloodHound with **Cypher custom queries** from the community (e.g., the `hausec` or `CompassSecurity` custom-queries.json) for ADCS (ESC edges), MSSQL links, and RBCD-specific paths.
- Re-collect data periodically during long engagements — AD state changes, and stale sessions/ACLs will mislead your path analysis.
- Never run BloodHound ingestors on production DCs without client sign-off — SharpHound's LDAP/SMB queries can be noisy and trigger EDR/SIEM alerts.

#### 10. Troubleshooting

| Issue | Cause | Fix |
|---|---|---|
| Web UI stuck on "Loading" | Neo4j/Postgres containers not fully healthy | `docker compose ps`, check `docker compose logs graph-db` |
| Upload fails with "Invalid schema version" | SharpHound version mismatch | Use SharpHound version matching your BHCE release notes |
| Electron GUI crashes on launch | Missing sandbox permissions in container/root | Launch with `bloodhound --no-sandbox` |
| Neo4j refuses connection on 7687 | Neo4j not started / wrong bolt password | `sudo neo4j console`, reset password via `neo4j-admin set-initial-password` |
| Graph extremely slow with large datasets | Insufficient Neo4j heap/pagecache | Edit `neo4j.conf`: increase `dbms.memory.heap.max_size` and `dbms.memory.pagecache.size` |
| "Initial password" never appears in logs | Container restarted after first init | Reset via `docker compose exec app bloodhound-cli config` (BHCE) or reinstall stack |

### 15.2 SharpHound

#### 1. Tool Introduction

SharpHound is the official data collector ("ingestor") for BloodHound. Written in C#, it runs on a Windows host (physical, VM, or a compromised domain-joined machine) and enumerates Active Directory via LDAP and SMB/RPC calls to gather the objects and relationships that BloodHound visualizes: users, groups, computers, GPOs, OUs, trusts, sessions, local admin rights, and ACLs. The output is a set of JSON files bundled into a timestamped `.zip` archive, ready for upload into BloodHound.

SharpHound ships in several forms:
- **SharpHound.exe** – standalone compiled binary.
- **SharpHound.ps1** – PowerShell wrapper (reflectively loads the C# assembly in-memory, useful for AV/EDR evasion during authorized engagements).
- **BloodHound.py** – a Python re-implementation (Impacket-based) that can be run from Linux/Kali directly against a domain without needing a Windows foothold, provided you have valid domain credentials.

This section documents the official C#/PowerShell SharpHound as well as the Linux-native BloodHound.py collector, both commonly used from Kali.

#### 9. Tips and Best Practices

- Use `DCOnly` when you want zero touch on member computers — it pulls sessions from DC security event logs instead, which is far less noisy.
- Always `--zip` your output; BloodHound CE expects a zip (or individual JSONs) — raw unzipped folders from older versions can confuse ingestion.
- Match SharpHound's version to your BloodHound version; the JSON schema changes between major releases.
- On Windows targets with EDR, prefer the PowerShell in-memory reflective load over dropping `SharpHound.exe` to disk, and rename functions/variables if signature-based detection is a concern (only within authorized red-team engagements).
- Use `--Throttle` and `--Jitter` to slow down collection and blend in with normal LDAP/SMB traffic on sensitive engagements.
- Run `CertServices` collection whenever ADCS is present in the environment — it unlocks ESC1–ESC8 edge detection in BloodHound.
- Loop `Session` collection over hours, not minutes — session data is a snapshot and stale within minutes on busy networks.

#### 10. Troubleshooting

| Issue | Cause | Fix |
|---|---|---|
| `Could not find LDAP` / connection refused | Wrong DC IP or firewall blocking 389/636 | Verify with `nmap -p389,636 <dc_ip>`, confirm DNS resolves domain |
| `KRB5KDC_ERR_PREAUTH_FAILED` | Bad credentials or clock skew | Verify creds; sync clock: `sudo ntpdate <dc_ip>` |
| Zero computers collected | Account lacks read rights on target OUs, or WinRM/SMB blocked | Try `DCOnly` first; check firewall rules to member servers |
| SharpHound.exe flagged/blocked by AV/EDR | Signature detection | Use `SharpHound.ps1` reflective load, obfuscate strings (authorized engagements only) |
| Ingestion into BloodHound fails ("invalid zip") | Corrupted transfer or version mismatch | Re-download output, verify SharpHound/BloodHound version compatibility |
| `dns-tcp` resolution errors | UDP DNS blocked on network | Add `--dns-tcp` to bloodhound-python |

### 15.3 Certipy

#### 1. Tool Introduction

Certipy is a Python offensive toolkit for enumerating and abusing misconfigurations in Active Directory Certificate Services (AD CS). AD CS is Microsoft's PKI role, and when misconfigured it opens a family of well-documented privilege-escalation and persistence primitives cataloged by the "Certified Pre-Owned" research (SpecterOps) as ESC1 through ESC16 (Escalation Scenarios). Certipy can enumerate Certificate Authorities and templates, identify which ESC vulnerabilities apply, request certificates abusing those vulnerabilities, and use the resulting certificates to authenticate via PKINIT (obtaining a TGT) or to extract NTLM hashes via the `UnPAC-the-hash` technique. It effectively replaced/extended the older `Certify.exe` (C#/Windows-only) with a fully cross-platform, Linux-native implementation.

#### 9. Tips and Best Practices

- Always start with `certipy-ad find -vulnerable` before attempting any exploitation — it tells you exactly which ESC applies.
- Use `-old-bloodhound` / `-bloodhound` output to correlate ADCS findings directly inside your BloodHound graph alongside ACL/session data.
- ESC1 requires the "Client Authentication" EKU and enrollee-supplied-subject enabled — verify both in `find` output before attempting `req`.
- After obtaining a PFX, always run `certipy-ad auth` — this both gets you a TGT (Kerberos) AND, critically, the NT hash via UnPAC-the-hash for pass-the-hash flexibility.
- For ESC8 relay attacks, combine with `ntlmrelayx`-style coercion tools (PetitPotam, PrinterBug, Coercer) to force machine authentication to your relay listener.
- Clean up any Shadow Credential key material you add during testing (`certipy-ad shadow remove`) to avoid leaving persistence artifacts in a client environment.
- Keep Certipy updated — new ESC techniques (ESC11 through ESC16) are actively researched and added to the tool.

#### 10. Troubleshooting

| Issue | Cause | Fix |
|---|---|---|
| `KDC_ERR_PREAUTH_FAILED` during `find`/`req` | Wrong credentials or clock skew | Verify creds, sync time with `ntpdate <dc_ip>` |
| `CA not accessible via RPC` | Firewall/RPC dynamic ports blocked | Use `-web` flag to try HTTP enrollment instead, or check port 445/135 access |
| `Access is denied` on `req` | Account lacks Enroll permission on the template | Re-run `find -vulnerable` to confirm actual enrollment rights |
| `auth` fails to get NT hash | PKINIT succeeded but UnPAC not supported by DC (older Windows) | Use `-no-hash` and rely solely on the TGT/ccache for further attacks |
| Relay (ESC8) never receives connections | No coercion triggered, or HTTP enrollment not enabled on CA | Verify with `find`, enable coercion (e.g. PetitPotam) targeting the relay listener |
| `Failed to resolve` domain controller | DNS not pointing to AD | Add `-dc-ip` explicitly, or set `/etc/resolv.conf` to the DC |

### 15.4 Kerbrute

#### 1. Tool Introduction

Kerbrute is a fast, Go-based tool for performing Kerberos pre-authentication attacks against Active Directory: username enumeration, password spraying, brute-forcing, and password validation. Its core trick is abusing the Kerberos AS-REQ exchange: when a client requests a TGT for a username that does not exist, the KDC responds with `KRB5KDC_ERR_C_PRINCIPAL_UNKNOWN`; if the username exists but the password is wrong, it responds with `KRB5KDC_ERR_PREAUTH_FAILED`. This distinction lets Kerbrute enumerate valid usernames **without generating a single Windows Security Event ID 4625 (failed logon)** on the target — because pre-authentication failures are logged differently (Event ID 4771) and username enumeration via AS-REQ generates far less noise than LDAP/SMB-based enumeration, making it a favored stealthy first step.

#### 9. Tips and Best Practices

- Always run `userenum` first to build a validated username list — spraying against unvalidated names wastes attempts and increases lockout risk unnecessarily.
- Check the domain's account lockout policy (threshold, observation window) before spraying; use `--safe` and pace attempts (`--delay`) to stay under the threshold.
- Kerberos pre-auth enumeration is far quieter than LDAP/SMB enumeration but is **not silent** — it still generates Kerberos AS-REQ traffic and can trigger Event ID 4768/4771 monitoring on well-instrumented environments.
- Combine with realistic username-format generation (e.g., `firstname.lastname`, `flastname`) derived from OSINT for higher success rates.
- Space out spray attempts across different days/times to mimic normal failed-login patterns and avoid a spike that triggers SOC alerting.
- Use `--dc` to target the DC directly by IP when DNS isn't configured on your attack host, avoiding resolution failures.

### 15.5 Rubeus

#### Introduction

Rubeus is a C# toolset built for raw Kerberos interaction and abuse, originally derived from Benjamin Delpy's Kekeo project and developed/maintained by the GhostPack collection (SpecterOps). It is used to perform Kerberoasting, AS-REP roasting, ticket requesting/renewal, overpass-the-hash, pass-the-ticket, ticket harvesting/monitoring, S4U2Self/S4U2Proxy abuse (constrained delegation), Golden/Silver ticket creation support, and Kerberos ticket manipulation on Windows hosts. Rubeus is a **Windows-only** binary (it targets the .NET Framework and Win32 Kerberos APIs), so on Kali Linux it is typically staged onto a compromised Windows host, executed via C2, or run through Wine for testing — it is not natively run against Linux targets. Kali is normally the platform used to *host, compile, or transfer* Rubeus, while the tool itself executes on the Windows target/pivot.

#### Tips and Best Practices

- Prefer `/opsec` and AES-based requests over RC4 where possible; RC4 TGS requests for kerberoasting are a well-known detection signature (Event ID 4769 with encryption type `0x17`).
- Use `/simple` output when scripting hash extraction to avoid parsing extra metadata.
- Always clean up injected tickets with `purge` when finished on a shared/interactive host to reduce forensic footprint.
- Combine with PowerView/BloodHound output to target only accounts with SPNs set on privileged users, rather than kerberoasting the entire domain (noisy and often low-value).
- Store harvested `.kirbi` tickets off-host quickly; LSASS-resident tickets can be cleared by logoff/reboot.

#### Troubleshooting

- **`KDC_ERR_PREAUTH_FAILED`**: wrong password/hash, or the account requires pre-auth (AS-REP roasting doesn't apply).
- **`KRB-ERR-RESPONSE-TOO-BIG` / connection errors**: ensure UDP/TCP 88 is reachable to the DC; use `/dc:<fqdn>` explicitly rather than relying on DNS discovery.
- **Empty kerberoast results**: confirm the requesting account has read rights to `servicePrincipalName` attributes (usually true for any authenticated user) and that `/ldapfilter` isn't overly restrictive.
- **`ptt` succeeds but access still denied**: verify the ticket's target SPN actually matches the resource being accessed, and that the ticket hasn't expired.
- **AV/EDR flags Rubeus.exe**: use `execute-assembly` from a C2 to run in-memory rather than dropping the binary to disk, or obfuscate/rename the assembly.

### 15.6 Mimikatz

Mimikatz is one of the most widely used post-exploitation credential-access tools in the Windows security ecosystem. Originally created by Benjamin Delpy (gentilkiwi) as a proof-of-concept to demonstrate flaws in Microsoft's authentication implementations, it has since become a staple tool in penetration testing, red teaming, and Active Directory security assessments. Mimikatz is capable of extracting plaintext passwords, password hashes, PIN codes, and Kerberos tickets directly from memory (LSASS process), as well as performing attacks such as Pass-the-Hash, Pass-the-Ticket, Over-Pass-the-Hash, and Golden/Silver Ticket forgery. On Kali Linux, Mimikatz itself is a native Windows binary (written in C, compiled for Windows), so it is typically staged and executed against a compromised Windows target rather than run natively on the Linux attack box. However, Kali provides several supporting tools and a Linux-compatible implementation path so operators can prepare, deliver, and interact with Mimikatz output as part of a broader engagement workflow.

This chapter explains what Mimikatz does, how to obtain and stage it from Kali Linux, its command syntax and options, and how to use it effectively during an authorized penetration test.

#### 15.6.1 Introduction

Mimikatz was designed to interact directly with the Windows Local Security Authority Subsystem Service (LSASS), the process responsible for enforcing security policy and storing credential material in memory after a user authenticates. Because Windows caches credentials in memory to support Single Sign-On (SSO), Mimikatz can read this memory space (with sufficient privileges) and extract secrets such as:

- Cleartext passwords (on older/unpatched systems, or when WDigest is enabled)
- NTLM password hashes
- Kerberos tickets (TGTs and service tickets)
- Cached domain credentials
- DPAPI master keys and protected secrets

Mimikatz is modular, organized into functional groups ("modules") such as `sekurlsa`, `lsadump`, `kerberos`, `crypto`, `vault`, and `ts` (Terminal Services). It requires local Administrator or SYSTEM privileges on the target host to access LSASS memory, and it is frequently flagged by antivirus and EDR products, making evasion (obfuscation, in-memory execution, or use of forks) a common companion topic.

In a Kali Linux context, Mimikatz is generally:
- Downloaded/staged onto Kali as a Windows PE binary
- Transferred to a compromised Windows host (via SMB, HTTP, Evil-WinRM, etc.)
- Executed on that Windows host, either interactively or via a remote shell
- Its output is then parsed and used for lateral movement or privilege escalation

#### 15.6.2 Installation

Mimikatz is not a native Linux tool, so "installation" on Kali means fetching the compiled Windows binaries so they can be staged to a target.

#### 15.6.3 Syntax

Mimikatz is executed on the Windows target and uses a command interpreter with module-scoped commands. The general syntax pattern is:

```
mimikatz.exe [module::command] [arguments]
```

It can be run interactively (dropping into a `mimikatz #` prompt) or non-interactively by chaining commands with the `-c` style invocation used by post-exploitation frameworks (e.g., Metasploit's `load kiwi`, or PowerShell wrappers).

Interactive invocation:

```
C:\Windows\Temp> mimikatz.exe
```

This opens the interactive shell:


#### 15.6.4 Command Line Options / Modules

Mimikatz is organized by module. The most commonly used modules and commands in an authorized engagement are:

##### 15.6.4.1 privilege module

- `privilege::debug` — requests SeDebugPrivilege, required before touching LSASS memory.

##### 15.6.4.2 sekurlsa module (LSASS memory credential extraction)

- `sekurlsa::logonpasswords` — dumps credentials of all logged-on users from LSASS memory.
- `sekurlsa::wdigest` — dumps WDigest cleartext credentials (if enabled).
- `sekurlsa::tickets` — lists Kerberos tickets currently in memory.
- `sekurlsa::pth` — performs a Pass-the-Hash attack (`/user`, `/domain`, `/ntlm`, `/run`).
- `sekurlsa::minidump` — loads a previously captured LSASS minidump file for offline parsing.

##### 15.6.4.3 lsadump module (SAM/registry/domain secrets)

- `lsadump::sam` — dumps local SAM database hashes (requires access to registry hives or live SAM).
- `lsadump::secrets` — dumps LSA secrets from the registry.
- `lsadump::cache` — dumps cached domain logon credentials (mscash).
- `lsadump::dcsync` — performs a DCSync attack, impersonating a Domain Controller to request password data for a given account (`/user:<target>`).

##### 15.6.4.4 kerberos module

- `kerberos::list` — lists Kerberos tickets for the current session.
- `kerberos::ptt` — Pass-the-Ticket, injects a `.kirbi` ticket file into the current session.
- `kerberos::golden` — forges a Golden Ticket (requires krbtgt hash, domain SID, domain name).
- `kerberos::purge` — clears all Kerberos tickets from the current session.

##### 15.6.4.5 crypto module

- `crypto::capi` — patches CryptoAPI to allow exporting non-exportable private keys.
- `crypto::certificates` — lists/exports certificates and private keys.

##### 15.6.4.6 token module

- `token::list` — lists available access tokens.
- `token::elevate` — impersonates a listed token (commonly SYSTEM).

##### 15.6.4.7 misc / event / process modules

- `misc::cmd` — restores `cmd.exe` if disabled by GPO.
- `event::drop` — attempts to blind Windows Event Logging.
- `process::list` — lists running processes with PIDs.

##### 15.6.4.8 general options

- `exit` — closes the Mimikatz session.
- `log <file>` — logs all console output to a file.

#### 15.6.7 Common Use Cases

- **Post-exploitation credential harvesting** — after gaining local admin on a workstation, dump all cached logon credentials for lateral movement.

- **Domain persistence testing** — demonstrating impact of a Golden Ticket attack when the krbtgt hash is compromised, to justify a krbtgt password reset (twice) as a remediation.

- **Credential exposure audits** — checking whether WDigest is enabled, exposing cleartext passwords in memory, and recommending it be disabled via GPO.

#### 15.6.9 Tips and Best Practices

- Always confirm written authorization and scope before extracting or handling live credentials.
- Run `privilege::debug` before any `sekurlsa` command; without SeDebugPrivilege, LSASS access will fail.
- Prefer taking an LSASS memory dump (`procdump -ma lsass.exe`) and parsing it offline with `sekurlsa::minidump` to reduce time spent live in a monitored process.
- Expect AV/EDR detection; understand that evasion techniques (in-memory loading via reflective DLL, patchless AMSI bypass, forked/obfuscated builds) may be needed and should be authorized as part of the engagement's rules of engagement.
- Immediately treat recovered credentials (especially krbtgt and Domain Admin hashes) as extremely sensitive; store, transmit, and dispose of them per the engagement's data-handling policy.
- After a Golden Ticket demonstration, recommend resetting the krbtgt password twice (to invalidate both current and previous hash) as remediation guidance.
- Use `log` to keep an audit trail of every command executed, which is useful both for reporting and for chain-of-custody purposes.

#### 15.6.10 Troubleshooting

| Issue | Likely Cause | Resolution |
|---|---|---|
| `ERROR kuhl_m_sekurlsa_acquireLSA` | Not running as Administrator/SYSTEM | Re-launch elevated (`runas`, or via PsExec/Evil-WinRM as an admin account) |
| `privilege::debug` returns `ERROR kuhl_m_privilege_simple` | Token doesn't have SeDebugPrivilege available | Ensure the account used has local Administrator rights; UAC may need bypassing |
| Mimikatz binary deleted immediately after write | AV/EDR quarantined the file | Use an unmodified but currently undetected build, in-memory execution, or authorized AV exclusion for the test window |

| Execution blocked by Windows Defender real-time protection | Signature-based detection of mimikatz.exe | Coordinate a defender exclusion with the client, or use process injection/loader techniques within agreed scope |

### 15.7 Evil-WinRM

Evil-WinRM is a Ruby-based tool built for penetration testers to interact with Windows Remote Management (WinRM), a management protocol Windows uses to allow remote command execution over HTTP/HTTPS (typically ports 5985/5986). Unlike raw `winrs` or PowerShell Remoting on Windows, Evil-WinRM is purpose-built for offensive engagements: it provides a feature-rich interactive shell with capabilities like file upload/download, in-memory PowerShell script/DLL loading (bypassing disk-based AV signatures), Pass-the-Hash authentication, and colorized output, all runnable natively from Kali Linux. It has become the de facto standard tool for gaining and interacting with a shell on a Windows target once valid credentials or hashes are known, and is heavily used in Active Directory penetration testing, OSCP-style exams, and CTF competitions such as those on Hack The Box.

This chapter covers Evil-WinRM's background, installation on Kali Linux, full command-line option reference, usage patterns, and practical operational guidance.

#### 15.7.1 Introduction

WinRM is Microsoft's implementation of the WS-Management protocol, enabling remote management and script execution on Windows hosts. It is enabled by default on Windows Server editions and can be enabled on workstations via `Enable-PSRemoting`. WinRM listens on TCP port 5985 (HTTP) and 5986 (HTTPS) by default.

Evil-WinRM connects to a target's WinRM service and provides an interactive pseudo-shell equivalent to a PowerShell session, but with additional attacker-oriented conveniences:

- Authentication via password, NTLM hash (Pass-the-Hash), or Kerberos ticket
- Upload and download files between attacker and target
- Load and execute PowerShell scripts and C# assemblies reflectively (in memory, avoiding disk writes)
- Command history and tab-completion
- Support for SSL/TLS connections
- Logging of session activity

Because it operates over a legitimate, commonly-open Windows management protocol, Evil-WinRM often blends in better than dropping a custom reverse shell payload, and is frequently the preferred lateral-movement and access tool once valid domain or local credentials have been obtained (e.g., via Mimikatz, Kerberoasting, or password spraying).

#### 15.7.7 Common Use Cases

- **Post-credential-compromise shell access** — after obtaining valid domain credentials or an NTLM hash (via Mimikatz, Responder, Kerberoasting, etc.), Evil-WinRM provides an immediate interactive shell.
- **Lateral movement in Active Directory environments** — moving between domain-joined hosts that expose WinRM (common on member servers and domain controllers).
- **In-memory tool execution** — running enumeration or privilege escalation tools (winPEAS, PowerView, SharpHound) without writing them to disk, reducing AV detection.
- **CTF and certification exam workflows** — Evil-WinRM is the standard shell tool for Hack The Box, OSCP, and similar hands-on exams whenever WinRM (port 5985/5986) is open.
- **File exfiltration/infiltration during engagements** — using `upload`/`download` to move loot and tools between attacker and target machine.
- **Data collection for AD attack path mapping** — loading and running SharpHound via `Invoke-Binary` to collect BloodHound ingestion data.

#### 15.7.9 Tips and Best Practices

- Confirm WinRM is actually open before attempting a connection: `nmap -p5985,5986 <target>`.
- Use `-H` (Pass-the-Hash) whenever only an NTLM hash is available; there's no need to crack it first.
- Preload commonly used offensive PowerShell scripts and binaries with `-s`/`-e` at connection time to save time mid-engagement.
- Prefer `Invoke-Binary` for running unmanaged EXEs in memory rather than uploading them to disk, to reduce forensic footprint and AV detection.
- Use `-log` on every engagement session for accurate reporting and evidence of actions taken.
- When WinRM is closed but WinRM can be remotely enabled with existing admin creds, note this as a finding; enabling it yourself for further access should stay within engagement scope.
- Combine Evil-WinRM with BloodHound/SharpHound data collection for effective, low-noise Active Directory attack path discovery.
- Watch for account lockout policies before password spraying against WinRM; a single bad guess against many accounts is safer than many guesses against one.

#### 15.7.10 Troubleshooting

| Issue | Likely Cause | Resolution |
|---|---|---|
| `WinRM::WinRMAuthorizationError` | Invalid credentials or account lacks remote access rights | Verify credentials; confirm the account is in `Remote Management Users` or is a local/domain admin |
| Connection times out | WinRM service not running or port filtered by firewall | Confirm with `nmap -p5985 <target>`; enable WinRM via other access if authorized |
| `OpenSSL::SSL::SSLError` | Mismatched SSL settings for port 5986 | Add `--ssl` flag and confirm target certificate; try without `-P 5986` if SSL isn't actually configured |
| `Errno::ECONNREFUSED` | Wrong IP/port, or WinRM disabled | Double check target IP and confirm the service with `nmap -sV -p5985,5986` |
| Ruby gem load errors on startup | Missing/broken Ruby gem dependencies | Reinstall with `sudo gem install winrm winrm-fs evil-winrm` |
| `Invoke-Binary` fails silently | Target has AMSI/Defender actively blocking in-memory execution | Use an AMSI bypass technique within engagement scope, or an alternate loader |
| Kerberos auth fails with `-r` | Clock skew between attacker and DC, or missing KRB5 config | Sync time with `ntpdate`/`chronyd`; verify `/etc/krb5.conf` realm settings |

### 15.8 ldapdomaindump

ldapdomaindump is a Python-based Active Directory enumeration tool that connects to a domain controller over LDAP and dumps comprehensive information about domain users, groups, computers, trusts, and policies into both human-readable HTML reports and machine-readable JSON/greppable files. Written by Dirk-jan Mollema as part of the "dirkjanm" toolset (the same author behind `mitm6` and `pywerview`), it is widely used during the internal reconnaissance phase of Active Directory penetration tests because it requires nothing more than valid (even low-privileged) domain credentials, or in some misconfigured environments, an anonymous/null LDAP bind. The output gives an attacker (or auditor) a fast, structured overview of the domain's user base, privileged group memberships, password policy, and computer inventory — all without needing to run noisy tools against every host individually.

This chapter details the tool's purpose, installation, syntax, options, and practical use on Kali Linux.

#### 15.8.1 Introduction

Lightweight Directory Access Protocol (LDAP) is the primary protocol Active Directory uses to store and query directory information — users, groups, computers, organizational units, group policies, and trust relationships. Any authenticated domain user (by default) can query a significant portion of this data, making LDAP enumeration one of the highest-value, lowest-noise reconnaissance techniques in an AD environment.

ldapdomaindump automates this process. It performs an LDAP bind to a target domain controller and systematically enumerates:

- Domain users, including attributes like last logon, password last set, and account flags (disabled, locked out, "password never expires," etc.)
- Domain groups and their memberships, especially privileged groups (Domain Admins, Enterprise Admins)
- Computer accounts (workstations and servers) with OS version information
- Domain trusts
- Domain password and lockout policy
- Group Policy Object (GPO) metadata

It produces output in several formats simultaneously: styled HTML pages (for easy browsing), JSON (for scripting/parsing), and grep-friendly plaintext, making it equally useful for manual review and downstream automation (e.g., feeding data into BloodHound-style analysis or custom scripts).

#### 15.8.7 Common Use Cases

- **Initial internal reconnaissance** — as soon as any valid domain credentials (even a low-privileged user) are obtained, run ldapdomaindump to get a full picture of the domain structure.
- **Identifying privileged accounts and groups** — quickly locating Domain Admins, Enterprise Admins, and other high-value group memberships for targeting.
- **Password policy assessment** — checking minimum password length, lockout threshold, and complexity requirements to plan safe password-spraying attacks that avoid account lockout.
- **Stale/misconfigured account discovery** — identifying accounts with "password never expires," disabled accounts still in privileged groups, or accounts with old `pwdLastSet` values.
- **Computer inventory and OS fingerprinting** — reviewing `operatingSystem` attributes across `domain_computers` output to find outdated/unsupported OS versions (e.g., Windows Server 2008, Windows 7).
- **Feeding downstream tooling** — using the JSON output as input for custom scripts, or cross-referencing with BloodHound data for a fuller attack-path picture.
- **Trust relationship mapping** — reviewing `domain_trusts` output to identify cross-domain or cross-forest trusts that could be leveraged for lateral movement.

#### 15.8.8.1 Bash script to run a full dump and organize output by date

```bash
#!/bin/bash
# run_ldapdomaindump.sh - automate a dated domain dump
DC_IP="10.10.10.5"
USER='CORP\jsmith'
PASS='Summer2026!'

```

```bash
    echo "Usage: $0 <path_to_dump_dir>"
```

##### 15.8.8.3 Bash script to loop the dump across multiple discovered domain controllers

```bash
#!/bin/bash
# multi_dc_dump.sh - run ldapdomaindump against every DC in a list
DC_LIST="dc_ips.txt"
USER='CORP\jsmith'
PASS='Summer2026!'

while read -r dc; do
    echo "[*] Dumping $dc"
    ldapdomaindump -u "$USER" -p "$PASS" -o "/home/kali/loot/dc_${dc//./_}" "ldap://${dc}"
```

#### 15.8.9 Tips and Best Practices

- Always attempt with valid low-privileged credentials first; domain-wide LDAP read access rarely requires elevated rights.
- Test an anonymous bind (`ldapdomaindump ldap://<target>` with no `-u`/`-p`) early — some legacy or misconfigured DCs still permit it and require zero credentials.
- Use `-r` sparingly on large domains; hostname resolution significantly increases run time.
- Review `domain_policy` output before running any password spraying attack, to calculate a safe number of guesses under the lockout threshold.
- Cross-reference `domain_users_by_group.html` for a fast visual view of nested group membership, especially for privileged groups.
- Store dump output securely; it contains a full inventory of the organization's users, computers, and security policy, which is sensitive reconnaissance data.
- Combine ldapdomaindump's fast overview with BloodHound/SharpHound for deeper attack-path graph analysis; they complement rather than replace each other.
- Use `--no-html` on engagements where you only need machine-parseable data, to save disk space and processing time.

#### 15.8.10 Troubleshooting

| Issue | Likely Cause | Resolution |
|---|---|---|
| `ldap3.core.exceptions.LDAPBindError` | Invalid credentials or wrong auth type | Verify username format (`DOMAIN\user`); try `-at SIMPLE` if NTLM bind fails |
| Connection refused | LDAP port closed or filtered | Confirm with `nmap -p389,636,3268 <target>`; check network path/firewall rules |
| Anonymous bind fails | DC does not permit anonymous LDAP (default on modern AD) | Use valid credentials instead; anonymous bind is increasingly rare on hardened domains |
| Dump completes but files are nearly empty | Bind account has extremely restricted read permissions | Confirm account isn't in a restricted OU; test with a different low-privileged account if available |
| SSL handshake failure with `-s` | Target requires a specific TLS version or valid certificate chain | Try without `-s` on port 389 first, or verify DC's LDAPS certificate configuration |
| Script hangs on large domains | Full (non-minimal) dump against a very large AD environment | Use `-m` for a minimal/faster dump, or `--no-json`/`--no-html` to reduce processing overhead |
| `dns.resolver.NXDOMAIN` errors with `-r` | Internal DNS not reachable from Kali | Specify the domain's internal DNS server explicitly with `-n <dns_ip>` |
