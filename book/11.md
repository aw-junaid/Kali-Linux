## Chapter 11 – Wireless Security

Wireless networks are one of the most attacked surfaces in modern IT environments. Unlike wired networks, wireless signals travel through the air and beyond physical walls, making them accessible to anyone within range — including attackers sitting in a parking lot, a neighboring apartment, or a coffee shop across the street. This chapter is dedicated to **wireless security auditing and penetration testing**, covering the most widely used tools in Kali Linux for assessing the security posture of Wi-Fi networks.

Wireless penetration testing involves identifying weaknesses in wireless encryption protocols (WEP, WPA, WPA2, WPA3), authentication mechanisms (WPS, PSK, Enterprise/802.1X), and client-side configurations. The tools covered in this chapter allow a security professional to:

- Discover and enumerate wireless networks and connected clients
- Capture wireless traffic, including authentication handshakes
- Test the strength of pre-shared keys (PSK) through dictionary and brute-force attacks
- Identify and exploit vulnerabilities in WPS (Wi-Fi Protected Setup)
- Perform man-in-the-middle (MITM) attacks on wireless clients
- Convert and process captured data for offline password cracking

All tools discussed in this chapter are pre-installed or easily installable on **Kali Linux**, the industry-standard penetration testing distribution.

> **Legal and Ethical Notice:** Every technique, command, and example in this chapter is intended strictly for **authorized penetration testing, security research, Capture The Flag (CTF) competitions, and educational lab environments** using equipment and networks you own or have explicit written permission to test. Unauthorized access to wireless networks, interception of communications, or denial-of-service attacks (such as deauthentication) against networks you do not own or have authorization to test is illegal in most jurisdictions under laws such as the Computer Fraud and Abuse Act (CFAA), the UK Computer Misuse Act, and equivalent legislation worldwide. Always obtain a signed scope-of-work or rules-of-engagement document before testing any wireless network that is not your own.

#### Prerequisites

Before working through this chapter, you should have:

- A working installation of **Kali Linux** (2023.x or later recommended), either bare-metal, in a VM, or on a dedicated pentest laptop
- A **wireless network adapter that supports monitor mode and packet injection** (e.g., Alfa AWUS036ACH, AWUS036NHA, Panda PAU09, or similar chipsets such as Atheros AR9271 or Ralink RT3070/RT5372)
- Basic familiarity with the Linux command line
- A basic understanding of networking concepts (MAC addresses, SSIDs, BSSIDs, channels, frequency bands 2.4GHz/5GHz)
- An isolated lab access point (your own router) configured for testing, ideally with WPA2-PSK and WPS enabled/disabled as needed for different exercises

### 11.1 Aircrack-ng Suite

#### 1. Tool Introduction

**Aircrack-ng** is the most widely known and foundational suite of tools for assessing Wi-Fi network security. First released in 2006 as a successor to the original Aircrack tool, it has grown into a complete ecosystem of command-line utilities that cover every stage of a wireless assessment: monitoring, packet capture, traffic injection, deauthentication, and cryptographic key recovery.

The suite is not a single binary but a collection of specialized tools, each handling one part of the workflow:

- **airmon-ng** – Enables and manages monitor mode on wireless interfaces, and detects/kills interfering processes.
- **airodump-ng** – Captures raw 802.11 frames, displays nearby access points and clients, and writes capture files (`.cap`/`.pcap`) including WPA/WPA2 handshakes.
- **aireplay-ng** – Generates and injects traffic, most commonly used for deauthentication attacks to force a handshake, as well as fake authentication, ARP replay, and fragmentation attacks.
- **aircrack-ng** – The cracking engine itself; takes a capture file and a wordlist (or uses statistical attacks for WEP) to recover the encryption key.
- **airdecap-ng** – Decrypts WEP/WPA/WPA2 capture files once the key is known.
- **airbase-ng** – Turns a wireless card into a fake/rogue access point.
- **airolib-ng** – Precomputes and stores Pairwise Master Keys (PMKs) for faster cracking against known SSIDs.
- **packetforge-ng** – Creates custom encrypted packets for injection.

Aircrack-ng supports WEP and WPA/WPA2-PSK cracking, and (with limitations) can be used in workflows that touch WPA3 transition-mode networks, though WPA3-SAE itself is designed to resist offline dictionary attacks.

#### 7. Common Use Cases

- **Baseline reconnaissance:** discovering all wireless networks and clients in range before an engagement.
- **WPA/WPA2 handshake capture and offline cracking** against weak pre-shared keys.
- **Forcing reconnection via deauthentication** to capture a fresh handshake or PMKID.
- **Legacy WEP auditing** on older infrastructure still using deprecated encryption.
- **Rogue AP / evil twin testing** to assess client behavior against untrusted networks.
- **Injection capability testing** of wireless adapters before an engagement.
- **Building precomputed hash databases** (`airolib-ng`) to accelerate cracking against a known SSID across multiple engagements.

#### 8. Automation with Bash

The following script automates monitor mode setup, target scanning, handshake capture, and cracking:

```bash
#!/bin/bash
# auto_aircrack.sh - Automated WPA2 handshake capture & crack
# Usage: sudo ./auto_aircrack.sh <interface> <BSSID> <channel> <wordlist>

set -e

IFACE=$1
BSSID=$2
CHANNEL=$3
WORDLIST=$4
CAPDIR="./captures_$(date +%Y%m%d_%H%M%S)"

if [[ -z "$IFACE" || -z "$BSSID" || -z "$CHANNEL" || -z "$WORDLIST" ]]; then
    echo "Usage: sudo $0 <interface> <BSSID> <channel> <wordlist>"
    exit 1
fi

mkdir -p "$CAPDIR"

echo "[*] Killing interfering processes..."
airmon-ng check kill

echo "[*] Enabling monitor mode on $IFACE..."
airmon-ng start "$IFACE"
MONIF="${IFACE}mon"

echo "[*] Starting capture on channel $CHANNEL..."
airodump-ng --bssid "$BSSID" -c "$CHANNEL" -w "$CAPDIR/capture" "$MONIF" &
DUMP_PID=$!

sleep 5
echo "[*] Sending deauth to force handshake..."
aireplay-ng -0 10 -a "$BSSID" "$MONIF"

echo "[*] Waiting for handshake capture (30s)..."
sleep 30
kill $DUMP_PID 2>/dev/null || true

CAPFILE=$(ls "$CAPDIR"/capture-01.cap 2>/dev/null)
if [[ -z "$CAPFILE" ]]; then
    echo "[!] No capture file found."
    exit 1
fi

echo "[*] Attempting crack with wordlist $WORDLIST..."
aircrack-ng -w "$WORDLIST" -b "$BSSID" "$CAPFILE" | tee "$CAPDIR/crack_result.txt"

echo "[*] Restoring interface to managed mode..."
airmon-ng stop "$MONIF"
service NetworkManager start

echo "[*] Done. Results saved in $CAPDIR/"
```

Run it:

```bash
sudo chmod +x auto_aircrack.sh
sudo ./auto_aircrack.sh wlan0 AA:BB:CC:DD:EE:01 6 /usr/share/wordlists/rockyou.txt
```

#### 9. Tips and Best Practices

- Always run `airmon-ng check kill` before starting monitor mode to prevent NetworkManager from resetting the interface mid-capture.
- Lock `airodump-ng` to a single channel (`-c`) once you've identified your target — this dramatically increases capture reliability compared to channel-hopping.
- Save captures in `.cap` format (not `.ivs`) if you plan to later convert them for Hashcat/John — `.ivs` discards data needed for WPA cracking.
- Prefer capturing the **PMKID** (via a clientless attack, see `hcxdumptool`) over deauth-based handshake capture when possible — it's faster and doesn't require a connected client.
- Use `-p <n>` in `aircrack-ng` to match your CPU thread count for faster dictionary attacks.
- Verify a "captured" handshake actually contains all 4 EAPOL frames before spending hours cracking it — a partial handshake cannot be cracked.
- Keep your adapter's firmware and drivers up to date; injection reliability varies significantly across chipsets.

#### 10. Troubleshooting

| Problem | Cause | Solution |
|---|---|---|
| `Interface doesn't support monitor mode` | Chipset/driver limitation | Use a supported adapter (Atheros, Ralink chipsets) |
| Fixed channel -1 error in airodump-ng | Driver quirk | Add `-i, --ignore-negative-one` flag |
| No handshake captured after deauth | Client not reconnecting, too far, or filtering deauth frames | Increase `-0` count, move closer, target a different client |
| `aircrack-ng` says "No valid WPA handshakes found" | Only 2/4 EAPOL frames captured | Re-run deauth attack, capture longer |
| Monitor mode interface disappears | NetworkManager reasserting control | Re-run `airmon-ng check kill`, disable NetworkManager for that interface |
| Injection test fails (`aireplay-ng -9`) | Adapter doesn't support injection, or too far from AP | Try a different adapter, move closer, change channel |
| `airbase-ng` clients can't get IP | No DHCP server bound to `at0` | Configure `dnsmasq`/`dhcpd` on the `at0` interface |

### 11.2 Wifite

#### 1. Tool Introduction

**Wifite** is a Python-based automated wireless auditing tool designed to attack multiple WEP, WPA, WPA2, and WPS-enabled networks in sequence with minimal user interaction. Rather than manually running `airmon-ng`, `airodump-ng`, `aireplay-ng`, `reaver`, and `bully` separately, Wifite orchestrates all of these tools automatically, picking the most effective attack for each target based on its detected configuration.

Wifite is commonly used for:
- Rapid, hands-off surveys of an entire wireless environment
- Automatically capturing handshakes and PMKIDs from every visible network
- Automatically attempting WPS PIN attacks where WPS is enabled
- Batch-cracking captured handshakes against a wordlist at the end of a session

The current actively maintained version is **Wifite2**, written by `derv82`, which is what ships in modern Kali Linux.

Wifite is pre-installed on Kali Linux. To verify or reinstall:

#### 9. Tips and Best Practices

- Use `--kill` on every run to avoid NetworkManager interference mid-scan.
- Start with `--pmkid` only for a fast, clientless first pass — it doesn't require deauthenticating anyone and often succeeds instantly against vulnerable routers.
- Use `--pow <n>` to focus effort on the strongest, closest targets first rather than wasting time on weak, unreliable signals.
- Combine `-c <channel>` with a known target's channel to speed up scanning significantly by skipping channel hopping.
- Review `hs/` (handshakes) and `wpa/` output directories after a sweep — Wifite keeps organized captures you can re-crack later with different wordlists.
- Use `--wps-only` when you specifically want to test WPS resilience without spending time on handshake-based attacks.
- Increase `--wpat` (WPA attack timeout) for networks with low client activity, since deauth-triggered reconnections can take longer.

### 11.3 Reaver

#### 1. Tool Introduction

**Reaver** is a tool that performs a brute-force attack against the **WPS (Wi-Fi Protected Setup)** PIN authentication mechanism found on many consumer and small-business wireless routers. WPS was designed to make it easy for non-technical users to connect devices to a Wi-Fi network using an 8-digit PIN instead of typing the full WPA/WPA2 passphrase.

The critical design flaw exploited by Reaver is that the WPS PIN authentication process validates the PIN in two halves (the first 4 digits, then the last 3 digits plus a checksum digit), rather than validating all 8 digits at once. This reduces the effective keyspace from 10^8 (100,000,000) possible PINs down to roughly 11,000 possible combinations, making it feasible to brute-force the PIN — and from it, recover the full WPA/WPA2 passphrase — in a matter of hours (or minutes with certain optimizations/vulnerabilities like "pixie dust" attacks).

Reaver was originally developed by Tactical Network Solutions and released as open source; the most current maintained version is **Reaver 1.6.6** (`reaver-wps-fork-t6x`), which ships with Kali Linux.

```bash
#!/bin/bash
# reaver_auto.sh - Scan for WPS targets and attack each automatically
# Usage: sudo ./reaver_auto.sh <interface>

    echo "Usage: sudo $0 <interface>"

BSSIDS=$(awk '$5=="No" {print $1}' "$LOGDIR/wash_results.txt" | grep -E '([0-9A-F]{2}:){5}[0-9A-F]{2}')

for BSSID in $BSSIDS; do
    echo "[*] Attacking $BSSID with Pixie Dust first..."
    timeout 60 reaver -i "$IFACE" -b "$BSSID" -K 1 -vv | tee "$LOGDIR/${BSSID//:/-}_pixie.log"

    if grep -q "WPA PSK" "$LOGDIR/${BSSID//:/-}_pixie.log"; then
        echo "[+] Cracked $BSSID via Pixie Dust!"
        continue
    fi

    echo "[*] Pixie Dust failed, falling back to brute-force for $BSSID..."
    timeout 3600 reaver -i "$IFACE" -b "$BSSID" -d 2 -l 60 -vv | tee "$LOGDIR/${BSSID//:/-}_brute.log"
done

echo "[*] All attacks complete. Logs saved in $LOGDIR/"
```

#### 9. Tips and Best Practices

- Always run `wash` first — never blindly attack a target that doesn't have WPS enabled.
- Try `-K 1` (Pixie Dust) before a full brute-force run; it's non-intrusive to the AP and, if successful, takes seconds rather than hours.
- Use `-d` (delay) and `-l` (lock-delay) conservatively against APs with lockout protection to avoid triggering extended lockouts.
- Lock the channel with `-c` and `-f` once identified, to avoid wasted time on channel hopping.
- Save session state — Reaver automatically checkpoints progress in `/etc/reaver/` or `/var/lib/reaver/`, allowing you to resume interrupted attacks.
- Combine with `Bully` (see 11.4) as a fallback — different WPS implementations respond better to one tool or the other.
- Recognize that many modern routers (post-2015) have fixed the underlying vulnerability or enforce strict lockouts, making Reaver far less effective against updated firmware.

#### 10. Troubleshooting

| Problem | Cause | Solution |
|---|---|---|
| "WARNING: Failed to associate" | AP out of range or filtering | Move closer, verify BSSID/channel |
| Stuck at 0% for a long time | AP rate-limiting responses | Increase `-d` delay, try `-A` auto mode |
| "WPS transaction failed (code: 0x02/0x03)" | AP timing out mid-exchange | Add `-t` timeout adjustment, try `--dh-small` |
| Repeated lockouts | AP has anti-brute-force lockout enabled | Increase `-l`, be patient, or switch to Pixie Dust/Bully |
| Pixie Dust attack fails | AP uses a strong PRNG (not vulnerable) | Fall back to standard brute-force with `-d`/`-l` tuning |
| Session doesn't resume | Session file corrupted or removed | Delete stale session files in `/var/lib/reaver/`, start fresh |

### 11.4 Bully

#### 1. Tool Introduction

**Bully** is a WPS brute-force attack tool written in C, created as a more robust and portable alternative to Reaver. It targets the same underlying design flaw in the WPS protocol — the two-stage PIN validation — but was built from the ground up to handle edge cases more gracefully: implementations that respond unusually to malformed packets, APs with aggressive rate-limiting, and adapters/drivers that Reaver handles poorly.

Key differences from Reaver:
- Written in C rather than relying on external libraries as heavily, giving it a smaller footprint and better performance on constrained hardware.
- More configurable retry/timeout logic, often succeeding against APs where Reaver stalls or crashes.
- Built-in support for pin-checksum validation and better session-state handling.
- Actively used as Wifite's alternative WPS engine (`--bully` flag).

Bully is maintained as part of the `bully` package in Kali Linux, originally authored by Brad Antoniewicz.

```bash
CHANNEL=$3
ESSID=$4
LOGFILE="bully_${BSSID//:/-}_$(date +%Y%m%d_%H%M%S).log"

if [[ -z "$IFACE" || -z "$BSSID" || -z "$CHANNEL" || -z "$ESSID" ]]; then
    echo "Usage: sudo $0 <interface> <bssid> <channel> <essid>"
    exit 1
fi

echo "[*] Starting Bully attack against $ESSID ($BSSID) on channel $CHANNEL"

sudo bully "$IFACE" -b "$BSSID" -c "$CHANNEL" -e "$ESSID" \
    -d -l 60 -r 3 -t 5 -S -v 3 2>&1 | tee "$LOGFILE"

if grep -q "WPA PSK" "$LOGFILE"; then
    KEY=$(grep "WPA PSK" "$LOGFILE" | awk -F': ' '{print $2}')
    echo "[+] SUCCESS: Recovered PSK for $ESSID: $KEY"
else
    echo "[!] Attack did not complete successfully. Check $LOGFILE for details."
fi
```

#### 9. Tips and Best Practices

- Enable `-d` (lock detection) by default to avoid wasting attempts against a locked AP — this is safer and more polite than force mode.
- Only use `-f` (force) when explicitly testing whether an AP's lockout mechanism can be bypassed, as part of an authorized assessment.
- Use `-S` for any long-running attack — WPS brute-forcing can take hours, and session persistence avoids restarting from PIN zero after an interruption.
- Cross-check results between Bully and Reaver; a target that fails on one frequently succeeds on the other due to subtle protocol-handling differences.
- Increase `-r` (retries) and `-t` (timeout) values when working against distant or weak-signal targets to reduce false "failed" states caused by dropped packets, not actual PIN rejection.
- Combine with `wash` (from the Reaver package) for initial WPS-enabled target discovery, since Bully itself does not include a scanning mode.

#### 10. Troubleshooting

| Problem | Cause | Solution |
|---|---|---|
| "Timeout waiting for beacon" | Wrong channel or AP out of range | Verify channel with `wash`, move closer |
| Constant "M3 timeout" | Weak signal / severe packet loss | Increase `-t` and `-r`, move closer |
| Attack immediately reports "locked" | AP enforces lockout after previous attempts (possibly from Reaver) | Wait out lock period with `-l`, or pause testing |
| PIN never found despite full range tested | AP has patched checksum vuln or uses randomized responses | Target likely not vulnerable; try Pixie Dust via Reaver instead |
| Session doesn't resume with `-S` | Session state file missing/corrupted | Remove stale session files, restart attack from scratch |
| High CPU/network usage on constrained hardware | Verbosity too high, excessive logging | Lower `-v` level, disable file logging |

### 11.5 Kismet

#### 1. Tool Introduction

**Kismet** is a wireless network detector, packet sniffer, and wireless intrusion detection system (WIDS). Unlike active attack tools such as Reaver or Bully, Kismet operates primarily as a **passive monitoring and reconnaissance framework** — it listens to wireless traffic without transmitting, making it useful both for offensive reconnaissance (mapping out an environment before an engagement) and defensive purposes (detecting rogue access points, deauthentication attacks, and other wireless anomalies).

Key characteristics of Kismet:
- Works across multiple RF protocols: 802.11 Wi-Fi (a/b/g/n/ac/ax), Bluetooth, Zigbee, and software-defined radio (SDR) sources.
- Runs as a **client-server** application: a background server process collects data from wireless sources (`kismet_server`), and a web-based UI (`kismet_client` / browser UI on port 2501) displays it.
- Detects hidden (non-broadcasting) SSIDs by observing probe requests/responses and data frames.
- Logs GPS coordinates when paired with a GPS device, enabling wardriving.
- Includes alerting for known attack signatures: deauth floods, WEP/WPA cracking attempts detected via traffic patterns, AP spoofing, and more.
- Exports data in multiple formats (`.kismet` SQLite database, `.pcapng`, CSV, XML) for further analysis or import into other tools.

#### 9. Tips and Best Practices

- Regularly export `.kismet` SQLite logs to `.pcapng` for compatibility with Wireshark and other analysis tools.
- Add your user to the `kismet` group rather than always running as root, for better operational security and easier permission management.
- Use the REST API (`/system/status.json`, `/devices/summary/devices.json`) to integrate Kismet data into custom dashboards or SIEM pipelines.

### 11.6 Bettercap

#### 1. Tool Introduction

**Bettercap** is a powerful, modular, and extensible framework for network reconnaissance and attack, written in Go. While it is best known for wired/Wi-Fi man-in-the-middle (MITM) attacks such as ARP spoofing, it also includes a dedicated **802.11 (Wi-Fi) module** capable of network discovery, deauthentication, handshake/PMKID capture, and rogue access point creation — effectively overlapping with and, in some workflows, replacing parts of the Aircrack-ng suite with a more modern, scriptable interface.

Key capabilities relevant to wireless security:
- **wifi.recon** — passive/active discovery of nearby access points and associated clients.
- **wifi.deauth** — targeted or broadcast deauthentication to force reconnections.
- **wifi.assoc** — association-request based PMKID capture, which can trigger some routers to leak the PMKID without any connected client.
- **wifi.ap** — creation of rogue/fake access points.
- Support for a **Lua-based scripting engine (caplets)**, allowing complex automated attack chains.
- A REST API and web UI for remote control and dashboarding.
- Modules extending beyond Wi-Fi: ARP spoofing, DNS spoofing, HTTP/HTTPS proxying, BLE (Bluetooth Low Energy) attacks, and packet sniffing — making it a genuinely all-in-one network attack platform.

#### 9. Tips and Best Practices

- Try `wifi.assoc` (PMKID capture) before `wifi.deauth` — it's less disruptive since it doesn't require kicking a client off the network, and many home routers are vulnerable to it.
- Use `wifi.recon.channel` to lock onto your target's channel once identified; this significantly improves capture reliability versus full-spectrum hopping.
- Save caplets for repeatable, standardized attack sequences across engagements — this also improves reporting consistency.
- Use `set wifi.deauth.skip` to exclude critical infrastructure devices (e.g., known medical or safety equipment) from deauth during an authorized engagement, per your rules of engagement.
- Enable the REST API only on `127.0.0.1` or a secured interface — exposing it broadly creates a significant security risk on its own.
- Combine Bettercap's Wi-Fi capture output directly with `hcxpcapngtool` (11.8) for Hashcat-ready hash conversion.

### 11.7 hcxdumptool

#### 1. Tool Introduction

**hcxdumptool** is a small, purpose-built utility for capturing raw IEEE 802.11 traffic and actively soliciting **PMKID** and **EAPOL 4-way handshake** data from access points. It was developed as part of the `hcxtools`/`hcxdumptool` project by ZeroBeat specifically to feed data into modern GPU-accelerated cracking workflows using **Hashcat**.

Unlike `airodump-ng`, which passively captures whatever traffic happens to occur, hcxdumptool actively sends crafted frames (association requests, EAPOL requests) designed to solicit a PMKID response from access points that support fast BSS transition/roaming (a leak present in many consumer routers), all **without needing a connected client** or a full deauthentication-based handshake capture. It can also passively capture full 4-way handshakes when clients connect normally, and includes anti-idle/anti-detection features not found in older tools.

Key characteristics:
- Captures data directly to `.pcapng` format, the modern standard for packet capture that also embeds RF metadata.
- Supports simultaneous multi-channel/multi-BSSID targeting.
- Includes built-in filters to target specific BSSIDs, exclude specific clients, and limit capture scope.
- Designed to work seamlessly with `hcxpcapngtool` (11.8) for converting captures into Hashcat-compatible hash formats (`.hc22000`).
- Far less network-disruptive than deauth-heavy tools, since PMKID capture requires no client interaction.

#### 9. Tips and Best Practices

- Prefer hcxdumptool over deauth-heavy `aireplay-ng` workflows when stealth or minimal network disruption is a priority — PMKID capture requires no client interaction at all.
- Always use `--autobpf` to apply an optimized packet filter; it significantly reduces capture file size and processing time.
- Use `--filterlist_ap` with `--filtermode=1` (whitelist) to stay strictly within the authorized scope of an engagement — this is both a technical and ethical safeguard.
- Combine `--exitoneapol` with scripted loops to quickly sweep many targets, spending minimal time per AP once a handshake/PMKID is captured.
- Save all captures in `.pcapng` (not legacy `.cap`) — it's the format `hcxpcapngtool` expects and preserves RF metadata that older formats discard.
- Run `hcxdumptool -F` (disable deauth) as a default in environments with WIDS/WIPS monitoring to avoid triggering alerts unnecessarily.

### 11.8 hcxpcapngtool

#### 1. Tool Introduction

**hcxpcapngtool** is the conversion and analysis component of the `hcxtools` project, designed to process raw capture files (`.pcapng`, `.pcap`, `.cap`) — such as those produced by `hcxdumptool`, `airodump-ng`, `tcpdump`, or `Kismet` — and extract cryptographic material (PMKID and EAPOL 4-way handshakes) into hash formats consumable by modern GPU-accelerated cracking tools like **Hashcat** and **John the Ripper**.

Where `hcxdumptool` (11.7) is the **capture** tool, `hcxpcapngtool` is the **conversion/extraction** tool — the two form a complete pipeline: capture raw wireless traffic, then convert it into a crackable hash. This is now the standard, actively maintained replacement for the older `aircrack-ng -J` / `cap2hccapx` conversion workflow.

Key characteristics:
- Outputs the modern unified **`.hc22000`** hash format (Hashcat mode `22000`), which covers both PMKID and EAPOL-based WPA/WPA2 handshakes in a single format.
- Performs handshake **integrity validation** — checking nonce values, replay counters, and message authentication codes (MICs) to discard corrupted or unusable handshakes before they waste GPU cracking time.
- Extracts additional useful metadata: ESSID lists, BSSID/vendor info, GPS coordinates (if present), and can generate potential candidate wordlists from captured probe requests.
- Supports batch processing of multiple capture files at once.
- Can filter output by ESSID, BSSID, or handshake/PMKID type.

#### 9. Tips and Best Practices

- Always run `-x` (info mode) first on a new capture to verify it actually contains usable PMKID/EAPOL data before spending time on conversion and cracking.
- Prefer the unified `.hc22000` format over legacy `.hccapx`/`.cap` conversions — it supports both PMKID and full handshakes in one file and is the actively maintained standard.
- Use `-E` to extract ESSID lists early in an engagement; they can inform targeted wordlist generation (e.g., appending company name variations).
- Keep `--nonce-error-corrections` at its default unless you specifically need strict validation (`=0`) for evidentiary/reporting purposes, since default settings recover more usable handshakes from imperfect captures.
- Batch-process capture files from multiple sessions/days into a single hash file before a large cracking run, rather than running Hashcat separately per file — this is far more efficient on GPU time.
- Combine `-c`/`--filtermac` filtering with your rules-of-engagement scope document to ensure only authorized targets' data is ever processed or cracked.
