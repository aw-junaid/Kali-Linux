### 27.3 Nmap Cheat Sheet

Nmap is a network discovery and port-scanning tool. This reference covers scan types, host discovery, timing, and the Nmap Scripting Engine (NSE) at a syntax level, for use in authorized testing and lab environments.

#### 1 Basic Scan Syntax

| Command | Purpose |
|---|---|
| `nmap <target>` | Default SYN scan of top 1000 ports |
| `nmap -p- <target>` | Scan all 65535 ports |
| `nmap -p 22,80,443 <target>` | Scan specific ports |
| `nmap -p 1-1024 <target>` | Scan a port range |
| `nmap <cidr>` | Scan an entire subnet, e.g. `192.168.1.0/24` |
| `nmap -iL targets.txt` | Scan targets listed in a file |

#### 2 Host Discovery

| Command | Purpose |
|---|---|
| `nmap -sn <cidr>` | Ping sweep, no port scan |
| `nmap -PR <target>` | ARP ping (local network) |
| `nmap -PE <target>` | ICMP echo ping |
| `nmap -Pn <target>` | Skip host discovery, treat host as up |

#### 3 Scan Types

| Flag | Scan Type |
|---|---|
| `-sS` | TCP SYN scan (stealth/default with privileges) |
| `-sT` | TCP connect scan (no raw sockets needed) |
| `-sU` | UDP scan |
| `-sA` | ACK scan (firewall rule mapping) |
| `-sV` | Service/version detection |
| `-sC` | Run default NSE scripts |
| `-O` | OS detection |

#### 4 Timing and Performance

| Flag | Purpose |
|---|---|
| `-T0` to `-T5` | Timing template, 0 = slowest/stealthiest, 5 = fastest |
| `--min-rate <n>` | Minimum packets per second |
| `--max-retries <n>` | Cap retransmissions |
| `-n` | Skip DNS resolution (faster) |

#### 5 Output Options

| Flag | Purpose |
|---|---|
| `-oN out.txt` | Normal text output |
| `-oX out.xml` | XML output |
| `-oG out.gnmap` | Grepable output |
| `-oA out` | All formats at once, base name `out` |
| `-v` / `-vv` | Increase verbosity |

#### 6 Nmap Scripting Engine (NSE)

| Command | Purpose |
|---|---|
| `nmap --script=default <target>` | Run default script category |
| `nmap --script=vuln <target>` | Run vulnerability-detection scripts |
| `nmap --script=<name> <target>` | Run a specific named script |
| `nmap --script-help=<name>` | Show help for a script |
| `ls /usr/share/nmap/scripts/` | List installed NSE scripts locally |

#### 7 Common Combined Examples

| Command | Purpose |
|---|---|
| `nmap -sS -sV -O -p- -T4 <target>` | Full aggressive TCP scan with version/OS detection |
| `nmap -sU --top-ports 20 <target>` | Quick UDP scan of top 20 ports |
| `nmap -A <target>` | Aggressive scan: OS, version, script, traceroute |

> As with all scanning activity, only run these against systems you are authorized to test — unsolicited scanning of third-party infrastructure may violate computer misuse laws.

### 27.7 Hashcat Cheat Sheet

Hashcat is a GPU-accelerated password recovery tool used in authorized password-audit and lab contexts, for example to test password policy strength on hashes you have legitimate access to.

#### 1 Basic Syntax

```
hashcat -m <hash-mode> -a <attack-mode> <hashfile> <wordlist-or-mask>
```

| Flag | Purpose |
|---|---|
| `-m` | Hash type/mode number |
| `-a` | Attack mode number |
| `-o out.txt` | Write cracked results to a file |
| `--show` | Show already-cracked hashes from the potfile |
| `--status` | Print periodic status updates |

#### 2 Common Hash Modes

| Mode | Hash Type |
|---|---|
| 0 | MD5 |
| 100 | SHA1 |
| 1400 | SHA256 |
| 1700 | SHA512 |
| 1000 | NTLM |
| 3000 | LM |
| 500 | md5crypt (Unix) |
| 1800 | sha512crypt (Unix) |
| 22000 | WPA-PBKDF2-PMKID+EAPOL (Wi-Fi) |

#### 3 Attack Modes

| Mode | Name | Description |
|---|---|---|
| 0 | Straight | Wordlist-based, optionally with rules |
| 1 | Combination | Combines two wordlists |
| 3 | Brute-force/Mask | Character-set-based mask attack |
| 6 | Hybrid Wordlist + Mask | Wordlist with mask appended |
| 7 | Hybrid Mask + Wordlist | Mask prepended to wordlist |

#### 4 Mask Attack Syntax

| Placeholder | Character Set |
|---|---|
| `?l` | lowercase a–z |
| `?u` | uppercase A–Z |
| `?d` | digits 0–9 |
| `?s` | special characters |
| `?a` | all of the above |

Example: `hashcat -m 0 -a 3 hash.txt ?u?l?l?l?l?d?d`

#### 5 Rules

| Command | Purpose |
|---|---|
| `-r rules/best64.rule` | Apply a rule file to mutate wordlist entries |
| `--stdout` | Preview generated candidates without cracking |

#### 6 Performance and Device Options

| Flag | Purpose |
|---|---|
| `-O` | Optimized kernel (faster, some limits on password length) |
| `-w <1-4>` | Workload profile (higher = more aggressive) |
| `-d <device-id>` | Restrict to a specific GPU/device |
| `--force` | Bypass certain hardware/driver warnings (use cautiously) |
| `-I` | Show detected devices |

#### 7 Session and Resume

| Flag | Purpose |
|---|---|
| `--session=<name>` | Name a session for later resume |
| `--restore` | Resume an interrupted session |

### 27.8 Hydra Cheat Sheet

THC-Hydra is a parallelized network login brute-forcer supporting many protocols, intended for authorized credential-strength testing.

#### 1 Basic Syntax

```
hydra -l <user> -P <wordlist> <target> <service>
hydra -L <userlist> -P <wordlist> <target> <service>
```

| Flag | Purpose |
|---|---|
| `-l <user>` | Single username |
| `-L <file>` | Username list |
| `-p <pass>` | Single password |
| `-P <file>` | Password list |
| `-C <file>` | Combined user:pass list |
| `-s <port>` | Non-default port |
| `-V` | Verbose, show each attempt |
| `-f` | Stop on first valid credential found |
| `-t <n>` | Number of parallel tasks/threads |

#### 2 Common Service Modules

| Service | Example |
|---|---|
| SSH | `hydra -l root -P pass.txt ssh://<ip>` |
| FTP | `hydra -l admin -P pass.txt ftp://<ip>` |
| RDP | `hydra -l admin -P pass.txt rdp://<ip>` |
| SMB | `hydra -l admin -P pass.txt smb://<ip>` |
| MySQL | `hydra -l root -P pass.txt mysql://<ip>` |
| POP3/IMAP | `hydra -l user -P pass.txt pop3://<ip>` |

#### 3 HTTP Form Attacks

```
hydra -l admin -P pass.txt <ip> http-post-form \
  "/login:username=^USER^&password=^PASS^:Invalid login"
```

| Field | Purpose |
|---|---|
| `^USER^` | Placeholder for username list entry |
| `^PASS^` | Placeholder for password list entry |
| Final field | String that indicates a failed login attempt |

#### 4 Output and Restore

| Flag | Purpose |
|---|---|
| `-o out.txt` | Write found credentials to file |
| `-b <format>` | Output format (text/json/jsonv1) |
| `-I` | Ignore an existing restore file, start fresh |
| Restore file (`hydra.restore`) | Auto-created; resume with `hydra -R` |

#### 5 Timing and Evasion

| Flag | Purpose |
|---|---|
| `-t <n>` | Reduce thread count to avoid lockouts/detection |
| `-w <seconds>` | Wait time between connection attempts |
| `-W <seconds>` | Wait time for a response before timeout |

> CVE details change over time as new affected versions or exploitation techniques are discovered. Always check the current NVD entry rather than relying solely on a static table like this one.

### 27.11 MITRE ATT&CK Reference

MITRE ATT&CK is a publicly maintained knowledge base of adversary tactics and techniques, organized into a matrix. This section summarizes the Enterprise matrix's tactic categories and how they relate to material covered elsewhere in this book.

#### 2 Mapping Tactics to This Book's Chapters

| Tactic | Related Chapters |

#### 3 Using ATT&CK in Reporting

- Reference specific technique IDs (e.g., **T1110** – Brute Force, **T1059** – Command and Scripting Interpreter) in findings to standardize terminology with blue teams and clients.
- The **sub-technique** level (e.g., T1110.001 – Password Guessing) adds precision when documenting exactly how a technique was executed.
- MITRE also publishes **ATT&CK for Mobile** and **ATT&CK for ICS** matrices for those respective domains.

#### 4 Where to Verify Current Information

Tactic and technique IDs are occasionally revised as MITRE updates the framework. Check attack.mitre.org for the current version before citing an ID in a formal report.

### 27.12 OWASP Top 10

The OWASP Top 10 is a regularly updated, community-driven list of the most critical web application security risks. The list below reflects the most recent widely adopted revision (2021); always check owasp.org for the current published edition before citing it in a formal deliverable.

#### 1 The Ten Categories

| Rank | Category | Summary |
|---|---|---|
| A01 | Broken Access Control | Users acting outside their intended permissions |
| A02 | Cryptographic Failures | Weak or missing protection of sensitive data in transit/at rest |
| A03 | Injection | Untrusted data interpreted as commands or queries (SQLi, XSS, etc.) |
| A04 | Insecure Design | Missing or ineffective security controls at the design stage |
| A05 | Security Misconfiguration | Insecure default configs, verbose errors, unnecessary features enabled |
| A06 | Vulnerable and Outdated Components | Using libraries/frameworks with known vulnerabilities |
| A07 | Identification and Authentication Failures | Weak session/credential handling |
| A08 | Software and Data Integrity Failures | Unverified updates, plugins, or CI/CD pipeline trust issues |
| A09 | Security Logging and Monitoring Failures | Insufficient detection/response capability |
| A10 | Server-Side Request Forgery (SSRF) | Application fetches a remote resource without validating the destination |

#### 2 Relating Categories to Testing Activities

| Category | Typical Test Technique |
|---|---|
| Injection | SQLMap, manual payload testing, Burp Intruder |
| Broken Access Control | IDOR testing, role/permission matrix review |
| Authentication Failures | Hydra brute-force testing, session token analysis |
| Security Misconfiguration | Manual review, config scanners, default credential checks |
| Vulnerable Components | Dependency scanning (e.g., `npm audit`, `pip-audit`), version fingerprinting |
| SSRF | Manual parameter testing with internal/metadata URLs, Burp Collaborator |

#### 3 Notes

- The OWASP Top 10 is not exhaustive — it highlights the most impactful and common categories, not every possible flaw.
- OWASP also publishes specialized lists worth knowing: the **OWASP API Security Top 10**, **OWASP Mobile Top 10**, and the **OWASP Testing Guide**, which provides step-by-step methodology for each category above.

### 27.13 Glossary

Definitions of terms used throughout this book, in alphabetical order.

#### 1 A–F

| Term | Definition |
|---|---|
| Attack Surface | The sum of all points where an unauthorized user could attempt to enter or extract data |
| Brute Force | Attempting many credential combinations systematically until one succeeds |
| C2 (Command and Control) | Infrastructure an attacker uses to communicate with compromised systems |
| CVE | Common Vulnerabilities and Exposures — a standardized identifier for a publicly known vulnerability |
| CVSS | Common Vulnerability Scoring System — a numeric severity score for vulnerabilities |
| DMZ | Demilitarized Zone — a network segment exposed to untrusted networks, isolated from the internal LAN |
| Enumeration | The process of actively gathering detailed information about a target (users, shares, services) |
| Exploit | Code or a technique that takes advantage of a vulnerability |
| Fuzzing | Sending malformed or randomized input to a program to find crashes or unexpected behavior |

#### 2 G–M

| Term | Definition |
|---|---|
| Hash | A fixed-length output produced by a one-way function applied to input data |
| IDS/IPS | Intrusion Detection/Prevention System |
| Lateral Movement | Techniques used to move from one compromised host to others within a network |
| LOLBins | "Living off the land" binaries — legitimate OS tools abused for malicious purposes |
| Meterpreter | An in-memory Metasploit payload providing an interactive post-exploitation shell |

#### 3 N–S

| Term | Definition |
|---|---|
| NSE | Nmap Scripting Engine |
| OSINT | Open Source Intelligence — information gathered from publicly available sources |
| Payload | Code delivered and executed on a target as part of an exploit |
| Pivoting | Using a compromised host as a relay to reach otherwise unreachable network segments |
| Privilege Escalation | Gaining higher-level access than originally granted |
| Reconnaissance | Information-gathering phase preceding an attack |
| Shell | A command-line interface for interacting with a compromised or remote system |

#### 4 T–Z

| Term | Definition |
|---|---|
| TTPs | Tactics, Techniques, and Procedures — patterns of adversary behavior |
| Vulnerability | A weakness that can be exploited to violate a system's security policy |
| Wordlist | A file of candidate values (passwords, usernames, directories) used in brute-force or fuzzing attacks |
| Zero-Day | A vulnerability that is unknown to the vendor and has no available patch |
