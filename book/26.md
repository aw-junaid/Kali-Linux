## Chapter 26 – Real-World Labs

Chapter 26 moves the book out of theory and into practice. Every technique discussed in earlier chapters — scanning, web exploitation, Active Directory attacks, wireless auditing, password cracking, malware triage, and forensic investigation — is only useful once it can be executed correctly, under time pressure, against a real target, with a real toolset.

This chapter is built as a set of eight standalone labs. Each lab is scoped to a specific domain of offensive or defensive security work, and each lab is built entirely out of individual **tool files** — one markdown file per tool, written to the same exhaustive standard. There is no shortcut version of a tool's documentation here: if the tool has fifteen command-line flags, all fifteen are documented; if a workflow needs eight practical examples to actually make sense, it gets eight.

The intent is that a reader with only this chapter, a Kali Linux VM, and a legally authorized lab target could reproduce every command shown, understand every flag used, and walk away with working automation scripts they can reuse on their next engagement.

**Lab environment assumptions** used throughout this chapter, unless a tool file states otherwise:
- Attacker machine: Kali Linux (rolling release), fully updated (`sudo apt update && sudo apt full-upgrade -y`)
- Target machines: intentionally vulnerable lab VMs (e.g., Metasploitable2/3, DVWA, a small Active Directory lab domain, a WPA2 test AP owned by the reader) — never a system the reader does not own or have written authorization to test
- Shell: bash
- Privilege notation: commands prefixed with `sudo` require root; commands without it are run as a standard user

**How each lab is structured (26.1–26.8):**
- A short scenario/objective for the lab
- The attack or analysis chain, phase by phase
- A list of the tools used in that phase, each linked out to its own dedicated tool file
- A "lab walkthrough" narrative tying the individual tools together into one continuous exercise

### 26.1 Reconnaissance Lab

**About this lab:** Reconnaissance is where every engagement starts and where most of the real risk of scope violation lives — get it wrong here and everything downstream is built on a bad target list. This lab walks through building a complete external attack-surface picture of a domain using nothing but passive OSINT and light active scanning: who owns it, what subdomains and hosts exist, what's running on them, and who the people behind it are. The output of this lab — a validated list of live hosts, ports, and services — is the direct input to the Web Application Lab (26.2) and Active Directory Lab (26.3).

**Flow:** WHOIS/domain intel → email & subdomain harvesting → subdomain enumeration → active host/port/service scanning → consolidated target list.



#### Whois

**Introduction:** `whois` queries domain registrars and Regional Internet Registries (ARIN, RIPE, APNIC, LACNIC, AFRINIC) for registration data — owner, registrar, creation/expiry dates, name servers, and IP block ownership. It's completely passive: no packet reaches the target's own infrastructure, only the registrar's database. It's always the first command run in an engagement.

**Install:**
```bash
sudo apt update && sudo apt install -y whois
```

**Syntax:** `whois [OPTIONS] OBJECT` — object can be a domain, IP, or ASN (`AS15169`).

**Key options:**
| Flag | Purpose |
|||
| `-h HOST` | Query a specific WHOIS server |
| `-p PORT` | Non-standard port (default 43) |
| `-r` | Disable recursive referral lookups |
| `-H` | Suppress legal disclaimer text |

**Basic usage:**

```bash
whois example.com
```
```
Domain Name: EXAMPLE.COM
Registrar: RESERVED-Internet Assigned Numbers Authority
Creation Date: 1995-08-14T04:00:00Z
Registry Expiry Date: 2025-08-13T04:00:00Z
Name Server: A.IANA-SERVERS.NET
```

**Practical examples:**
```bash
# IP ownership check (confirm scope before scanning)
whois 8.8.8.8 | grep -iE "orgname|netrange"
```
```
NetRange:   8.8.8.0 - 8.8.8.255
Organization: Google LLC (GOGL)
```
```bash
# ASN lookup - find every netblock an org announces
whois AS15169 | grep -iE "asname|orgname"
```
```
ASName:  GOOGLE
OrgName: Google LLC
```
```bash
# Bulk lookup across a domain list
for d in example.com testdomain.com; do
  echo "== $d =="; whois "$d" | grep -iE "registrar:|expiry"
done
```

**Tips:** Registrant contact data is almost always GDPR-redacted now — don't rely on `whois` for emails, use theHarvester instead. Always confirm IP block ownership before scanning a shared-hosting IP.



#### theHarvester

**Introduction:** theHarvester aggregates emails, subdomains, employee names, and hosts from dozens of public sources — search engines, certificate transparency (crt.sh), Shodan, VirusTotal, LinkedIn — in a single Python tool. It's the go-to for building a phishing/password-spray target list and a first-pass subdomain list.

**Install:**
```bash
sudo apt update && sudo apt install -y theharvester
# Optional: add API keys for better coverage
sudo nano /etc/theHarvester/api-keys.yaml
```

**Syntax:** `theHarvester -d <domain> -b <source>`

**Key options:**
| Flag | Purpose |
|||
| `-b <source>` | Data source: `crtsh, bing, google, hackertarget, shodan, linkedin, all` |
| `-l <num>` | Limit results |
| `-r` | Resolve discovered hosts to IPs |
| `-c` | DNS brute-force |
| `-t` | Check subdomain takeover |
| `-f <file>` | Save HTML/XML report |

**Basic usage:**

```bash
theHarvester -d testdomain.com -b crtsh -f crtsh_report
```
```
[*] Hosts found: 27
www.testdomain.com
staging.testdomain.com
vpn.testdomain.com
[*] Saved HTML report to crtsh_report.html
```

**Practical examples:**
```bash
# Email harvesting via search engines
theHarvester -d testdomain.com -b bing,duckduckgo -l 200
```
```
[*] Emails found: 12
hr@testdomain.com
j.smith@testdomain.com
```
```bash
# Subdomain takeover check (dangling CNAME hunting)
theHarvester -d testdomain.com -b crtsh -t
```
```
[!] old-blog.testdomain.com -> CNAME points to unclaimed S3 bucket (POSSIBLE TAKEOVER)
```
```bash
# LinkedIn employee sweep for a social-engineering target list
theHarvester -d testcorp.com -b linkedin -l 100
```

**Tips:** Configure free API keys for `securityTrails`/`shodan`/`hunter` — coverage jumps significantly. Don't hammer `-b all` on a large domain; it gets CAPTCHA'd. Cross-check against Sublist3r for subdomain overlap.



#### Sublist3r

**Introduction:** Sublist3r is a fast, focused subdomain enumeration tool pulling from search engines, crt.sh, Netcraft, VirusTotal, and ThreatCrowd, with an optional brute-force pass. It trades exhaustiveness (which `amass` offers) for speed — ideal when you need a quick subdomain list to start scanning against.

**Install:**
```bash
sudo apt update && sudo apt install -y sublist3r
```

**Syntax:** `sublist3r -d <domain> [options]`

**Key options:**
| Flag | Purpose |
|||
| `-b` | Enable brute-force verification |
| `-t <n>` | Brute-force threads |
| `-p <ports>` | Port-scan discovered subdomains |
| `-e <engines>` | Restrict to specific search engines |
| `-o <file>` | Save output |

**Basic usage:**

```bash
sublist3r -d testdomain.com -o subs.txt
```
```
[-] Searching now in Google, Bing, Yahoo, Baidu, Netcraft, DNSdumpster, Virustotal, ThreatCrowd...
[-] Total Unique Subdomains Found: 14
[-] Saving results to file: subs.txt
```

**Practical examples:**
```bash
# Live-host filtering after enumeration
sublist3r -d testdomain.com -o subs.txt
while read -r sub; do
  nmap -p 80,443 --open -oG - "$sub" 2>/dev/null | grep -q open && echo "$sub is live"
done < subs.txt
```
```bash
# Port-scanning discovered subdomains directly
sublist3r -d testdomain.com -p 80,443,8080
```
```
www.testdomain.com:80 - Open
admin.testdomain.com:8080 - Open
```

**Tips:** Passive sources return historical DNS records — always live-check afterward. Combine with theHarvester and crt.sh manual checks for full coverage; no single tool finds everything.



#### Nmap

**Introduction:** Nmap is the industry-standard tool for host discovery, port scanning, service/version detection, OS fingerprinting, and scriptable vulnerability checks via the Nmap Scripting Engine (NSE). It's the bridge between passive recon above and active exploitation in Lab 26.2 — its output is the definitive target list every later lab consumes.

**Install:**
```bash
sudo apt update && sudo apt install -y nmap
nmap --version
```

**Syntax:** `nmap [scan types] [options] {targets}`

**Key options:**
| Flag | Purpose |
|||
| `-sS` | TCP SYN scan (fast, stealthy, needs root) |
| `-sV` | Service/version detection |
| `-O` | OS fingerprinting |
| `-A` | OS + version + scripts + traceroute |
| `-p-` | All 65535 ports |
| `-sC` | Default safe NSE scripts |
| `--script <name>` | Run specific NSE script(s) |
| `-T<0-5>` | Timing template (0=paranoid, 5=insane) |
| `-oA <base>` | Save in all output formats |
| `--open` | Show only open ports |
| `-Pn` | Skip host discovery, treat host as up |

**Basic usage:**
```bash
nmap 192.168.56.10
```
```
PORT     STATE SERVICE
21/tcp   open  ftp
22/tcp   open  ssh
80/tcp   open  http
445/tcp  open  microsoft-ds
Nmap done: 1 IP address (1 host up) scanned in 2.14 seconds
```

**Practical examples:**
```bash
# Full port sweep with version detection
nmap -sS -sV -p- -T4 --open 192.168.56.10
```
```
21/tcp   open  ftp      vsftpd 2.3.4
22/tcp   open  ssh      OpenSSH 4.7p1 Debian
80/tcp   open  http     Apache httpd 2.2.8
3306/tcp open  mysql    MySQL 5.0.51a-3ubuntu5
```
```bash
# Vulnerability script sweep against SMB
nmap --script smb-vuln* -p 445 192.168.56.10
```
```
| smb-vuln-ms08-067: 
|   VULNERABLE:
|_    State: VULNERABLE
```
```bash
# Scan a full target list, save all formats
nmap -iL targets.txt -sV -oA recon_scan
```

**Tips:** Run a fast `-sn` ping sweep or `--top-ports` scan first on a subnet before committing to a slow `-p-` full sweep. Always save with `-oA` — the XML feeds Metasploit's `db_import`. Never scan outside authorized scope.



**Lab wrap-up:** Merge the outputs of all four tools above into one deduplicated target list (`sort -u` across the subdomain and IP outputs), then run a final `nmap -sV -oA final_targets` pass across it. That file is the handoff artifact into 26.2 Web Application Lab.

### 26.2 Web Application Lab

**About this lab:** With a validated list of live web hosts from 26.1, this lab covers the standard web-app testing chain: content/endpoint discovery, automated vulnerability scanning, manual request interception and manipulation, and CMS-specific auditing. The goal is to find and confirm — not silently exploit — vulnerabilities such as injection flaws, exposed admin panels, misconfigurations, and outdated components, producing findings ready for a client report. All work here targets intentionally vulnerable lab apps (DVWA, bWAPP, Metasploitable2's web apps, or `testphp.vulnweb.com`-style legal test sites) — never production systems without written authorization.

**Flow:** content discovery (Gobuster) → automated scanning (Nikto) → CMS auditing (WPScan) → manual interception & testing (Burp Suite).



#### Gobuster

**Introduction:** Gobuster is a fast, Go-based brute-forcing tool for discovering hidden directories, files, subdomains, and virtual hosts on a web server, using a wordlist against the target. It's the standard first step in web recon — mapping the parts of an application that aren't linked from the visible UI.

**Install:**
```bash
sudo apt update && sudo apt install -y gobuster
```

**Syntax:** `gobuster <mode> [options]` — modes: `dir`, `dns`, `vhost`, `fuzz`, `s3`.

**Key options:**
| Flag | Purpose |
|||
| `-u <url>` | Target URL |
| `-w <wordlist>` | Wordlist file |
| `-x <ext>` | File extensions to append (`php,html,txt`) |
| `-t <n>` | Threads (default 10) |
| `-s <codes>` | Status codes to show |
| `-b <codes>` | Status codes to blacklist/hide |
| `-o <file>` | Output file |
| `-k` | Skip SSL certificate verification |
| `-r` | Follow redirects |

**Basic usage:**

```bash
gobuster dir -u http://192.168.56.10 -w /usr/share/wordlists/dirb/common.txt
```
```
===============================================================
/index.php            (Status: 200) [Size: 891]
/admin                (Status: 301) [Size: 315]
/uploads               (Status: 301) [Size: 315]
/config.php            (Status: 200) [Size: 0]
===============================================================
```

**Practical examples:**
```bash
# Directory brute-force with common extensions
gobuster dir -u http://192.168.56.10 -w /usr/share/wordlists/dirb/big.txt -x php,txt,bak -t 50
```
```bash
# Virtual host discovery
gobuster vhost -u http://testdomain.com -w /usr/share/wordlists/SecLists/Discovery/DNS/subdomains-top1million-5000.txt
```
```
Found: admin.testdomain.com (Status: 200)
```
```bash
# Filter out noisy 404s, only show real hits
gobuster dir -u http://192.168.56.10 -w common.txt -b 404,403 -o gobuster_results.txt
```

**Tips:** Match the wordlist size to time budget — `common.txt` for a quick pass, `big.txt`/SecLists for thorough coverage. Always add `-x php,asp,aspx,jsp` matching the detected backend tech from `whatweb`/`nmap -sV`.

#### SQLMap

**Introduction:** SQLMap is an automated tool for detecting and confirming SQL injection points and, where authorized, extracting database information through them. In this lab it's used purely as a **detection and confirmation** tool against a known-vulnerable lab target (e.g., DVWA's SQLi module) — not for building custom injection payloads. It supports all major DBMS engines and multiple injection techniques (boolean-blind, time-blind, UNION, error-based, stacked queries).

**Install:**
```bash
sudo apt update && sudo apt install -y sqlmap
```

**Syntax:** `sqlmap -u "<url>" [options]`

**Key options:**
| Flag | Purpose |
|||
| `-u <url>` | Target URL with parameter |
| `--data <postdata>` | Test POST body parameters |
| `--cookie <cookie>` | Send session cookie |
| `-p <param>` | Test only a specific parameter |
| `--dbs` | Enumerate databases (post-confirmation) |
| `--batch` | Non-interactive mode, accept defaults |
| `--risk <1-3>` / `--level <1-5>` | Control test aggressiveness/coverage |
| `--technique <BEUSTQ>` | Restrict to specific injection technique(s) |

**Basic usage (against DVWA lab instance, low security setting):**
```bash
sqlmap -u "http://192.168.56.10/dvwa/vulnerabilities/sqli/?id=1&Submit=Submit" \
  --cookie="PHPSESSID=abc123; security=low" --batch
```
```
[INFO] testing connection to the target URL
[INFO] GET parameter 'id' appears to be 'AND boolean-based blind' injectable
[INFO] the back-end DBMS is MySQL
sqlmap identified the following injection point(s):
Parameter: id (GET)
    Type: boolean-based blind
    Type: UNION query
```

**Practical examples:**
```bash
# Confirm and enumerate databases only (authorized lab target)
sqlmap -u "http://192.168.56.10/dvwa/vulnerabilities/sqli/?id=1" \
  --cookie="security=low" --batch --dbs
```
```
available databases [2]:
[*] dvwa
[*] information_schema
```
```bash
# Restrict to a single, less-invasive technique
sqlmap -u "http://192.168.56.10/dvwa/vulnerabilities/sqli/?id=1" \
  --cookie="security=low" --technique=B --batch
```

**Tips:** Always start with `--risk 1 --level 1` on unfamiliar targets before increasing aggressiveness. `--batch` is convenient but review prompts manually on real engagements to avoid unintended destructive tests. This tool confirms injection exists — actual data extraction from a client system requires explicit written authorization scoped to that action.



#### WPScan

| `--random-user-agent` | Rotate user-agent per request |

#### Burp Suite

**Introduction:** Burp Suite (Community/Professional) is the industry-standard interception proxy for manual web application testing. It sits between browser and server, letting the tester view, intercept, modify, and replay every HTTP(S) request — essential for testing business logic flaws, authentication bypasses, and anything automated scanners miss.

**Install:** Burp Suite Community Edition ships preinstalled on Kali; verify/launch:
```bash
burpsuite &
```

**Core workflow (not flag-driven — Burp is primarily GUI-driven):**
1. **Proxy setup** — Burp listens on `127.0.0.1:8080` by default; configure the browser (or use Burp's built-in Chromium) to use it as an HTTP/HTTPS proxy.
2. **Install CA certificate** — visit `http://burp` in the proxied browser, download `cacert.der`, import into the browser's trust store to intercept HTTPS without warnings.
3. **Proxy > Intercept** — toggle interception on/off; forward, drop, or edit requests in-flight.
4. **Repeater** — send an intercepted request here to manually resend it with edits, over and over, to test parameter tampering, IDOR, or injection by hand.
5. **Intruder** (Community: rate-limited) — automate payload injection across a request's parameters for fuzzing or brute-force.
6. **Target > Site map** — passively builds a map of every endpoint the proxy has seen.

**Practical examples:**
- **IDOR testing:** intercept a request like `GET /account?id=1042`, send to Repeater, change `id` to a neighboring value, resend, and check whether another user's data is returned.
- **Auth bypass testing:** intercept a login POST, drop the request after capturing it, replay it with a manipulated `role=user` parameter changed to `role=admin`.
- **Header/cookie tampering:** modify `X-Forwarded-For` or session cookies in Repeater to test access-control logic.
- **Automated fuzzing with Intruder:** mark the `id` parameter as a payload position, load a numeric range payload set (1–500), and run to find every accessible object ID.

**Tips:** Always scope Burp's Target > Scope to the authorized test domain(s) before running Intruder — it's easy to accidentally send automated traffic outside scope. Use Repeater, not Intruder, for anything requiring careful, low-volume manual verification. Save the project file regularly on long engagements.



**Lab wrap-up:** Chain the tools in order — Gobuster for content discovery, Nikto for automated triage, WPScan if a CMS is detected, then hand every interesting endpoint/parameter to Burp Suite for manual confirmation, and SQLMap for any parameter that looks injectable. Document confirmed findings with request/response evidence for the report.

### 26.3 Active Directory Lab

**About this lab:** Most real-world internal engagements land in a Windows Active Directory environment, and AD attack paths are rarely about a single exploit — they're about chaining small misconfigurations (weak ACLs, over-permissioned groups, stale trust relationships, crackable service accounts) into domain compromise. This lab covers the standard internal chain: unauthenticated enumeration, credential harvesting via network poisoning, authenticated enumeration, and attack-path mapping. All work assumes a purpose-built AD lab domain (e.g., a small "GOAD"-style lab or a single lab domain controller you own).

**Flow:** unauthenticated enumeration (Kerbrute) → network credential capture (Responder) → authenticated enumeration & lateral movement checks (NetExec/CrackMapExec, Impacket) → attack path graphing (BloodHound).



#### Kerbrute

**Introduction:** Kerbrute is a fast tool for enumerating valid Active Directory usernames and brute-forcing/spraying passwords via Kerberos pre-authentication, without generating the Windows Event ID 4625 (failed logon) that NTLM-based brute-forcing triggers — making it notably stealthier for username enumeration.

**Install:**
```bash
sudo apt update && sudo apt install -y kerbrute
# or from source
git clone https://github.com/ropnop/kerbrute.git && cd kerbrute && make all
```

**Syntax:** `kerbrute <mode> -d <domain> --dc <dc_ip> <wordlist> [options]` — modes: `userenum`, `passwordspray`, `bruteuser`, `bruteforce`.

**Key options:**
| Flag | Purpose |
|||
| `-d <domain>` | Target AD domain |
| `--dc <ip>` | Domain controller IP |
| `-o <file>` | Output file |
| `-t <n>` | Threads |
| `--safe` | Stop on account lockout detection |
| `-v` | Verbose |

**Basic usage:**
```bash
kerbrute userenum -d lab.local --dc 192.168.56.5 usernames.txt
```
```
2026/07/19 10:12:03 >  VALID USERNAME:       jsmith@lab.local
2026/07/19 10:12:03 >  VALID USERNAME:       admin@lab.local
2026/07/19 10:12:04 >  Done! Tested 500 usernames (2 valid) in 3.1 seconds
```

**Practical examples:**
```bash
# Username enumeration against the domain controller
kerbrute userenum -d lab.local --dc 192.168.56.5 /usr/share/wordlists/seclists/Usernames/xato-net-10-million-usernames.txt -o valid_users.txt
```
```bash
# Password spray using confirmed valid usernames (single password, low lockout risk)
kerbrute passwordspray -d lab.local --dc 192.168.56.5 valid_users.txt 'Summer2026!' --safe
```
```
2026/07/19 10:20:11 >  VALID LOGIN:  jsmith@lab.local:Summer2026!
```

**Tips:** Always use `--safe` in a real engagement to avoid triggering account lockout policies. Username enumeration via Kerberos pre-auth generally does not trigger failed-logon alerting the way NTLM does — confirm this assumption against the specific target's logging before relying on it.



#### Responder

**Introduction:** Responder is an LLMNR, NBT-NS, and mDNS poisoner that answers name-resolution broadcast requests on the local network, tricking Windows hosts into sending their NTLMv2 password hashes to the attacker's machine. It's one of the highest-yield tools in internal AD assessments because LLMNR/NBT-NS are enabled by default on most Windows networks.

**Install:** Preinstalled on Kali; verify:
```bash
responder --version
```

**Syntax:** `responder -I <interface> [options]`

**Key options:**
| Flag | Purpose |
|||
| `-I <iface>` | Network interface to listen on |
| `-w` | Start WPAD rogue proxy server |
| `-r` | Enable NetBIOS answers for workstation service |
| `-f` | Fingerprint hosts that send requests |
| `-v` | Verbose output |
| `-A` | Analyze mode only, don't poison |

**Basic usage:**
```bash
sudo responder -I eth0
```
```
[+] Listening for events...
[LLMNR] Poisoned answer sent to 192.168.56.22 for name FILESHARE
[SMB] NTLMv2-SSP Hash captured:
jdoe::LAB:1122334455667788:AABBCCDD...:0101000000000000...
```

**Practical examples:**
```bash
# Passive analysis first to confirm LLMNR/NBT-NS traffic exists before poisoning
sudo responder -I eth0 -A
```
```bash
# Full poisoning run, hashes logged automatically to /usr/share/responder/logs/
sudo responder -I eth0 -wrf
```
```
[+] Captured NTLMv2 hash for jdoe -> saved to SMB-NTLMv2-SSP-192.168.56.22.txt
```
```bash
# Feed captured hashes directly into Hashcat for offline cracking (Lab 26.5)
hashcat -m 5600 SMB-NTLMv2-SSP-192.168.56.22.txt /usr/share/wordlists/rockyou.txt
```

**Tips:** Responder is loud and network-wide — only run it in a scoped, authorized internal engagement window, ideally coordinated with the client's blue team. #### NetExec (CrackMapExec successor)

### 26.4 Wireless Lab

**About this lab:** Wireless auditing has a hard legal boundary that the other labs don't: radio signals cross property lines, so every command here must run against an access point the reader owns or has explicit written authorization to test — never a neighbor's or public network. This lab covers the standard 802.11 audit chain: putting the wireless adapter into monitor mode, discovering nearby networks, capturing a WPA/WPA2 handshake, and testing password strength against it offline.

**Flow:** monitor mode setup (aircrack-ng suite) → network discovery → handshake/PMKID capture → automated end-to-end audit (Wifite) → offline password testing (handed off to Lab 26.5).

### 26.5 Password Cracking Lab

**About this lab:** Every earlier lab produces credential material of one form or another — NTLMv2 hashes from Responder, Kerberoast tickets from Impacket, WPA handshakes from the Wireless Lab, or password hashes pulled from a compromised database. This lab covers turning that captured material into plaintext passwords offline, and testing credentials online where authorized. All targets here are the reader's own hashes/lab accounts — cracking material obtained without authorization is illegal regardless of technique.

**Flow:** wordlist/rule preparation (CeWL) → offline hash cracking (Hashcat, John the Ripper) → authorized online credential testing (Hydra).



#### Hashcat

**Introduction:** Hashcat is the fastest general-purpose password recovery tool available, using GPU acceleration to test billions of candidate passwords per second against a captured hash. It supports hundreds of hash types (identified by a `-m` mode number) and multiple attack modes: straight dictionary, combinator, brute-force mask, and rule-based mutation.

**Install:**
```bash
sudo apt update && sudo apt install -y hashcat
hashcat -I   # list detected GPU/CPU compute devices
```

**Syntax:** `hashcat -m <mode> -a <attack_mode> <hashfile> <wordlist/mask> [options]`

**Key options:**
| Flag | Purpose |
|||
| `-m <n>` | Hash type (0=MD5, 1000=NTLM, 1800=sha512crypt, 5600=NetNTLMv2, 13100=Kerberoast, 22000=WPA/WPA2) |
| `-a <n>` | Attack mode (0=dictionary, 1=combinator, 3=brute-force/mask, 6/7=hybrid) |
| `-r <rule file>` | Apply a rule file to mutate wordlist entries |
| `-o <file>` | Output cracked results |
| `--show` | Show already-cracked hashes from potfile |
| `-1/-2/-3/-4` | Define custom charsets for mask attacks |
| `--force` | Bypass warnings (use cautiously) |
| `-w <1-4>` | Workload profile (performance tuning) |

**Basic usage:**
```bash
hashcat -m 0 -a 0 md5_hashes.txt /usr/share/wordlists/rockyou.txt
```
```
Session..........: hashcat
Status...........: Cracked
5f4dcc3b5aa765d61d8327deb882cf99:password
Recovered........: 1/1 (100.00%) Digests
```

**Practical examples:**
```bash
# NTLM hashes dumped from secretsdump.py
hashcat -m 1000 -a 0 ntlm_hashes.txt /usr/share/wordlists/rockyou.txt
```
```bash
# NetNTLMv2 hashes captured via Responder
hashcat -m 5600 responder_hashes.txt /usr/share/wordlists/rockyou.txt
```
```bash
# Kerberoast TGS hashes from GetUserSPNs.py
hashcat -m 13100 kerberoast_hashes.txt /usr/share/wordlists/rockyou.txt
```
```
$krb5tgs$23$*svc_sql$LAB.LOCAL*:Summer2026!
```
```bash
# Dictionary + rule-based mutation (catches "Password1!" style variants)
hashcat -m 1000 -a 0 ntlm_hashes.txt /usr/share/wordlists/rockyou.txt -r /usr/share/hashcat/rules/best64.rule
```
```bash
# Mask attack for an 8-character all-numeric PIN
hashcat -m 0 -a 3 pin_hashes.txt ?d?d?d?d?d?d?d?d
```
```bash
# WPA2 handshake cracking (converted .cap to .hc22000 first)
hashcat -m 22000 labnet.hc22000 /usr/share/wordlists/rockyou.txt
```
```bash
# Review already-cracked results from the potfile without re-running
hashcat -m 1000 ntlm_hashes.txt --show
```

**Tips:** Always try a fast dictionary + rules pass before a slow brute-force mask attack — most real-world passwords fall to `rockyou.txt` + `best64.rule`. Use `hashcat -I` to confirm GPU acceleration is active; CPU-only cracking is dramatically slower for modes like NTLM. Convert `.cap` files to Hashcat's `.hc22000` format with `hcxpcapngtool` before mode `22000`.



#### John the Ripper

**Introduction:** John the Ripper (JtR) is the long-standing classic password cracker, notable for its automatic hash-format detection, flexible rule engine, and strong support for cracking Unix `/etc/shadow`, ZIP/RAR archive passwords, and SSH private key passphrases — cases Hashcat handles less conveniently. Kali ships the "jumbo" community edition with the widest format support.

**Install:**

**Key options:**
| Flag | Purpose |
|||

| `--rules` | Apply mutation rules to wordlist |
| `--format=<type>` | Force a specific hash format |
| `--show` | Show cracked passwords from `john.pot` |
| `--incremental` | Brute-force mode |

**Basic usage:**

letmein!         (asmith)

```

**Practical examples:**
```bash

```
```bash

#### Hydra

**Introduction:** Hydra (THC-Hydra) is a fast, parallelized online login brute-forcer supporting dozens of protocols — SSH, FTP, HTTP(S) forms, SMB, RDP, MySQL, and more. Unlike Hashcat/John (offline, against a stolen hash), Hydra attacks a live authentication service directly, making it far noisier and riskier to run without explicit authorization and a defined testing window (account lockouts, alerting).

| `-o <file>` | Output file |
| `-e nsr` | Also try null password, same-as-login, reversed login |

**Basic usage:**
```bash
hydra -l admin -P /usr/share/wordlists/rockyou.txt ssh://192.168.56.10
```

```

**Practical examples:**
```bash

Prefer offline cracking (Hashcat/John) wherever a hash can be obtained instead of hammering a live service.

```bash

### 26.6 Malware Analysis Lab

**About this lab:** Malware analysis splits into static (examining a sample without running it) and dynamic (observing behavior in a controlled, isolated environment) analysis. This lab covers the standard triage chain used to characterize an unknown binary safely: initial static triage, string/signature extraction, and disassembly/decompilation. All work here happens against known, publicly available malware samples (e.g., from a controlled corpus like theZoo, run only inside an isolated, network-segmented, snapshot-able VM) — never on a live production network, and never in a way that risks the sample executing outside the sandbox.

**Flow:** file identification and string extraction → signature-based classification (YARA) → static disassembly/decompilation (radare2, Ghidra).

**Safety note:** every command below assumes an air-gapped or host-only-networked analysis VM with reverting snapshots. Malware samples should never be executed on a system connected to a production network.



#### file & strings

**Introduction:** `file` and `strings` are the first two commands run against any unknown binary — `file` identifies the file type/format from its header (not its extension, which can be spoofed), and `strings` extracts printable text sequences, often revealing embedded URLs, IP addresses, file paths, registry keys, or debug strings that hint at functionality before any disassembly is needed.

**Install:** both are part of standard Linux base packages (`binutils`), preinstalled on Kali.

**Syntax:**
```
file <filename>
strings [options] <filename>
```

**Key `strings` options:**
| Flag | Purpose |
|||
| `-a` | Scan entire file, not just data sections |
| `-n <len>` | Minimum string length to report (default 4) |
| `-t x` | Show byte offset in hex |
| `-e l/b` | Extract 16-bit little/big-endian (Unicode) strings |

**Basic usage:**
```bash
file sample.exe
```
```
sample.exe: PE32 executable (GUI) Intel 80386, for MS Windows
```

**Practical examples:**
```bash
# Extract meaningful strings only, minimum length 8
strings -n 8 sample.exe | head -20
```
```
GetProcAddress
LoadLibraryA
http://malicious-c2[.]example/gate.php
CreateRemoteThread
VirtualAllocEx
```
```bash
# Show file offset alongside each string (helps cross-reference in a disassembler)
strings -t x sample.exe | grep -i "http"
```
```
  4a1c http://malicious-c2[.]example/gate.php
```
```bash
# Extract wide-character (Unicode) strings often used in Windows malware
strings -e l sample.exe | head
```
```bash
# Grep for common suspicious API names across the binary
strings sample.exe | grep -E "VirtualAlloc|CreateRemoteThread|WriteProcessMemory|WSASocket"
```

**Tips:** A binary with almost no readable strings is a strong indicator of packing/obfuscation — pivot to entropy analysis or unpacking before further static analysis. Suspicious API name clusters (`VirtualAllocEx` + `WriteProcessMemory` + `CreateRemoteThread`) are a classic process-injection signature worth flagging immediately.



#### YARA

**Introduction:** YARA is a pattern-matching engine purpose-built for malware classification — write a "rule" describing strings, byte sequences, or conditions characteristic of a malware family, and YARA scans files (or running memory) to flag matches. It's the standard tool for both identifying known malware families and building custom detection signatures during an investigation.

**Install:**
```bash
sudo apt update && sudo apt install -y yara
```

**Syntax:** `yara [options] <rules_file> <target>`

**Key options:**
| Flag | Purpose |
|||
| `-r` | Recursively scan a directory |
| `-s` | Print matching strings, not just rule names |
| `-c` | Print number of matches (count mode) |
| `-t <tag>` | Only evaluate rules with a specific tag |
| `-f` | Fast matching mode |
| `-w` | Suppress warnings |

**A basic YARA rule (`suspicious_pe.yar`):**
```
rule Suspicious_Process_Injection
{
    meta:
        description = "Flags PE binaries referencing classic injection APIs"
        author = "analysis lab"
    strings:
        $api1 = "VirtualAllocEx"
        $api2 = "WriteProcessMemory"
        $api3 = "CreateRemoteThread"
        $mz = { 4D 5A }  // MZ header
    condition:
        $mz at 0 and 2 of ($api*)
}
```

**Basic usage:**
```bash
yara suspicious_pe.yar sample.exe
```
```
Suspicious_Process_Injection sample.exe
```

**Practical examples:**
```bash
# Scan an entire directory of samples recursively, show matched strings
yara -r -s suspicious_pe.yar ./samples/
```
```
Suspicious_Process_Injection ./samples/sample1.exe
0x4a1c:$api1: VirtualAllocEx
0x4a3f:$api2: WriteProcessMemory
```
```bash
# Use a public rule set (e.g., YARA-Rules project) against a sample
git clone https://github.com/Yara-Rules/rules.git
yara -r rules/malware/ sample.exe
```
```
Trojan_Generic_Malware sample.exe
```
```bash
# Count-only mode across many files
yara -c -r suspicious_pe.yar ./samples/
```
```
./samples/sample1.exe: 1
./samples/sample2.exe: 0
```

**Tips:** Combine string-based conditions with structural ones (like the `$mz at 0` header check above) to cut false positives dramatically. Public rule repositories (YARA-Rules, ReversingLabs) are a fast way to check a sample against thousands of known-family signatures before writing custom rules.



#### radare2

**Introduction:** radare2 (r2) is a free, scriptable command-line reverse engineering framework — a disassembler, debugger, and binary analysis toolkit in one. It's the open-source alternative to commercial tools for examining a binary's assembly, control flow, imported functions, and (with a debugger attached) runtime behavior.

**Install:**
```bash
sudo apt update && sudo apt install -y radare2
```

**Syntax:** `r2 [options] <file>` — then interactive commands inside the r2 shell.

**Key startup flags:**
| Flag | Purpose |
|||
| `-A` | Run full auto-analysis on load |
| `-d` | Debug mode (run and attach to the binary) |
| `-w` | Open in write mode |
| `-q` | Quit after running commands (batch mode) |

**Key in-shell commands:**
| Command | Purpose |
|||
| `aaa` | Analyze all (functions, xrefs, strings) |
| `afl` | List all identified functions |
| `pdf @ <func>` | Disassemble a specific function |
| `iz` | List strings in the binary |
| `ii` | List imported functions |
| `axt <addr>` | Find cross-references to an address |
| `VV` | Enter visual graph mode (control flow graph) |

**Basic usage:**
```bash
r2 -A sample.exe
```
```
[0x004010a0]> afl
0x004010a0    12 128  main
0x00401200     4  48  sub_401200
```

**Practical examples:**
```bash
# Auto-analyze then disassemble main
r2 -A -q -c "afl; pdf @ main" sample.exe
```
```
0x004010a0      push ebp
0x004010a1      mov  ebp, esp
0x004010a3      call sub_401200
```
```bash
# List imported Windows API functions (quick capability triage)
r2 -q -c "ii" sample.exe
```
```
[Imports]
nth vaddr      bind type name
1   0x00402010 GLOBAL FUNC kernel32.dll_VirtualAllocEx
2   0x00402014 GLOBAL FUNC kernel32.dll_CreateRemoteThread
```
```bash
# Find cross-references to a suspicious string (e.g., a hardcoded C2 URL address)

```
```bash

[0x004010a0]> s main
[0x004010a0]> VV
```

**Tips:** Always run `aaa` (or start with `-A`) before browsing functions — without analysis, r2 only sees raw bytes, not identified functions/strings. `ii` (imports) is often the fastest single command to get a capability overview of an unfamiliar sample.



#### Ghidra

**Introduction:** Ghidra is the NSA's free, open-source software reverse engineering suite, offering a full GUI disassembler and — its standout feature — a powerful decompiler that reconstructs C-like pseudocode from raw assembly, making complex malware logic dramatically easier to read than a pure disassembly listing.

**Install:**
```bash
sudo apt update && sudo apt install -y ghidra
```

**Core workflow (GUI-driven):**
1. **Launch:** `ghidra` (or `ghidraRun` from the install directory)
2. **New Project** → `File > New Project` → choose Non-Shared (standalone lab use)
3. **Import Binary** → `File > Import File`, select the sample; Ghidra auto-detects format/architecture
4. **Analyze** → double-click the imported file, accept "Analyze now?" with default analyzers enabled (this runs auto-analysis equivalent to r2's `aaa`)
5. **Navigate:** Symbol Tree (left) lists functions; double-click any function to view both the assembly listing and the **Decompile** pane (right) showing reconstructed C pseudocode
6. **Cross-references:** right-click any variable/call → `References > Show References To` to trace data/control flow

**Headless batch analysis (for automating analysis across many samples without the GUI):**
```bash
$GHIDRA_HOME/support/analyzeHeadless /home/claude/ghidra_projects LabProject \
  -import sample.exe -postScript ExtractFunctions.java -deleteProject
```
```
INFO  ANALYZING sample.exe
INFO  Auto-Analysis complete
```

**Practical examples:**
- **Reading decompiled main logic:** after import + analyze, double-click `main` in the Symbol Tree; the Decompile pane shows something like:
  ```c
  int main(void) {
      HANDLE hMem = VirtualAllocEx(hProc, NULL, 0x1000, ...);
      WriteProcessMemory(hProc, hMem, shellcode, 0x1000, NULL);
      CreateRemoteThread(hProc, NULL, 0, hMem, NULL, 0, NULL);
      return 0;
  }
  ```
  — immediately readable as classic process injection, versus dozens of raw assembly lines.
- **Renaming/annotating:** rename recovered variables/functions as their purpose becomes clear (`L` key to rename a symbol) — builds a readable analysis trail for the report.
- **String-to-function tracing:** use the Defined Strings window (`Window > Defined Strings`) to jump from a suspicious string straight to the function referencing it.
- **Headless scripting** for bulk triage: write a Java/Python Ghidra script to auto-extract function names and suspicious API call lists across an entire sample corpus without opening the GUI per file.

**Tips:** Let full auto-analysis complete before navigating — interrupting it produces incomplete decompilation. The Decompile pane is the single highest-value feature for a first pass; read it before the raw assembly listing. For batch/CI-style analysis of many samples, `analyzeHeadless` with a custom post-script is far faster than manual GUI review.



**Lab wrap-up:** Start every unknown sample with `file` + `strings` for a two-minute triage, run it against a YARA rule set for family/technique classification, then move to radare2 for a quick command-line capability check (imports, strings, control flow) and Ghidra when full decompiled logic is needed for the report. Never execute the sample outside an isolated, snapshotted analysis VM.

### 26.7 Digital Forensics Lab

**About this lab:** Digital forensics is the investigative counterpart to the offensive labs — instead of breaking in, this lab covers reconstructing what happened after the fact: examining a disk image, memory dump, or file for evidence of compromise while preserving evidentiary integrity (hashing, write-blocking, documented chain of custody). This lab covers disk/file-system investigation, memory forensics, and metadata/embedded-file extraction, using publicly available forensic practice images (e.g., from DFIR training sets) as targets.

**Flow:** image acquisition & integrity verification → disk/file-system triage (Autopsy) → memory analysis (Volatility3) → file metadata & embedded-content extraction (ExifTool, binwalk).



#### Autopsy

**Introduction:** Autopsy is the graphical front-end to The Sleuth Kit, the standard open-source digital forensics platform for examining disk images and file systems. It handles file-system parsing, deleted-file recovery, timeline generation, keyword search, and hash-based known-file filtering, and is used widely by law enforcement and incident responders alike.

**Install:**
```bash

```

**Core workflow (GUI-driven):**
1. **Verify image integrity first**, before opening in Autopsy:
   ```bash

   ```
2. **Launch:** `autopsy` (starts a local web-based UI, typically `http://localhost:9999`)
3. **Create New Case** → enter case name, examiner details (chain-of-custody metadata)
4. **Add Data Source** → point to the disk image (`.dd`, `.E01`, etc.) — Autopsy auto-detects the format
5. **Run Ingest Modules** → enable Hash Lookup, File Type Identification, Keyword Search, Recent Activity, EXIF Parser, Timeline
6. **Review results** across the tabs: File Types, Deleted Files, Keyword Hits, Timeline, Web Activity (browser history/cache reconstruction)

#### binwalk

**Introduction:** binwalk scans a binary blob (firmware images, disk images, unknown files) for embedded files and executable code by signature, then extracts them — the standard tool for firmware forensics and for pulling hidden/appended files out of a container (e.g., a JPEG with a ZIP archive appended, or router firmware with an embedded file system).

**Practical examples:**
```bash

```

**Lab wrap-up:** Verify image/file integrity with hashing before touching any evidence. ### 26.8 Full Penetration Test Walkthrough

**About this lab:** This closing lab doesn't introduce new tools — it stitches Labs 26.1 through 26.7 into one continuous, methodology-driven engagement against a single authorized lab environment, from signed rules of engagement through to a delivered report.
