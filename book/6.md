## Chapter 6 – Directory & Content Discovery


Directory and content discovery is one of the foundational activities in web application reconnaissance and penetration testing. Web servers frequently host files, directories, backup copies, configuration files, administrative panels, and API endpoints that are not linked anywhere in the visible site navigation. These "hidden" resources are often forgotten by developers, left over from testing, or intentionally excluded from public menus — yet they remain accessible to anyone who requests the correct URL. Content discovery tools automate the process of guessing these paths by sending a very large number of HTTP requests, each built by combining a base URL with entries from a wordlist, and then analyzing the server's responses (status codes, response length, response time, and redirects) to determine which guesses correspond to real resources.

This chapter covers five of the most widely used content discovery utilities available in Kali Linux and other penetration testing distributions: **Gobuster**, **FFUF**, **Feroxbuster**, **Dirsearch**, and **WFuzz**. While all five tools share the same underlying goal — brute-forcing URLs against a wordlist — they differ in implementation language, performance characteristics, output formats, recursion behavior, and the breadth of protocols or discovery modes they support (directories, DNS subdomains, virtual hosts, S3 buckets, GET/POST parameters, and more). Understanding the strengths, syntax, and quirks of each tool allows a security professional to choose the right instrument for a given engagement, combine tools effectively, and interpret results with confidence.

The tools discussed in this chapter are intended strictly for authorized security testing, bug bounty programs, and educational lab environments (such as HackTheBox, TryHackMe, or your own local test servers). Running these tools against systems you do not own or do not have explicit written permission to test is illegal in most jurisdictions.

### 6.1 Gobuster

#### 1. Tool Introduction

Gobuster is a directory, file, DNS subdomain, virtual host, S3 bucket, and TFTP brute-forcing tool written in Go by OJ Reeves. Because it is written in Go, Gobuster compiles to a single static binary and takes advantage of Go's lightweight goroutine concurrency model, which makes it extremely fast compared to interpreted-language tools when sending large numbers of HTTP or DNS requests. Gobuster does not spider a site or parse HTML for links — it purely performs wordlist-based brute forcing, sending one request per candidate word (or per DNS query) and reporting back which candidates produced a "hit" based on status code or DNS resolution.

Gobuster is organized around **modes**, each invoked as a subcommand:

- `dir` – classic directory and file brute-forcing over HTTP/HTTPS.
- `dns` – subdomain brute-forcing via DNS resolution.
- `fuzz` – generic fuzzing using a `FUZZ` keyword, similar in spirit to FFUF.
- `vhost` – virtual host brute-forcing via the `Host` header.
- `s3` – Amazon S3 bucket enumeration.
- `gcs` – Google Cloud Storage bucket enumeration.
- `tftp` – TFTP file enumeration.

This modular design means Gobuster is really several specialized brute-forcing tools bundled behind one binary, which keeps the option set for each mode focused and easy to remember.

#### 2. How to Install

Gobuster ships as a package in Kali Linux's default repositories, and is also distributable as a single Go binary or via `go install`.


#### 8. Automation with Bash

A simple wrapper script that scans a list of targets sequentially and stores results per host:

```bash
#!/usr/bin/env bash
# gobuster_batch.sh - run gobuster dir scans against a list of hosts

WORDLIST="/usr/share/wordlists/dirb/common.txt"
EXTENSIONS="php,html,txt,bak"
OUTDIR="./gobuster_results"
THREADS=40

mkdir -p "$OUTDIR"

while IFS= read -r target; do
    [ -z "$target" ] && continue
    safe_name=$(echo "$target" | sed 's/[^a-zA-Z0-9]/_/g')
    echo "[*] Scanning $target"
    gobuster dir -u "$target" -w "$WORDLIST" -x "$EXTENSIONS" \
        -t "$THREADS" -q -o "$OUTDIR/${safe_name}.txt"
done < targets.txt

echo "[*] All scans complete. Results in $OUTDIR"
```

Usage: populate `targets.txt` with one URL per line, then run `bash gobuster_batch.sh`.

#### 9. Tips and Best Practices

- Always start with a smaller wordlist (e.g., `common.txt`) before escalating to larger lists like `raft-large-words.txt` to save time.
- Use `-t` thoughtfully; too many threads can crash a fragile target or trigger WAF/IDS rate-limiting.
- Use `--exclude-length` or `-b` to filter out repetitive "soft 404" responses that return HTTP 200 for nonexistent paths.
- Combine `-x` with relevant extensions based on the technology stack identified (e.g., `aspx` for IIS/.NET, `jsp` for Java).
- Use `-o` to always keep a persistent log of results for later comparison and reporting.
- Respect scope and rate limits; add `--delay` when testing against production or rate-limited systems.
- Use `-k` only for lab/test environments with self-signed certificates, not against systems where certificate validation matters.

#### 10. Troubleshooting

| Problem | Cause / Fix |
|---|---|
| All requests return the same status/size ("wildcard" response) | Target uses a catch-all page for unknown paths; use `--exclude-length` or `-b` to filter it out |
| `dns` mode reports everything as found | Wildcard DNS is enabled on the domain; use `--wildcard` to acknowledge and let Gobuster attempt detection |
| Connection refused / timeouts | Check target is reachable, correct protocol (http vs https), and firewall/WAF isn't blocking your IP |
| TLS certificate errors | Add `-k` to skip validation for self-signed certs in lab environments |
| Extremely slow scan | Increase `-t` threads, reduce wordlist size, or check network latency |
| No results at all | Verify wordlist path is correct and not empty; test URL manually with `curl` first |

### 6.2 FFUF

#### 1. Tool Introduction

FFUF, short for **"Fuzz Faster U Fool,"** is an open-source web fuzzer written in Go by Joona Hoikkala (joohoi). Unlike Gobuster, which separates functionality into fixed modes, FFUF is built around a single generic concept: the user places the literal keyword `FUZZ` anywhere in the URL, headers, POST body, or even the Host header, and FFUF substitutes wordlist entries into that position on every request. This design makes FFUF exceptionally flexible — the same binary can perform directory brute-forcing, virtual host discovery, parameter fuzzing, value fuzzing, and even multi-position fuzzing (fuzzing two variables simultaneously with two separate wordlists).

FFUF is prized in the penetration testing and bug bounty community for its speed (thanks to Go's concurrency model), its powerful response filtering/matching engine (by status code, size, word count, line count, or regex), and its ability to output results in several structured formats (JSON, CSV, HTML, Markdown) suitable for reporting or chaining into other tools.

#### 4. Command-Line Options

**Input options:**

| Flag | Description |
|---|---|
| `-w string` | Wordlist path, optionally with a keyword `-w list.txt:KEYWORD` |
| `-u string` | Target URL, must contain the fuzz keyword |

| `-recursion-depth int` | Maximum recursion depth (0 = unlimited) |

|---|---|

| Flag | Description |
|---|---|

| `-fr string` | Filter by regex |
| `-fs int` | Filter by response size |
| `-fw int` | Filter by word count |
| `-ft duration` | Filter by response time |

**General options:**

| Flag | Description |
|---|---|
| `-t int` | Number of concurrent threads (default 40) |

| `-rate int` | Requests per second limit |
| `-timeout int` | HTTP request timeout in seconds (default 10) |
| `-maxtime int` | Max total execution time in seconds |
| `-maxtime-job int` | Max time per fuzzing job (used with recursion) |
| `-o string` | Output file |

#### 8. Automation with Bash

```bash
#!/usr/bin/env bash
# ffuf_recon.sh - directory + vhost recon using ffuf

TARGET="$1"          # e.g. http://target.local
WORDLIST="/usr/share/wordlists/dirb/common.txt"
VHOSTLIST="vhosts.txt"
OUT="ffuf_out_$(date +%Y%m%d_%H%M%S)"

mkdir -p "$OUT"

echo "[*] Directory fuzzing $TARGET"
ffuf -w "$WORDLIST" -u "${TARGET}/FUZZ" -mc 200,301,302,403 \
     -o "$OUT/dirs.json" -of json -s

echo "[*] Vhost fuzzing on $TARGET"
ffuf -w "$VHOSTLIST" -u "$TARGET" -H "Host: FUZZ.$(echo $TARGET | sed 's~http://~~;s~https://~~')" \
     -fs 0 -o "$OUT/vhosts.json" -of json -s

echo "[*] Done. Results saved in $OUT/"
```

#### 9. Tips and Best Practices

- Always run a baseline request first (`curl -I <url>/doesnotexist123`) to identify soft-404 response characteristics before scanning, then pre-set `-fs`/`-fw`/`-fl` accordingly.
- Prefer `-ac` (auto-calibration) on unfamiliar targets to reduce false positives automatically.
- Use `-recursion` carefully — it can massively multiply request volume; pair it with `-recursion-depth` to bound scope.
- Use `-of json` or `-of csv` when you plan to feed results into another script or reporting pipeline.
- Use `-p` or `-rate` to throttle requests against production systems or when a WAF is present.
- Named keyword wordlists (`-w list.txt:KEYWORD`) make multi-position fuzzing commands far more readable than relying on positional `FUZZ`, `FUZZ2`, etc.

#### 10. Troubleshooting

| Problem | Cause / Fix |
|---|---|
| Every word "matches" | Soft-404 page returning HTTP 200; add `-fs`/`-fw` filters or use `-ac` |
| `FUZZ keyword not found` error | Ensure the literal string `FUZZ` appears in `-u`, `-d`, `-H`, or `-b` |
| Too slow | Increase `-t` threads (default 40 is often conservative); check network latency |
| Getting blocked / banned mid-scan | Lower `-rate`, add `-p` delay, rotate `-x` proxy, or add realistic headers |
| TLS handshake errors | Add `-k` to skip certificate verification for lab targets |
| Recursion produces excessive noise | Lower `-recursion-depth`, add stricter matchers, or disable recursion and drill down manually |

### 6.3 Feroxbuster

#### 1. Tool Introduction

Feroxbuster ("ferox" is Latin for "fierce," and it plays on "force" + "buster") is a fast, recursive content discovery tool written in Rust by Ben "epi" Risher. Its defining feature relative to Gobuster and Dirsearch is **automatic recursion**: whenever Feroxbuster discovers a directory during a scan, it automatically queues a new scan against that directory without requiring the user to manually re-run the tool. This makes it particularly effective for mapping deeply nested site structures in a single command. Feroxbuster is built on Rust's async runtime (Tokio), giving it very high throughput while maintaining low memory usage, and it includes a polished terminal UI showing live scan statistics, per-directory progress bars, and a running results table.

Feroxbuster also includes many quality-of-life features acquired over its development: automatic filtering of "wildcard" responses, extraction of links/paths from HTML and JavaScript responses to seed further scanning, support for scanning multiple targets from a file, and various output formats.

### 6.4 Dirsearch

#### 1. Tool Introduction

Dirsearch is a Python-based command-line web path scanner developed by Mauro Soria (maurosoria) and maintained by an active open-source community. ### 6.5 WFuzz

#### 1. Tool Introduction

WFuzz is a Python-based web application fuzzing framework created by Xavier Mendez (Xavi/Christian Martorella lineage project, now community-maintained). It predates many of the modern Go-based fuzzers and remains popular for its maturity and extreme flexibility: like FFUF, WFuzz uses a placeholder keyword — `FUZZ` — that can be substituted anywhere in a request (URL path, headers, POST data, cookies, even multiple positions with `FUZZ`, `FUZ2Z`, etc.). WFuzz's particular strength lies in its rich plugin/encoder ecosystem, its ability to iterate over multiple wordlists with different combination strategies (e.g., "product" vs "chain" iteration), and its baked-in payload generators (not just flat wordlists, but ranges, permutations, and external command output).

WFuzz is commonly used for directory/file brute-forcing, but it is equally at home fuzzing GET/POST parameters, HTTP headers (including for header-injection or host-header attacks), cookies, and even fuzzing multiple positions simultaneously to test for parameter-pollution or IDOR-style vulnerabilities.

**Kali Linux (APT — pre-installed on many Kali images):**


**Verify installation:**

#### 3. Syntax


The `-z` payload flag is central to WFuzz: it defines both a payload **type** (e.g., `file`, `range`, `list`) and its **source** (e.g., a wordlist path).

#### 4. Command-Line Options

**Payload / iteration:**

| Flag | Description |

| `-z <type>,<source>` | Define a payload; type can be `file`, `list`, `range`, `command`, etc. |
| `-w <FILE>` | Shortcut equivalent to `-z file,<FILE>` |
| `--zP <params>` | Extra parameters for the payload encoder/generator |
| `-c` | Colorize the console output |
| `--iterator=<method>` | Iteration strategy across multiple payload positions: `product`, `zip`, `chain` |

**Filtering / matching results:**

| Flag | Description |
|---|---|
| `--sc <codes>` | Show only these response status codes |
| `--hc <codes>` | Hide these response status codes |
| `--sl <n>` / `--hl <n>` | Show / hide by number of lines |
| `--sw <n>` / `--hw <n>` | Show / hide by number of words |
| `--sh <n>` / `--hh <n>` | Show / hide by response size (chars/bytes) |
| `--ss <regex>` / `--hs <regex>` | Show / hide responses matching a regex |
| `--filter <expr>` | Advanced filter expression (e.g., `c=200 and l>10`) |

**Request customization:**

| Flag | Description |
|---|---|
| `-X <method>` | HTTP method |
| `-d <data>` | POST data (can contain `FUZZ`) |
| `-H "<header>"` | Custom header (can contain `FUZZ`) |
| `-b "<cookie>"` | Cookie (can contain `FUZZ`) |
| `--follow` | Follow HTTP redirects |
| `-p <proxy>` | Proxy address, `--zP` compatible |
| `-t <n>` | Number of concurrent threads (default 10) |
| `-s <n>` | Delay in seconds between requests |
| `-R <depth>` | Recursion depth for recursive brute-forcing |
| `-A` | Ignore SSL/TLS certificate errors |

**Output:**

| Flag | Description |
|---|---|
| `-f <file>,<format>` | Write output to file in given format (`raw`, `json`, `html`, `csv`, etc.) |
| `-v` | Verbose output |
| `-o <format>` | Change console output format (`text`, `json`, `html`, `csv`) |
| `--dry-run` | Show what would be requested without sending requests |

#### 7. Common Use Cases

- Directory and file brute-forcing with granular multi-criteria filtering (status, size, word count, regex).
- Parameter, header, and cookie fuzzing for discovering hidden functionality or testing input handling.
- Multi-position fuzzing for combined username/password testing or parameter-pollution style checks (in authorized testing only).
- Numeric ID fuzzing to probe for Insecure Direct Object Reference (IDOR) vulnerabilities.
- Virtual host and Host-header based discovery, similar to FFUF's vhost fuzzing.
- Fine-grained, scriptable fuzzing for edge cases not well covered by simpler "one wordlist, one URL" tools.

#### 8. Automation with Bash

```bash
#!/usr/bin/env bash
# wfuzz_sweep.sh - directory + vhost sweep using wfuzz

TARGET="$1"                    # e.g. http://target.local
DIRLIST="/usr/share/wordlists/dirb/common.txt"
VHOSTLIST="vhosts.txt"
OUT="wfuzz_out_$(date +%Y%m%d_%H%M%S)"
mkdir -p "$OUT"

echo "[*] Directory fuzz on $TARGET"
wfuzz -z file,"$DIRLIST" --hc 404 -f "$OUT/dirs.json,json" "$TARGET/FUZZ"

DOMAIN=$(echo "$TARGET" | sed 's~https\?://~~')
echo "[*] Vhost fuzz on $DOMAIN"
wfuzz -z file,"$VHOSTLIST" -H "Host: FUZZ.$DOMAIN" --hh 0 \
      -f "$OUT/vhosts.json,json" "$TARGET/"

echo "[*] Sweep complete -> $OUT"
```

#### 9. Tips and Best Practices

- Always pair a first "baseline" request against a known-nonexistent path to learn the soft-404 signature (size/word/line count), then use `--hh`/`--hw`/`--hl` to filter it out.
- Prefer `--filter` expressions for complex multi-condition logic instead of stacking many `--sc`/`--hc` flags.
- Use `--iterator=product` deliberately — it multiplies request count by the size of every payload list, so scope it carefully with smaller lists first.
- Use `-t` and `-s` together to throttle scans against rate-limited or production targets.
- Save results with `-f file,json` (or `csv`/`html`) for reporting and to diff against future scans.
- Because WFuzz can fuzz literally any part of a request, it is often the right tool once Gobuster/FFUF/Feroxbuster/Dirsearch have already mapped the obvious surface and deeper, more targeted fuzzing is needed.

#### 10. Troubleshooting

| Problem | Cause / Fix |
|---|---|
| All results returned (soft-404 not filtered) | Add `--hc`, `--hh`, `--hw`, or `--filter` based on a baseline request |
| `Fatal exception: Unknown iterator` | Check `--iterator` value spelling (`product`, `zip`, `chain`) |
| Scan far slower than expected | Reduce `-s` delay to 0, or increase `-t` threads carefully |
| SSL errors | Add `-A` to ignore certificate validation issues on lab targets |
| `FUZZ` not being replaced | Confirm the literal keyword `FUZZ` (case-sensitive) appears in the URL/header/data, and that `-z` payload count matches the number of `FUZZ`/`FUZ2Z` placeholders |
| Deprecation warnings on run | Common with older WFuzz + newer Python versions; update via pip or use the Kali-packaged version |
