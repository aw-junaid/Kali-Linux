`ffuf` (Fuzz Faster U Fool) is a fast web fuzzer written in Go. It's used to discover hidden directories, files, parameters, and other web resources by sending numerous requests and analyzing the responses.  It's a powerful and versatile tool for web application security testing and reconnaissance.

**What `ffuf` Does:**

`ffuf` takes a target URL and a set of input values (wordlists, payloads, etc.) and sends HTTP requests with those values. It then analyzes the responses based on various criteria (status codes, content length, etc.) to identify interesting resources.

**Key Features and Capabilities:**

*   **Fast and Efficient:** Written in Go, `ffuf` is designed for speed.
*   **Highly Configurable:** Offers many options to customize the fuzzing process.
*   **Wordlist-Based:** Uses wordlists for directory and file discovery.
*   **Payload-Based:** Can use different types of payloads for parameter fuzzing.
*   **Recursion:** Supports recursive directory discovery.
*   **Filtering:** Allows you to filter results based on various criteria.
*   **Output Formats:** Can output results in different formats.
*   **Extensible:** Can be extended with custom matchers and filters.

**How to Use `ffuf`:**

  **Installation:** `ffuf` is often available in package repositories (e.g., `apt install ffuf` on Debian/Ubuntu, `brew install ffuf` on macOS).  You can also download pre-compiled binaries or build it from source.

**Basic Usage (Directory Discovery):**

```bash
ffuf -w wordlist.txt -u http://target.com/FUZZ
```

`-w wordlist.txt`: Specifies the wordlist file.
`-u http://target.com/FUZZ`: The target URL. `FUZZ` is a placeholder that `ffuf` replaces with the values from the wordlist.

**Basic Usage (Parameter Fuzzing):**

```bash
ffuf -w wordlist.txt -u http://target.com/?param=FUZZ
```

This will fuzz the `param` parameter.

  **Options:**

**HTTP OPTIONS:**

*   `-H <header>`: Add a custom header (e.g., `-H "Cookie: value"`). Can be used multiple times.
*   `-X <method>`: HTTP method to use (e.g., `-X POST`, `-X PUT`).
*   `-b <cookie>`: Cookie data (e.g., `-b "name1=value1; name2=value2"`).  Used for the "copy as curl" functionality.
*   `-cc <cert>`: Path to the client certificate file.
*   `-ck <key>`: Path to the client key file (required if `-cc` is used).
*   `-d <data>`: Data for POST requests.
*   `-http2`: Use HTTP/2 protocol.
*   `-ignore-body`: Do not fetch the response content (for faster scanning when only headers are important).
*   `-r`: Follow redirects.
*   `-raw`: Do not encode the URI.
*   `-recursion`: Scan recursively. The URL must end in `FUZZ`.
*   `-recursion-depth <depth>`: Maximum recursion depth.
*   `-recursion-strategy <strategy>`: Recursion strategy: `default` (redirect-based) or `greedy` (recurse on all matches).
*   `-replay-proxy <proxy>`: Replay matched requests using this proxy.
*   `-sni <sni>`: Target TLS SNI (Server Name Indication).  Does *not* support the `FUZZ` keyword.
*   `-timeout <seconds>`: HTTP request timeout.
*   `-u <url>`: *Required*. The target URL. Use `FUZZ` as a placeholder for the input values.
*   `-x <proxy>`: Proxy URL (SOCKS5 or HTTP).

**GENERAL OPTIONS:**

*   `-V`: Show version information.
*   `-ac`: Automatically calibrate filtering options.
*   `-acc <string>`: Custom auto-calibration string. Implies `-ac`.
*   `-ach`: Per-host auto-calibration.
*   `-ack <keyword>`: Auto-calibration keyword (default: `FUZZ`).
*   `-acs <strategy>`: Custom auto-calibration strategies. Implies `-ac`.
*   `-c`: Colorize output.
*   `-config <file>`: Load configuration from a file.
*   `-json`: JSON output.
*   `-maxtime <seconds>`: Maximum running time for the entire process.
*   `-maxtime-job <seconds>`: Maximum running time per job.
*   `-noninteractive`: Disable the interactive console.
*   `-p <delay>`: Delay between requests (seconds or a range, e.g., `0.1` or `0.1-2.0`).
*   `-rate <requests>`: Rate of requests per second.
*   `-s`: Silent mode (minimal output).
*   `-sa`: Stop on all error cases (implies `-sf` and `-se`).
*   `-scraperfile <file>`: Custom scraper file path.
*   `-scrapers <groups>`: Active scraper groups (default: all).
*   `-se`: Stop on spurious errors.
*   `-search`: Search for a FFUFHASH payload from ffuf history.
*   `-sf`: Stop when >95% of responses return 403 Forbidden.
*   `-t <threads>`: Number of concurrent threads.
*   `-v`: Verbose output (shows full URL and redirects).

**MATCHER OPTIONS:**

These options define what responses are considered a "match" (i.e., interesting).

*   `-mc <codes>`: Match HTTP status codes (e.g., `-mc 200,302,404` or `-mc all`).
*   `-ml <lines>`: Match the number of lines in the response.
*   `-mmode <operator>`: Matcher set operator (`and` or `or`).
*   `-mr <regexp>`: Match using a regular expression.
*   `-ms <size>`: Match HTTP response size.
*   `-mt <time>`: Match time to first byte (e.g., `-mt >100` or `-mt <500`).
*   `-mw <words>`: Match the number of words in the response.

**FILTER OPTIONS:**

These options define what responses are *excluded* from the results.  They work similarly to the matcher options.

*   `-fc <codes>`: Filter HTTP status codes.
*   `-fl <lines>`: Filter by the number of lines.
*   `-fmode <operator>`: Filter set operator (`and` or `or`).
*   `-fr <regexp>`: Filter using a regular expression.
*   `-fs <size>`: Filter HTTP response size.
*   `-ft <time>`: Filter time to first byte.
*   `-fw <words>`: Filter by the number of words.

**INPUT OPTIONS:**

*   `-D`: DirSearch wordlist compatibility mode (used with `-e`).
*   `-e <extensions>`: Comma-separated list of extensions (e.g., `-e php,html`).
*   `-enc <encoders>`: Encoders for keywords (e.g., `FUZZ:urlencode,b64encode`).
*   `-ic`: Ignore wordlist comments.
*   `-input-cmd <command>`: Command producing the input (requires `--input-num`).
*   `-input-num <number>`: Number of inputs to test (used with `--input-cmd`).
*   `-input-shell <shell>`: Shell to be used for running command
*   `-mode <mode>`: Multi-wordlist operation mode: `clusterbomb`, `pitchfork`, `sniper`.
*   `-request <file>`: File containing the raw HTTP request.
*   `-request-proto <protocol>`: Protocol to use with the raw request (default: `https`).
*   `-w <wordlist>`: Wordlist file path (and optional keyword, e.g., `/path/to/wordlist:KEYWORD`).

**OUTPUT OPTIONS:**

*   `-debug-log <file>`: Write internal logging to a file.
*   `-o <file>`: Write output to a file.
*   `-od <directory>`: Directory to store matched results.
*   `-of <format>`: Output format: `json`, `ejson`, `html`, `md`, `csv`, `ecsv`, or `all`.
*   `-or`: Don't create the output file if there are no results.

**Key Concepts:**

*   **Fuzzing:** The process of sending numerous requests with different inputs to discover vulnerabilities or hidden resources.
*   **Wordlists:** Lists of potential directory names, file names, or parameter values.
*   **Payloads:** Data sent in the HTTP request (e.g., values for parameters).
*   **Recursion:** Scanning subdirectories.
*   **Filtering:** Excluding certain results based on criteria like status code or content length.

**Use Cases:**

*   **Directory and File Discovery:** Finding hidden directories and files.
*   **Parameter Fuzzing:** Identifying vulnerable parameters.
*   **Subdomain Fuzzing:** Discovering subdomains.
*   **Web Application Security Testing:** Identifying vulnerabilities and misconfigurations.

**Important Considerations:**

*   **Wordlist/Payload Selection:** The effectiveness of `ffuf` depends heavily on the quality and relevance of the wordlists and payloads.
*   **Rate Limiting:** Be mindful of rate limiting and avoid overloading the target server.
*   **False Positives:** `ffuf` might report false positives.  It's crucial to verify the findings manually.
*   **Ethical Use:** Only use `ffuf` on systems that you have explicit permission to test. Unauthorized scanning is illegal and unethical.

`ffuf` is a fast and powerful web fuzzer that can be used for a variety of tasks.  Its flexibility and speed make it a valuable tool for web application security testing and reconnaissance.  However, it's important to use it responsibly and ethically.  Always obtain proper authorization before scanning any target.  Choose your wordlists and payloads carefully, be mindful of the impact on the target server, and verify the results.
